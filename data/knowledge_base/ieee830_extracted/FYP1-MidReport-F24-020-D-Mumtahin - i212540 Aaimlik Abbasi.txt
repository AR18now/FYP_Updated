Chapter 1 
 Introduction 
 Mumtahin focuses on revolutionizing how the current system of assignment 
 demos are conducted by integrating a lot of automation and AI into the system. It is 
 aimed to decrease bias allegations, manual labor and reducing conflicts for students and 
 teacher assistants for PF students submitting their assignments in C++. 
 1.1 Problem Statement 
 We are developing this system as a tool to alleviate students and assignments checkers 
 from the burden of doing tasks manually. Allocation of time slots and the availability of 
 both parties at the slot is a huge problem. The time suitable to students often is not suitable 
 to the teacher assistant. students sometimes have to leave the class in the middle or miss 
 classes to wait in queue for the demos. Sometimes the evaluator is busy in the slot agreed 
 upon or has some urgent work which need to be dealt with and has to reschedule which 
 often puts off the evaluation for a week or more leading to confusion and frustration. 
 All these issues arise just because of time slots and then there are the issues where the 
 evaluator sometimes knows the person they are evaluating and then they get accused of 
 showing positive or negative bias towards the students. This is a serious ethical issue with 
 no clear solution. Then there is the issue of evaluator assigning and uploading marks, 
 analyzing the solution which are a feat on their own and need to be done. 
 1.2 Motivation 
 Being students ourselves who had to give countless demos for which we often missed 
 classes got reprimanded for being late, leaving early and gone for most of the class and 
 seeing how inefficient the system is we had the idea to automate and solve as much of 
 these problems as possible. This would take a lot of pressure off of students, evaluators 
 and faculty. 
 1.3 Problem Solution 
 We are aiming to create a web app where we will be working on creating a portal which 
 will have student credentials, their info, their facial features. The student will login to the 
 portal and will be checked if there is a face in the frame. Next step would be to verify 
 The person in the frame is the actual student supposed to be evaluated. Next step would 
 be opening the assignment in front of the student and asking them questions about the 
7 
 
 assignment or topics related to the assignment which will be chosen by the teacher The 
 answer will be turned into text and checked for accuracy of the answer and student will 
 be graded on the basis of the answer matching. 
 During the live demo the students facial expressions will be constantly monitored to make 
 sure they are not cheating and are confident in their answers. A report will be generated 
 with transcription of the conversation between student and AI Evaluator with the marks 
 of the student for the demo, the code submitted and the facial expressions throughout 
 the video. The plagiarism for the assignment will be check beforehand and students 
 and teachers will be informed of the cases involved in the act. Submission checking 
 will also be done before the evaluation to make sure the code submitted for the problem 
 corresponds with the problem assigned to the student

 Project Description 
 2.1 Scope 
 The main scope that our product encompasses is to automate the whole process of demo 
evaluation. This includes checking submission, marking assignments, asking AI generated 
questions based on the assignment, taking quizzes, evaluating demos and finally, generating a 
report of the whole evaluation. This means that not only the fatigue of the Teaching Assistant 
would be reduced, but also, an AI based demo conduction would ensure non-biasness and 
thus, a fair marking for every student. 
 2.2 Modules 
 Given the current scope, our product aims to implement the following modules: 
 2.2.1 Module 1 
 The first Module serves to implement the foundation of our product. A strong foundation would 
lead to an efficient workflow. Thus, offering the following features: (Client sideWeb Application) 
 1. Figma Workflow 
 2. Initial Usecases (Login, Register, Submission Download and Facial Detection) 
8 
 
 2.2.2 Module 2 
 With the basic foundation complete, our next objective is to start implementing the necessary 
functionalities. The second module would therefore offer the following features: (Client and 
Admin Side Web Application) 
 1. Assignment Plagiarism 
 2. Speech To Text Conversion 
 3. Text to Speech Conversion 
 4. Question generation based on Assignment 
 5. Answer Verification of the student by the AI model 
 2.2.3 Module 3 
 With the first two modules, the integral structure of our product is basically complete. Now we 
need to involve more important features that would improve the quality and credibility of our 
product. Thus, for our third module, we aim to implement the following features: (Client and 
Admin Side Web Application) 
 1. Facial Expression Evaluation to improve the credibility of our demos 
 2. Generate the report of demos 
 3. Submission Checking 
 2.2.4 Module 4 
 For our last module, our product is basically complete. For any product, Testing is as important 
as the rest of the implementation, so finally, our final module would offer testin as well as the 
deployment of our product. (Client and Admin Side Web Application)

3. SRS 
 We are developing tools to implement a standardized evaluation system. This system will 
incorporate live demo sessions, assignment assessments, and evaluations. Additionally, it will 
utilize facial recognition technology to enhance the integrity of the process. Our goal is to ensure 
consistent evaluation criteria across all assessments. This approach aims to provide a fair and 
reliable evaluation experience for students and educators.

17. Implementation Details 
 17.1. Front-End Implementation 
 The front-end implementation for the desktop application is developed using HTML, CSS, 
and Bootstrap to ensure a responsive and visually appealing interface. It includes key features 
like a dashboard, slot booking, quiz interface, and report viewer, all styled for consistency 
across platforms. The interface integrates seamlessly with the back-end services via secure 
APIs, with a focus on usability and accessibility. 

45 
 
 17.2 Back-End Implementation 
 The back-end is developed using Flask and MongoDB. Flask is used to create a web 
server that handles the application's requests, while MongoDB stores user data, reports, and 
other information in a scalable database. This back-end efficiently processes data and 
communicates with the front-end through secure APIs.
