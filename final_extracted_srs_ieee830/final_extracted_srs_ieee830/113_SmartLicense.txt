Project: SmartLicense - Automated Driving License Testing System

1. INTRODUCTION
1.1 Purpose
This document outlines the problem and solution for automating the driving license testing process using sensors, cameras, and AI models to ensure a more fair, accurate, and efficient testing process.

1.2 Scope
The system automates driver's license exams using sensors and AI. Features include driver performance monitoring (traffic laws, following distance, mirror use), assessment of reverse and parallel parking, processing test data via image processing models, automated decision-making, and web/desktop interfaces for candidates and administrators.

1.3 Definitions, Acronyms, Abbreviations
- ITS: Intelligent Tutoring System
- CNIC: Citizens National Identity Card
- AI: Artificial Intelligence

2. OVERALL DESCRIPTION
2.1 Product Perspective
The system replaces subjective human examiners with an objective, AI-driven assessment process, utilizing hardware sensors and cameras integrated via a Raspberry Pi.

2.2 Product Functions (Summary)
- Data gathering from proximity sensors and cameras.
- Data processing and cleaning.
- Model training for traffic sign recognition, driver attention, seat-belt usage, and mirror checking.
- Output processing to determine pass/fail and generate feedback.
- Admin panel for managing tests and reviewing results.
- User panel for viewing test results.
- Hardware integration for seamless communication.

2.3 User Classes and Characteristics
- ITP Admin: Manages and oversees the testing system. (≈50 users)
- Candidates: Undergo the driving test. (≈1,000 users/month)
- System Developers: Build and maintain the system.
- Regulatory Authorities: Ensure the system meets national standards.
- Maintenance Technicians: Maintain and troubleshoot hardware.

3. FUNCTIONAL REQUIREMENTS
3.1 Data Gathering
FR1.1: The system shall collect data from six proximity sensors (two front, two center, two back) to measure distances.
FR1.2: The system shall capture high-resolution video from a front-facing dashboard camera for traffic sign monitoring.
FR1.3: The system shall record video from interior cameras to assess driver compliance (seatbelt, attention, hand position).

3.2 Data Processing
FR2.1: The system shall clean, preprocess, and standardize data from all sensors and cameras.
FR2.2: The system shall synchronize data from multiple sensors and cameras for cohesive analysis.
FR2.3: The system shall preprocess video frames for smooth image analysis.

3.3 Model Training and Evaluation
FR3.1: The system shall use image processing and object detection models to observe driver behaviors.
FR3.2: The system shall utilize object detection ML models for traffic sign detection and driver response analysis.
FR3.3: The system shall evaluate candidate parking performance using sensor and video data.

3.4 Output Processing
FR4.1: The system shall automatically determine if a candidate passes or fails based on set criteria.
FR4.2: The system shall generate a detailed feedback report for each candidate on specific tasks.

3.5 Admin Panel Interface
FR6.1: The system shall provide a robust, .NET-based desktop application for administrators to manage tests and monitor data.
FR6.2: The system shall enable administrators to respond to user feedback.

3.6 User Panel Interface
FR5.1: The user shall be able to view a detailed report of the test and feedback via a web panel.

3.7 Hardware Integration
FR7.1: The system shall integrate proximity sensors, cameras, and a Raspberry Pi to capture and transmit data.
FR7.2: The system shall manage communication between hardware components and the central server.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Reliability
NFR1: The system shall automatically log errors and failures for review.

4.2 Usability
NFR2: Administrators shall be able to configure and control tests in no more than 3 steps.
NFR3: Drivers shall receive test results within 10 minutes of completion.
NFR4: Users shall be able to view test reports in no more than 2 steps.

4.3 Performance
NFR5: The system shall process sensor and camera data and generate results within 10 minutes.

4.4 Security
NFR6: The system shall use role-based authorization to control access to candidate data.
NFR7: The system shall prevent unauthorized access without permissions.
