1.2 Scope
The architecture and construction industries often face a significant challenge in bridging
the communication gap between clients and professionals. Clients typically struggle to
visualize 2D plans or even traditional 3D renders, resulting in misunderstandings, dissat-
isfaction, and costly revisions during or after the construction phase. These miscommu-
nications arise because clients are unable to fully grasp the scale, design details, or spatial
layout of their future spaces through static visuals. Furthermore, last-minute changes
due to unmet expectations can lead to project delays and budget overruns, impacting the
overall efficiency of the design and construction process.
To address this issue, we are currently creating ARch360, an application that will apply
the use of Augmented Reality to present architectural designs in the form of interactive
3D environments. Through this system, clients can get a feel of their future homes or
commercial space before construction takes place. Through real-time walkthroughs, and

3.3 Data Design
Design Models for Procedural Approach In ARchitect360, the information domain of
the system is transformed into a line of data structures that regulate architectural models,
user preferences, spatial data, and AI suggestions of the system. These data structures
then enable the system to provide complex functionality, including rendering 3-D models
in AR, customization of design, scanning the environment, and AI-driven design sugges-
tions.
Data Structures and Entities
1. Architectural 3D Models
• Data Structure: The 3D models are stored in any format like .FBX, .OBJ, or
AutoCAD. Unity parses the models and renders them by using mesh structures
comprising vertices, edges, and faces that define the geometry of the building.
• Storage and Processing: The geometries and models are loaded into Unity’s
MeshRenderer, which processes them for visualization. Models can be stored
locally or fetched from cloud storage depending on the system configuration.
• Organization: In Unity, it is organized by different building components like
walls, floors, and ceilings, structured into a scene hierarchy. This structur-
ing would permit users to interact with and make their own adjustments to
individual elements.
2. User Customization Data
• Data Structure: Wall colors, flooring, and decor customization options are
kept in the form of either key-value pairs or as objects in JSON. Every option
is tagged with model parts the respective option belongs to, for example, wall,
floor, etc.
• Storage and Processing : When a user chooses an option to customize, the
system will pick up an appropriate file for a certain material or texture involved-
such as wall colors or floor patterns-and apply it to the relevant component of
the given 3D model. This is handled dynamically in Unity using material
shaders.
• Organization: It stores information about user and session personalization;
hence the system will be able to remember user preferences across sessions
by storing them in a NoSQL database like MongoDB or Firebase.
3. Environment Scanning Data (Spatial Data)

3.3 Data Design
• Data Structure: The spatial data originates from environment scanning through
ARCore. This information is stored as point clouds and mesh maps. Such
point clouds contain 3D coordinates that represent a user’s environment ge-
ometry.
• Storage and Processing: ARCore processes the captured spatial data in real
time by creating a spatial map representative of the particular physical envi-
ronment. The data is to be used by the module of Area Analysis for accurate
placing of the virtual 3D models into the real world.
• Organization: Spatial data is organized in a 3D grid or octree structure so
that efficient mapping of large environments is possible. Each node in the
grid represents a real-world location for accurate positioning and placement
of virtual objects.
4. AI Data Structure Design Suggestions:
• Data Structure: Pre-trained language models and decision trees make the AI-
driven suggestions. The input parameters of the AI model are text or numeric
input data, such as user preferences (color, lighting, furniture); room dimen-
sions; and style preferences. A set of recommended design options shall be
the output based on the input.
• Storage and Processing : Storage and Processing shall be done by the AI,
which shall process the user’s input and give certain design recommendations.
These are stored in JSON format and utilized through customization options
applied to the 3-D models.
• Organization: The AI data generated shall be stored on a per-user profile ba-
sis, with suggestions connected to specific portions of the architectural model
for personalized changing recommendations.
5. Data Structure: User Profile and Session Data:
• The architectural design of the system will decide whether or not the user
profiles reside in a relational database, such as PostgreSQL, or otherwise in
the NoSQL database, MongoDB. With the profile, user information includes
past projects, preference about customizations, and AI-generated recommen-
dations.
• Storage and Processing: User profiles will be persisted within the system so
that users can revisit their previous choices in design, view recommendations,
and make changes between sessions.
• Data Organization: The data from the User is stored in tables or documents.
For each given user, there is an ID that connects, with that ID, their prefer-

Databases and Data Storage ARchitect360 uses both Relational and NoSQL databases

• NoSQL Database-e.g., Firebase or MongoDB

– Why : In essence, local caching improves the performance by allowing the
system to fetch generally accessed files at incredible speed.
