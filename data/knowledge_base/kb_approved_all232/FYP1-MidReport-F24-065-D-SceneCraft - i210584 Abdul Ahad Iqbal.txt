The other problem, which writers experience while working on a movie script, is tag-

slowing the scripting.
1.2 Scope
As a new tool developed to help make the conversion of book manuscripts to screenplay
easier, SceneCraft is a powerful tool for any writer. This project also includes the creation
of an effective and intuitive interface where book manuscripts may be uploaded and where
the program sends out properly formatted, industry-standard screenplays that users can
edit directly as part of the project scope. Elements like plot points, characters, and settings
from the story will be determined by our application. SceneCraft will then grow these
elements into a classic screenplay form from which the heart of the story will be retained
though it will be changed and put into a new visual narrative media form. The system
will also feature an editing tool that enables the user to edit the screenplay, suggesting
improvements from artificial intelligence. An automated feature in SceneCraft will be
added to highlight and tag key production elements within the Screenplay. The goal
of this project is to help increase the availability of opportunities where creative people
have the chance to apply their talents toward a book-to-screen project. SceneCraft will
enable the adaptation and editing tool but will not extend through the other stages of the
filmmaking process. Our goal is to bring people closer to where writing and screenplay
visuals meet, allowing those who want to tell stories to get them on the screen more easily
than before.

2.2.1 Module 1
The following are the functional requirements for Module 1:
1. FR1: The SceneCraft system shall implement a tagging model which will identify
and label key elements (e.g., characters, settings, dialogue) in a screenplay.
2. FR2: The SceneCraft system shall provide users with a script editor interface that
will allow users to edit and format screenplays.
3. FR3: The SceneCraft system shall implement a book scrapping functionality that
can extract content from EPUB files. The system shall process and extract a stan-
dard novel-length book while maintaining essential metadata such as title, author
and publication date.
2.2.2 Module 2
The following are the functional requirements for Module 2:
1. FR1: The SceneCraft system shall automatically extract entities (such as characters,
locations, and key plot points) from uploaded book manuscripts.
2.2.3 Module 3
The following are the functional requirements for Module 3:
1. FR1: The SceneCraft system shall transform book manuscript text into a neatly
organized screenplay framework that complies with industry norms and correctly
recognizes scene headings, action lines, character names, and dialogues.
2. FR2: The SceneCraft system shall create a real-time editing interface that enables
users to edit the scripted text while seeing the changes made by multiple users on
the same document flawlessly.
3. FR3: The SceneCraft system shall offer recommendations from AI for better dia-
logue and character growth while editing scenes that are relevant to users according
to their input and created upon triggering the suggestion feature.

REL-1: The SceneCraft system shall sustain an MTBF greater than 720 hours (30 days). A
disruption happens if it stops users from engaging with their projects or completing
essential actions within 5 minutes.
REL-2: The SceneCraft system shall set up daily automated backups for every project and
system configuration that permit users to retrieve from any backup point within 30
minutes.
REL-3: The SceneCraft system shall constantly assess its well-being and alert the team for
any unexpected issues or signs of trouble. Within 5 minutes of any detected issues
occurring, the maintenance team will receive alerts.
2.3.2 Usability
USE-1: The SceneCraft system shall allow users to change a manuscript to a screenplay
structure within 5 clicks from the main interface.
USE-2: The SceneCraft system shall offer assistance relevant to each core feature with a
single click or tap from every page.
USE-3: The SceneCraft system shall save user updates automatically every 2 minutes and
let users return to any older version with just two clicks.

3.3 Data Design
Our data is stored in and organized with MongoDB, a NoSQL document-based database.
It transforms the system’s information into a collection of documents, corresponding to
21
3. System Overview
major entities within the system. Here’s how the data is structured and processed:
3.3.1 Data Storage
1. We store each entity as a separate document for User, Project, Manuscript, Screen-
play, Version, SceneElement, and AISuggestion in their respective MongoDB col-
lections.
2. Entity relationships are maintained through reference fields (e.g. Screenplay docu-
ment entity has a reference to a Project by the projectId field).
3.3.2 Data Processing
1. When you upload a manuscript, it is added to the Manuscript collection and then
processed to create a new Screenplay document.
2. When users edit the screenplay new Version documents are created to keep track of
what has changed.
3. We extract SceneElements, storing them separately and linking them up with a
Screenplay for quick retrieval and manipulation.
4. The AI suggestions are stored and linked to the relevant Screenplay, as separate
documents.
3.3.3 Data Organization
1. The top-level organizational unit is Projects that contain Manuscripts and Screen-
plays.
2. Version collection is used in screenplays for tracking changes in time.
3. Screenplays are linked to separate scene elements so they can be updated and rear-
ranged quickly without altering the whole Screenplay document.
With this structure, the data can be managed flexibly and scale efficiently, including fea-
tures such as collaborative editing, version control, and AI-assisted writing, while pre-
serving the data integrity and making query and retrieval of information more efficient.
