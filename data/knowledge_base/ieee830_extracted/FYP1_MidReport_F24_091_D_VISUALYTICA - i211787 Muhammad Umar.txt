1. Introduction
1.2 Problem Statement
Data visualization is essential for businesses, especially today. Existing tools for graph
data (nodes and edges) are often costly, complex, and lack flexibility for specific visual-
ization needs. Users must manually transform raw graph data into insights, a process that
can be time-consuming and prone to errors.
These tools require extensive coding, making them difficult for non-programmers and
even technical users due to their steep learning curves. This limits the broader application
of graph visualization, hindering users from fully leveraging their data.
Visualytica addresses these issues by simplifying graph visualization. It reduces technical
complexity and improves interactivity, enabling users to easily analyze large datasets and
extract meaningful insights.
1.3 Scope
Visualytica is a development project aimed at the creation of a dedicated web platform for
visual analysis on graph databases. The platform will also allow users to build sophisti-
cated graph (node and edge) visualizations with an intuitive interface for customization
and interaction.
Visualytica will embed a concise yet powerful recommendation engine, customized to
graph visualizations. The algorithm will help you make sense of the structure in your
graph data so it can suggest which visualizations are most appropriate. Or, if the graph
is a tree it will recommend using trees for visualization. It also recommends layouts
like force-directed graphs or hierarchical visualizations for more complex networks. The
recommendation feature will help users choose the most intelligent visuals for their data.
The platform will also get in build support for interactive possibilities like navigating
through the graph structures, connections and patterns on-the-fly. By concentrating only
on graph visualization Visualytica streamlines the complex networks comprehension pro-
viding non-technical users with actionable insights.
The portal will also include basic dashboard capabilities, allowing for multiple chart views
to be together into a unifying interactive view of your data. In addition, the platform will
collate using a domain specific chatbot that informs users over visualization concepts such
as how to comprehend and expose connection graph data.

3.3 Data Design
In this section, we explain how the information from users and graphs is turned into data
that the system can use. We’ll describe how the data is stored, processed, and organized
to make the system work smoothly.
3.3.1 User Data
User-related information, such as login details and authentication, is handled through a
MongoDB database. MongoDB stores all the user data securely, like usernames and
passwords, ensuring that only the correct users can log in and interact with the system.
3.3.2 Graph Data
The graph data, such as nodes, edges, and graph types (like trees, DAGs, or simple
graphs), is stored in different formats depending on user input. These formats are sup-
ported to allow flexible input and processing:
• Adjacency Matrix: This format uses a matrix to show which nodes are connected.
Rows and columns represent nodes, and the matrix cells contain values indicating
whether there is an edge between nodes. This format is useful for analyzing dense
graphs.
• Edge List: The edge list format represents the graph as a list of all the edges,
where each entry shows a pair of connected nodes. This format is efficient for
sparse graphs and makes it easy to validate and plot graphs.
• Adjacency List: In the adjacency list format, each node has a list of its neighboring
nodes. This structure is lightweight and works well for storing large graphs with
fewer connections.
3.3.3 Data Structures
Several important data structures are used to handle and process the graph data:

• Adjacency Matrix: Stored as a 2D array, where each row and column represents a
node, and values in the matrix indicate whether a connection exists between nodes.
• Edge List: A list of edges where each element is a pair (or triple, if weighted) of
nodes. This format is ideal for graph visualization and validation tasks.
• Adjacency List: A dictionary or list of lists, where each key (or index) represents
a node, and the associated value is a list of all the nodes it’s connected to.
• Node and Edge Attributes: Any extra information about the nodes (like labels or
weights) is stored in dictionaries, making it easy to access this data when needed.
3.3.4 Database and Storage
• MongoDB: MongoDB is used only for handling user authentication. It stores user
accounts and passwords in a secure manner.
• Backend File Storage: The graph data uploaded by the user is stored in the back-
end’s/data directory. It is organized into different files based on user input, but no
user progress is saved for now.
