SOFTWARE REQUIREMENTS SPECIFICATION (IEEE 830)
Project: NAZAR - AI-Powered Mobile Application for Eye Health Screening
================================================================================

1. INTRODUCTION
1.1 Purpose
This document outlines the requirements for NAZAR, a mobile application designed to enhance eye health screening through advanced image processing technology, detecting cataracts and assessing refractive errors (myopia, hypermetropia, astigmatism) using smartphone cameras and AI models.

1.2 Scope
The app uses AI models (SVM, CNNs, ResNet) to analyze images of the eye captured by smartphone cameras. It includes modules for image capturing/preprocessing, AI model inference (cataract detection, refractive error classification), data management, map module for eye clinics, AR for eye positioning guidance, profile management, offline functionality, and tooltips/tutorials.

1.3 Definitions, Acronyms, Abbreviations
- AI: Artificial Intelligence
- SVM: Support Vector Machine
- CNN: Convolutional Neural Network
- AR: Augmented Reality
- API: Application Programming Interface

2. OVERALL DESCRIPTION
2.1 Product Perspective
NAZAR provides a cost-effective, accessible, and portable solution for early detection of eye conditions, particularly in underserved and rural areas, by leveraging widely available smartphone technology.

2.2 Product Functions (Summary)
- Image capturing and preprocessing: Capture/upload images, guide user for proper positioning, feedback.
- AI Model: Analyze images, detect cataracts and refractive errors, provide feedback and accuracy rate.
- Data Management: Store images securely (privacy compliance), encrypt data, retrieve history.
- Map Module: Display map of eye clinics.
- AR in Camera Module: Use AR to guide eye positioning.
- Profile Module: Create/manage profiles, maintain patient history.
- Offline Functionality: Save data offline, sync when online.
- ToolTip Module: Provide guidance via tooltips and tutorials for new users.

2.3 User Classes and Characteristics
- (The primary user is the patient/individual using the app for self-screening. Stakeholders include developers, NGOs, hospitals, etc.)

3. FUNCTIONAL REQUIREMENTS
3.1 Image Capturing and Preprocessing
FR1: The application shall be capable of allowing the user to capture images or upload images from smartphone cameras.
FR2: The application must guide the user to capture images of their eyes clearly.
FR3: The application should give feedback like eye positioning to ensure good quality images.
FR4: The system should be able to process images using inbuilt processors.

3.2 AI Model
FR1: The AI model should be capable of analyzing images and detecting eye diseases like cataract.
FR2: The AI model should work on the device chipsets.
FR3: After analyzing the image, the model should provide basic feedback about its predictions.
FR4: The AI model should give an accuracy rate of its result.

3.3 Data Management
FR1: The application should be capable of storing images for further boosting the dataset.
FR2: The data stored must comply with privacy laws of Pakistan.
FR3: The application should secure user data and should encrypt it when transmitting to the server.
FR4: The system should be capable of performing data retrieval to ensure storing of past history of patients.

3.4 Map Module
FR1: The application shall display a map of all eye clinics to help guide the user to the nearest centre.
FR2: The map must be responsive.
FR3: The map can be zoomed in and out, and be able to pan on the map.
FR4: The map module should be quickly loadable without any noticeable lags.

3.5 AR in Camera Module
FR1: The application shall use AR to guide users in positioning their eyes.
FR2: AR must be supportable for all types of cameras in smartphones (both back and front).
FR3: If eyes are not aligned properly, the app should notify the user.
FR4: Real-time feedback should be quick and instant, without any noticeable delays.

3.6 Profile Module
FR1: The application shall allow the user to create and manage profiles, and keep history of patients and users.
FR2: There should be multiple optionality and variation of profile management for NGOs, Hospitals, and normal users.
FR3: The user should be able to load the history, and Clinics can keep the patient records, and NGOs can track their work and dataset.
FR4: Data syncing should be quick when online.

3.7 Offline Functionality Module
FR1: The application shall allow NGOs to save patient data offline and access the application without active internet.
FR2: The application should allow syncing when the device is online and proper optimization of storage.
FR3: If the app detects no internet connection, it should notify the user and switch to offline functionality.
FR4: The eye scanning should not be hindered, and the AI model should provide accuracy comparable to online.

3.8 ToolTip Module
FR1: The application shall provide guidance via tooltips and tutorials for new users.
FR2: The tutorial should not interfere or obstruct the module when explaining it.
FR3: The tutorial should pop up when a new user opens the application.
FR4: The transition between tool tips should be smooth, and functionality should load instantaneously.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Performance
NFR1: The result of image scan should be given within a few seconds on mid to high-end mobile devices.

4.2 Security
NFR2: The application must provide users with at least basic level of security (encryption, prevent unauthorized access/disclosure, gather information with consent).

4.3 Scalability
NFR3: The AI model should be able to scale gradually to incorporate better sensitivity and accuracy.
NFR4: The application should be able to increase its user base.

4.4 Portability
NFR5: The application should be compatible with a wide range of smartphones across multiple platforms (iOS and Android).
