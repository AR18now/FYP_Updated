1.2 Scope
This project aims to make an advanced software system for credit card fraud detection
using an encoder-decoder Graph Neural Networks. Firstly, a synthetic dataset will be
generated that has been designed to be as close as possible to real-life transaction sce-
narios, allowing robust model training. Furthermore, the project will extract and include
crucial features through feature engineering, this will enhance the model’s capability of
differentiating between fair and fraudulent transactions. A key aspect of the project is to
make a graph-based model of credit card transactions, where customers and merchants
are represented as nodes, and transactions are represented as edges, enabling the system
to inspect complex relationships within the data. The GNN-based fraud detector at the
heart of the system will encode transaction data into compact representations and then
decode this representation to identify fraudulent activities. To make training faster and
improve accuracy, the project will implement batch normalization techniques, which will
normalize activation within the GNN layers. The user interface will be designed to be eas-
ily used by clients. The user interface will provide visualization of transaction data, some
performance metric dashboards, and also tools for investigating any fraudulent activities
the system detects. The system will also be scalable which means it will be handling
large chunks of data, and the system will also be able to adapt to evolving fraud tactics
in real-time. Lastly, the system will have the ability to integrate with existing banking
systems, which will facilitate easy acceptance by financial institutions. To conclude, this
project’s goal is finding a practical and efficient solution to the growing problem of credit
card fraud by emphasizing on accuracy, scalability, and ease of use.
1.3 Modules
The project is divided into modules, each of which tackles a different component of the
credit card fraud detection system. These modules will help break down the overall pro-
cess of development into sections which ensures the essential elements of the system are
developed, tested, and integrated. The team can concentrate on individual tasks, such
as data preparation, model development, and user interface, while having a clear path to
reach the overall project objectives. All this will be achieved by dividing the project into
these modules.

2. The system shall normalize the transaction data to ensure consistency across all
features.

for model training.

Feature 1: Graph Generation

resent customers and merchants, and edges represent transactions.
2. The system shall ensure that the graph generation process can scale to handle large
transaction datasets efficiently.
3. The system shall visualize the graph structure for easier understanding of relation-
ships between customers and merchants.

2.4 Non-Functional Requirements
2.4.1 Reliability
• NFR-R1: The system shall have a mean time between failures (MTBF) of no less
than 1,000 hours. A failure is defined as any event where the system is unable to
detect, process, or respond to fraudulent transactions in real time.
• NFR-R2: In case of a system failure, the system shall automatically attempt to
recover within 5 minutes by restarting the affected module (e.g., GNN model or
graph generator) and logging the incident.
• NFR-R3: The system shall provide continuous monitoring and alert users in case
of system failures, ensuring users are notified within 30 seconds of any failure.
• NFR-R4: System failures during real-time transaction processing shall lead to the
preservation of transaction data and automatic resumption from the failure point
once the system is back online.
2.4.2 Usability
• NFR-U1: The system shall allow a user to flag and investigate a fraudulent trans-
action within 5 clicks or interactions.
• NFR-U2: The user interface (UI) shall be designed for ease of use, such that a new
user can complete fraud investigation tasks without training within 1 hour of initial
interaction.
• NFR-U3: The system shall provide near real-time visual feedback (e.g., loading
indicators, transaction status) to ensure users are aware of ongoing processes during
fraud detection.
12
2.4 Non-Functional Requirements
• NFR-U4: The system shall ensure that error messages are clear and actionable,
providing users with steps to resolve issues within 10 seconds of error occurrence.
2.4.3 Performance
• NFR-P1: The system shall process up to 1,000 transactions per second (TPS) in
real-time without performance degradation.
• NFR-P2: The GNN model shall detect fraudulent transactions with a response time
of no more than 1 minute per transaction, regardless of dataset size.
• NFR-P3: The system shall support a maximum latency of 10 seconds for real-time
dashboard updates when processing transaction data.
• NFR-P4: The system shall scale automatically to handle an increase in transaction
volume up to 5x normal load without manual intervention.
2.4.4 Security
• NFR-S1: The system shall ensure that all transaction data is encrypted in transit
and at rest using 256-bit encryption.
• NFR-S2: The system shall require multi-factor authentication (MFA) for all users
to access critical components, such as fraud detection results and transaction logs.
• NFR-S3: The system shall automatically log out inactive users after 10 minutes of
inactivity to prevent unauthorized access.
