1. INTRODUCTION
1.1 Problem Statement
Pakistan faces one of the worst economic crises, causing many people to struggle with managing finances. Popular apps like JazzCash and EasyPaisa cover 56% of transactions but offer limited features. Budgeting apps like Mint and YNAB require manual entry, which is tedious and inconsistent.

1.2 Scope
ApnaBudget is a cross-platform budgeting mobile application targeting Pakistani population. Features include automated SMS and notification reading for detecting transactions, AI integration, bill/receipt image upload with automatic processing, financial planning and forecasting, scenario planning, stock visualization, insurance/savings/retirement plan comparison, tax templates, and reminders.

1.3 Definitions, Acronyms, Abbreviations
- AI: Artificial Intelligence
- SMS: Short Message Service
- OTP: One-Time Password
- MTBF: Mean Time Between Failures

2. OVERALL DESCRIPTION
2.1 Product Perspective
ApnaBudget provides financial awareness by helping people budget, plan, visualize, and forecast their finances, empowering them to make better financial decisions.

2.2 Product Functions (Summary)
- User authentication (email/password, Google login)
- Automated SMS and notification parsing for transactions
- Manual transaction entry
- Transaction categorization
- Budget creation for transaction categories
- Image upload for bills/receipts
- Spending visualization
- Financial planning and goal setting
- Scenario planning with inflation adjustment
- Stock visualization
- Insurance/savings/retirement plan comparison
- Tax template generation
- Bill payment reminders

2.3 User Classes and Characteristics
- Consumer: General population of Pakistan, track expenses and budget

2.4 Operating Environment
Mobile application (Flutter - cross-platform Android/iOS), SQLite for offline support, Firebase for remote updates, Kubernetes for microservices, MongoDB for storage.

3. FUNCTIONAL REQUIREMENTS
3.1 Module 1: SMS and Notification Reading For Automated Transactions
FR1.1: The system shall allow new users to register with name, email, password with SMS verification.
FR1.2: The system shall allow login with email/password and Google accounts.
FR1.3: The system shall provide password reset option.
FR1.4: The system shall support two-factor authentication.
FR1.5: The system shall automatically detect and read SMS containing transaction details.
FR1.6: The system shall automatically detect and read notifications containing transaction details.
FR1.7: The system shall extract transaction information (amount, time, merchant, date).
FR1.8: The system shall categorize transactions based on predefined categories.
FR1.9: The system shall allow users to manually verify and edit transaction details.
FR1.10: The system shall provide interface for manual transaction entry.
FR1.11: The system shall allow users to select from predefined categories when adding transactions manually.
FR1.12: The system shall allow users to manually reassign transaction categories.
FR1.13: The system shall support creation of custom categories.
FR1.14: The system shall allow users to create custom budgets for specific categories with amount and time.
FR1.15: The system shall track spending against budget in real-time and notify when approaching/exceeding budget.

3.2 Module 2: Uploading Bills and Receipts via Images
FR2.1: The system shall allow users to upload images of bills and screenshots for automatic processing.
FR2.2: The system shall allow users to capture photos of receipts/bills using in-app camera.
FR2.3: The system shall allow users to share screenshots directly from Photos app via custom share extension.
FR2.4: The system shall display extracted information for confirmation before logging.
FR2.5: The system shall allow users to create financial scenarios based on variables (income changes, estimated spending).
FR2.6: The system shall allow users to simulate impact of potential tax law changes.
FR2.7: The system shall incorporate periodically updated inflation data.
FR2.8: The system shall visualize changes using graphs.
FR2.9: The system shall allow sorting of transactions by date, amount, merchant.
FR2.10: The system shall provide filtering options (date ranges, categories, amounts, merchants).
FR2.11: The system shall notify users when about to exceed budget.

3.3 Module 3: Planning and Forecasting
FR3.1: The system shall allow users to create financial plans and goals (savings amounts).
FR3.2: The system shall allow custom plans with predefined options (car, mobile, Hajj/Umrah).
FR3.3: The system shall provide progress tracking with percentage completion and time remaining.
FR3.4: The system shall allow users to modify or update plans and goals.
FR3.5: The system shall allow users to create different financial scenarios.
FR3.6: The system shall allow users to input custom inflation rates manually.
FR3.7: The system shall allow users to compare multiple scenarios side by side.

3.4 Module 4: Final Tweaks and Additional Features
FR4.1: The system shall fetch and display historical stock data (price trends, volume, market performance).
FR4.2: The system shall allow users to select specific stocks and visualize performance.
FR4.3: The system shall provide comparison between insurance, savings, and retirement plans.
FR4.4: The system shall generate tax template based on income, transactions, and applicable tax laws.
FR4.5: The system shall allow users to export tax template as PDF or Word.
FR4.6: The system shall allow users to set reminders for bill payments, cash withdrawals, and custom tasks.
FR4.7: The system shall notify users when reminder is due.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Reliability
NFR1: Server failure definition: backend no responses due to pod issues, cluster problems, database availability.
NFR2: Backups in multiple zones.
NFR3: Disaster recovery and high availability servers.
NFR4: Failure detection using liveness probe provided by kubelet in Kubernetes.
NFR5: Rollback to previous deployment using Kubernetes control plane.

4.2 Usability
NFR6: Registration process ≤ 5 seconds after submission.
NFR7: Login response ≤ 3 seconds.
NFR8: SMS/notification transaction processing ≤ 1 minute.
NFR9: Parsing accuracy ≥ 95% for SMS/notifications.
NFR10: Data privacy with encryption at rest and in transit.
NFR11: Manual transactions saved ≤ 2 seconds.
NFR12: Manual transaction interface responsive for various screen sizes.
NFR13: Categorization process ≤ 10 seconds per transaction.
NFR14: Categorization accuracy ≥ 90%.
NFR15: Budget progress display ≤ 5 seconds.
NFR16: Image extraction accuracy ≥ 90%.
NFR17: In-app camera ≤ 3 taps to capture and process.
NFR18: Sorting/filtering ≤ 10 seconds for ≤ 1,000 transactions.

4.3 Security
NFR19: MD5 or SHA algorithm for encrypting transaction information.
NFR20: Strong password: ≥ 8 characters with uppercase letter and number/special character.
NFR21: AES algorithm for encrypting sensitive user information (passwords).
NFR22: External audits of security and data handling practices.
NFR23: Principle of least privilege for user data access.
