2.1 Use-cases 
 
1. Upload leaf Image: 
 
• Actor: User 
• Description: User uploads the image of a leaf to the system. 
• Precondition: The user has made an account and is logged into the application. 
• Main Flow: 
1. User selects a leaf image from gallery 
2. User uploads the leaf image for classification 
3. The system processes the image 
• Postcondition: The image is processed and classifies the leaf image into a specie. 
• Alternate Flow: If incorrect image is uploaded and accuracy is very low then 
error message shall be displayed to the user. 
 
 
2. Preprocess Image: 
 
• Actor: System 
• Description: Image is resized and augmented by the system to make it 
classification ready. 
• Precondition: Image has been correctly uploaded and preprocessed for the 
classifier . 
• Main Flow: 
1. Image is resized for the CNN. 
2. Image may be augmented to increase the dataset. 
3. The image is passed to the CNN so that specie classification can be done. 
• Postcondition: Image is sent to the classifier . 
• Alternate Flow: Error or exception is thrown when preprocessing cannot be done. 
 
 
3. Classify Species using CNN : 
 
• Actor: System 
• Description: Custom Trained CNN is used by the system to classify species 
according to the provided leaf images. 
• Precondition: Correctly preprocessed image has been received. 
• Main Flow: 
5 
 
 
1. CNN receives the preprocessed image 
2. CNN classifies the specie and gives a confidence score indicating how certain it 
 is. 
3. Classified Specie with a confidence score is returned to the user. 
• Postcondition: Specie identification along with confidence score is done . 
• Alternate Flow: Error message is displayed if confidence score is below (20 %) . 
 
4. Display Species Information : 
 
• Actor: User 
• Description: The system also displays the specie related information to the user 
• Precondition: Specie has been identified and confidence score has been 
generated. 
• Main Flow: 
1. System fetches specie data from the database. 
2. Specie name, confidence score, basic information regarding the specie is shown 
 to the user. 
• Postcondition: User views the identified specie along with the description . 
• Alternate Flow: If confidence score is below 20 percent an error message is 
displayed and consequently no information or identified specie is displayed. 
 
5. Provide Feedback: 
 
• Actor: User 
• Description: Feedback is provided by the user after his provided leaf image is 
classified by the system. 
• Precondition: The identified specie name and information has been reviewed by 
the user. 
• Main Flow: 
1. User provides feedback regarding the identification of the specie and can also 
 and can also provide comments on how improvements can be made. 
2. The feedback is stored in the database so that developers can review them later. 
• Postcondition: Feedback is stored in the database and “Thank you for the 
feedback (or some similar message) is shown to the user ”. 
• Alternate Flow: An exception is thrown if feedback is unable to be stored in the 
database. 
 
6. View Previous Classifications: 
 
• Actor: User 
• Description: User can see the classifications done in the past. 
• Precondition: User has used the application for classifications before. 
• Main Flow: 
1. User can select an option for previous classifications. 
2. User views previous classifications, confidence scores and relevant regarding to 
 the species classified in the past. 
Postcondition: User vies previous classifications. 
• Alternate Flow: User is shown empty history as no classifications were done by 
the in the past.

2.3 Non-Functional Requirements: 
 
2.3.1 Reliability 
 
Reliability of the system will be prioritized and will be gauged by an MTBF (Mean Time 
Between Failures), with a target of 500+ operation hours in optimal and normal conditions 
 
• Definition of Failure : 
Whenever uploading images failed, leaf classification failed, or a user did not get the 
result after 2 minutes maximum . 
 
• Error Detection Strategy: 
If the classification accuracy of the system falls below a certain value (e.g., 20%), an error 
message will be shown to the user, and he is given suggestion on what to do next . 
 
• Error Correction Strategy: 
8 
 
 
If an error is detected by the user, re -processing of the image will be done and if the 
failure is still here there, then an error message is displayed, and user is also guided on 
what needs to be done next (probably prompting to upload a better image and try again ) 
 
• Failure Consequences: 
If the failure occurs, leaf image would not be classified, Data integrity would still me 
maintained and images would be stored for later inspection. 
 
• Failure Recovery: 
If a failure happens and conditions are critical then system shall automatically restart and 
give an indication to the development team and tell the user when to be back as 
maintenance would be going on.

3.1 Architectural Design 
 
• Architectural Diagram 
 
 
• Description: 
As shown in the diagrammatic representation, the microservices architecture of a system designed to 
identify plant species from leaf images returning basic descriptions has components very well modularized, 
thus supporting individual development, scaling, and maintenance. 
 
• User Interface: 
Description: This Component represents the Front-end application where users can upload images of

the leaves. 
 
Function: User Interface is responsible for streamlining the entire process of collecting the image,

Description: This microservice processes the image received from the user.

Description: This microservice implements a deep learning model for Species classification.

State Transition Diagram 
 
 
 
Description: 
 State Transition diagram is used to display the flow of events that occurs in the lifespan of a software, 
basically explaining transition between states while interaction of the user with the system and explains how 
backend processes are handled such as sending the leaf image to the classifier, classification, preprocessing 
etc. 
 
Explanation of components: 
• Idle: In this state the system remains in an idle state until the user sends the leaf image to the 

18 
 
 
system. 
• Upload Image: When the user sends the image for classification, the system goes transitions to 
uploading state where it takes an image sent to it by the user. 
• Preprocessing: Image then preprocesses (resizes) so that it can be sent to the classifier thus system 
transitions to preprocessing state . 
• CNN Model: The system now transitions to prediction state where CNN runs to identify the specie 
on a preprocessed image. 
• Classification: If the confidence score is less than 20 percent it transitions to error handling state 
else it transitions to data fetching state. 
• Fetching Data: After successful classification, the system transitions to Fetching state where it 
fetches the species information from the database. 
• Display: After the data is fetched the system goes to Display state where it displays the specie name 
along with relevant information. 
• Feedback state: If the user decides to give feedback the program may go to feedback state and then 
to storing data state and this thing remains consistent with other features as well.
