1. INTRODUCTION
1.1 Purpose
This document outlines the problem statement, scope, and modules for RideMatch, a shared-fare carpooling app designed to help students socialize, connect, and commute with peers.

1.2 Scope
RideMatch matches users based on personality, likes, preferences, and routine. It finds optimal routes, provides a list of compatible peers, offers driver mode, in-app wallet, and a chatbot.

1.3 Definitions, Acronyms, Abbreviations
- AI: Artificial Intelligence
- NLP: Natural Language Processing
- DTW: Dynamic Time Warping
- API: Application Programming Interface

2. OVERALL DESCRIPTION
2.1 Product Perspective
RideMatch addresses the problems students face in finding affordable, safe, and socially fulfilling commuting options by using AI to match users for shared rides.

2.2 Product Functions (Summary)
- Social Profiling (NLP, Cosine Similarity).
- Route Matching (DTW).
- Personnel Selection (Collaborative Filtering).
- Driver Mode.
- Chatting Feature with ChatBot (Transformers).
- User Authentication.
- In-App Wallet.

2.3 User Classes and Characteristics
- Rider: Seeks to share rides for economical and social commuting.
- Driver: Offers rides by listing available routes and times.

2.4 Operating Environment
Mobile application.

3. FUNCTIONAL REQUIREMENTS
3.1 Social Profiling
FR1.1: The system must allow users to input their interests, preferences, and desired characteristics of travel companions during registration.
FR1.2: The system should utilize NLP to analyze and match user profiles based on descriptions.

3.2 Route Matching
FR2.1: The system must use Dynamic Time Warping to analyze and match routes, finding optimal shared travel routes.

3.3 Personnel Selection
FR3.1: The system should use collaborative filtering techniques to suggest potential ride companions based on mutual friends and interests.

3.4 Driver Mode
FR4.1: The system must provide a dedicated interface for drivers to view, accept, or decline ride requests.
FR4.2: The system shall allow drivers to broadcast their future departure schedules.

3.5 Chatting Feature With Inbuilt ChatBot
FR5.1: The system must support real-time messaging among users.
FR5.2: The system's ChatBot should assist users by providing automated responses and support.

3.6 User Authentication
FR6.1: The system must securely authenticate users using their mobile phone numbers.

3.7 In-App Wallet
FR7.1: The system's in-app wallet should enable users to securely store credits and make payments for rides.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Reliability
NFR1: The system should automatically recover from a failure within 2 minutes.

4.2 Usability
NFR2: The UI should be intuitive, allowing new users to navigate and perform essential tasks with minimal instruction.

4.3 Performance
NFR3: The system should respond to user requests within 2 seconds.
NFR4: Wallet transactions should be processed and reflected within 4 seconds.

4.4 Security
NFR5: Users' data must be encrypted and resistant to data leaks.

4.5 Scalability
NFR6: The system should scale effectively to accommodate user base growth without degrading performance.
