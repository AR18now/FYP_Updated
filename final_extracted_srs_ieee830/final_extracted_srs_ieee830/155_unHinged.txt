SOFTWARE REQUIREMENTS SPECIFICATION (IEEE 830)
Project: unHinged - A Procedurally Generated Game for Mental Health Awareness
================================================================================

1. INTRODUCTION
1.1 Purpose
This document outlines the requirements for a procedurally generated video game that blends entertainment with raising awareness about mental health issues (Schizophrenia, Nyctophobia, Alzheimer's) by portraying how they disrupt daily life.

1.2 Scope
To create a 2D, top-down, single-player survival game set in a mental asylum. The player's goal is to escape. The game features adaptive difficulty, procedurally generated levels, and is designed with guidance from mental health professionals to simulate real-world difficulties without promoting violent stereotypes.

1.3 Definitions, Acronyms, Abbreviations
- NPC: Non-Player Character
- FPS: Frames Per Second

2. OVERALL DESCRIPTION
2.1 Product Perspective
unHinged is an independent game that uses procedural generation to create unique levels and challenges each time, aiming to provide entertainment while fostering empathy and understanding for mental health struggles.

2.2 Product Functions (Summary)
- Procedurally generated levels (layout, challenges, objectives)
- 2D top-down perspective
- Survival gameplay (escaping asylum)
- Mental illness mechanics (challenges, hallucination/fear meters)
- Enemy NPCs with chasing behavior
- Inventory system with collectibles
- Main menu and settings
- Tutorial

2.3 User Classes and Characteristics
- Mobile Players: Seek games playable in short bursts on mobile systems.
- Explorers: Want to explore the game environment to find secrets or easter eggs.
- Achievers: Want to earn achievements, collectables, or bragging rights.

2.4 Operating Environment
Desktop PC game, designed to be lightweight and run on low-end systems, developed using Godot Engine.

3. FUNCTIONAL REQUIREMENTS
3.1 Module 1: Game Story & Prototypes
FR1.1: The Game Design Document shall detail blueprints for the project.
FR1.2: A Figma prototype shall show how the first few screens of the game may look.
FR1.3: The game shall be designed using Godot.
FR1.4: Assets shall be created using Blender.
FR1.5: A backstory tying into the game's themes shall be created.

3.2 Module 2: Basic Game Structure
FR2.1: The game must use 2D assets.
FR2.2: The game shall feature animations using 2D assets.
FR2.3: The game shall feature a main menu.
FR2.4: The player can start a new game.
FR2.5: The player can continue an old game.
FR2.6: The game shall have adjustable settings.
FR2.7: Level 1 shall be created.
FR2.8: The game shall have a tutorial.
FR2.9: The player can turn a torch on and off.
FR2.10: The level shall be procedurally generated.

3.3 Module 3: Advanced Game Structure
FR3.1: The game shall have NPCs.
FR3.2: Enemy NPCs shall chase the player.
FR3.3: The player shall experience different effects based on mental parameters.
FR3.4: The game shall have an inventory.
FR3.5: The game shall have items to collect and use.
FR3.6: The game must have sound effects for the atmosphere.
FR3.7: The game must have 3 levels.

3.4 Module 4: Test & Publishing
FR4.1: The game must be completable.
FR4.2: The game shall be published online for free.
FR4.3: The game shall be marketed to players.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Performance Requirements
NFR1: The game shall run at a stable 30-60 fps.
NFR2: Load times between levels shall be no more than 5 seconds.

4.2 Atmosphere Requirements
NFR3: The game shall convey the feelings and troubles associated with mental health issues through consistent use of assets and sound effects to create an oppressive, dark setting.

4.3 Replayability Requirements
NFR4: Procedural generation shall ensure that the player gets a different experience every time they run the game, increasing its lifetime.

4.4 Portability Requirements
NFR5: The game must be able to run on any system regardless of processing power, achieved by using 2D assets and the lightweight Godot engine.
