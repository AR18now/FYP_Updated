1. INTRODUCTION
1.1 Problem Statement
Urban commuting is increasingly costly due to under-utilization of private vehicles, contributing to traffic congestion and environmental degradation. Existing carpooling applications face challenges: limited user adoption, inefficient ride matching, and security concerns. Public transportation suffers from overfilling, fixed routes, uncomfortable conditions.

1.2 Scope
HumRahi is a mobile carpooling application with advanced route-matching algorithms, dynamic pricing model, real-time vehicle tracking, safety measures, female-only ride options, enhanced privacy settings, and real-time ride analytics using Apache Kafka.

1.3 Definitions, Acronyms, Abbreviations
- NLP: Natural Language Processing
- ETA: Estimated Time of Arrival
- MTBF: Mean Time Between Failures

2. OVERALL DESCRIPTION
2.1 Product Perspective
HumRahi provides affordable, environmentally friendly, and safe urban commuting options through sophisticated algorithms for dynamic pricing, real-time safety monitoring, route matching, and personalized user experiences.

2.2 Product Functions (Summary)
- User sign-up/login with verification
- Profile management
- Ride offer creation with route details
- Ride request creation with destination/timing
- Optimal ride matching based on proximity, route, preferences
- Real-time notifications
- In-app messaging
- Route optimization and navigation with traffic updates
- Dynamic pricing model
- In-app payment processing
- Real-time ride tracking and location sharing
- User ratings and reviews
- Driver license and vehicle verification
- Analytics dashboard
- Environmental impact reporting

2.3 User Classes and Characteristics
- Commuters: Primary users seeking affordable efficient transportation
- Drivers: Own vehicles, share rides to offset travel costs
- Female Commuters: Seek female-only ride options and enhanced privacy
- Administrators: Manage system functionality, safety compliance, user accounts

2.4 Operating Environment
Mobile application (Android/iOS), backend with Apache Kafka for real-time processing.

3. FUNCTIONAL REQUIREMENTS
3.1 UI/UX and Profile Management
FR1.1: The system shall allow users to sign up and log in using email, phone, or social media.
FR1.2: The system shall enable identity verification through email or phone.
FR1.3: The system shall provide profile editing and management.

3.2 Ride Matching and Booking
FR2.1: The system shall allow drivers to create ride offers (route details, availability, capacity).
FR2.2: The system shall allow passengers to create ride requests (destination, timing).
FR2.3: The system shall use algorithm to match ride offers with requests based on proximity, route, preferences.

3.3 Route Optimization and Navigation
FR3.1: The system shall provide route planning and navigation using mapping services.
FR3.2: The system shall provide real-time traffic updates and adjust routes.
FR3.3: The system shall calculate and display ETA and distance.

3.4 Dynamic Pricing Model
FR4.1: The system shall calculate estimated fare based on distance, demand, time before confirmation.
FR4.2: The system shall allow users to make payments using multiple options.
FR4.3: The system shall adjust pricing dynamically during peak times.

3.5 Safety and Security Features
FR5.1: The system shall allow real-time ride tracking and location sharing.
FR5.2: The system shall collect and display ratings and reviews.
FR5.3: The system shall verify driver licenses and vehicle information.

3.6 User Feedback and Support
FR6.1: The system shall provide in-app helpdesk and support options.
FR6.2: The system shall allow ride rating and feedback.
FR6.3: The system shall automatically analyze reviews using NLP for sentiment detection.

3.7 Notifications and Alerts
FR7.1: The system shall send push notifications for ride requests, confirmations, status updates.
FR7.2: The system shall send alerts for upcoming rides, cancellations, delays.
FR7.3: The system shall allow customizable notification settings.

3.8 User Analytics and Reporting
FR8.1: The system shall collect data on user preferences, ride patterns, feedback.
FR8.2: The system shall provide analytics dashboard for administrators.
FR8.3: The system shall generate reports on environmental impact and cost savings.

3.9 Real-time Ride Analytics and Processing Module
FR9.1: The system shall use Apache Kafka for real-time data collection on ride duration, distance, route deviations.
FR9.2: The system shall provide live analytics on traffic conditions during rides.
FR9.3: The system shall be scalable for increased live data volumes.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Reliability
NFR1: MTBF ≥ 1,000 hours.
NFR2: Server failure recovery mechanisms shall restore data within 5 minutes.
NFR3: Error logging with user notification within 1 minute of detection.

4.2 Usability
NFR4: New user sign-up < 2 minutes.
NFR5: Book a ride with ≤ 5 interactions from home screen.
NFR6: WCAG 2.1 Level AA accessibility compliance.

4.3 Performance
NFR7: Ride search results displayed within 2 seconds.
NFR8: System shall handle up to 1,000 concurrent users.
NFR9: Real-time traffic updates latency ≤ 5 seconds.

4.4 Security
NFR10: User authentication with minimum 8-character password.
NFR11: Account locked for 30 minutes after 5 consecutive failed login attempts.
NFR12: Sensitive data (payment, personal identification) encrypted with AES-256.
