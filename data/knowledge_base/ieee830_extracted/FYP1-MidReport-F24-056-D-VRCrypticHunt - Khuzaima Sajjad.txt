1. Introduction
1.2 Scope
The goal of this project is to create a puzzle-solving and exploration-based virtual real-
ity game that will captivate players in a 3D world while they search for hidden treasure
chests that are encrypted. To unlock the contents of a chest, the player must solve a
straightforward puzzle, which can be found inside the chest. The primary features of the
VR environment include movement, interactions with objects, and an easy-to-use user in-
terface. Additionally, the project will have varying degrees of difficulty, enabling a steady
rise in the intricacy of the riddles and the locations of the hidden jewels. This scalability
makes sure that players with different skill levels may still enjoy the game and challenges
those who are more experienced. The game’s immersive quality will also be improved
by visual and aural enhancements, with special focus paid to performance optimisation
for seamless VR operation. The project’s limitations include creating a wellorganised
gaming environment, putting in place fundamental VR controls, and making sure that
riches are encrypted. Advanced multiplayer capabilities and extremely intricate encryp-
tion schemes, however, are outside the purview of this document. The project’s main goal
will be to produce a strong, captivating single-player game with the potential for expan-
sion in the future. As development progresses, the game’s scalable design will enable
future updates, new stages, and increasingly challenging riddles. The end product will
be a well-executed, functional VR game that showcases both technological expertise and
imaginative design.

3.3 Data Design
The VR scavenger hunt system’s information domain is organized into data entities that
stand in for the key parts of the game, such as players, objects, treasure chests, puzzles,
and ambient features. Specific properties and associations are defined for each object to
enable effective data processing, organization, and storage.
33
3. System Overview
3.3.1 Major Data Entities
3.3.1.1 Player
Attributes: Player ID, Name, Current Location, Inventory (list of Item IDs), Progress
(current quest or level).
Storage: Player data is stored in a relational database table called Players, which maintains
unique player records and their associated progress.
3.3.1.2 Item
Attributes: Item ID, Name, Type (e.g., food, tool), Description, Quantity, Location (cur-
rent or initial).
Storage: Items are stored in a table called Items, which allows for efficient retrieval and
manipulation of item-related data.
3.3.1.3 Puzzle
Attributes: Puzzle ID, Description, Difficulty Level, Solution (encrypted), Associated
TreasureChest ID.
Storage: Puzzle data is organized in a table called Puzzles, linking each puzzle to its
corresponding treasure chest.
3.3.1.4 TreasureChest
Attributes: Chest ID, Location, Status (locked/unlocked), Associated Puzzle ID.
Storage: Treasure chests are maintained in a table named TreasureChests, which tracks
the status and associations with puzzles.
3.3.1.5 Hazard
Attributes: Hazard ID, Type (e.g., environmental danger, enemy), Location, Severity
Level.
Storage: Hazards are recorded in a table called Hazards, detailing their type and effects
on the player.
34
3.3 Data Design
3.3.2 Data Storage and Processing
The system stores and retrieves data elements in an organized manner using a relational
database management system (RDBMS). Foreign key relationships link each table, guar-
anteeing data integrity and allowing for sophisticated queries to extract insights like in-
ventory control and player progress.
3.3.2.1 Database Management System
PostgreSQL will be used by the system as the database management system, enabling
comprehensive data administration and handling. Each table in the database—Players,
Items, Puzzles, Treasure Chests, and Hazards—is intended to store the characteristics of
a different type of data entity. SQL queries are used to retrieve and manipulate data,
enabling effective data operations during games and interactions.
Data Storage:
We will use Firebase for user authentication and real-time data synchronization in addition
to PostgreSQL. Cloud-based storage solutions can be seamlessly integrated with Firebase,
enabling the safe storing of inventory data and user progress.
3.3.3 Data Organization
The object-oriented design of the system is reflected in the organization of data into enti-
ties, where each class is mapped to a database table. This mapping makes it possible for
the application logic and data storage to integrate seamlessly, making tasks like updating
player progress, adding things to inventory, and solving puzzles quickly and easily.
