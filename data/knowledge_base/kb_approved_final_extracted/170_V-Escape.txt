1. INTRODUCTION
1.1 Purpose
This document provides the skeleton of the V-Escape project, a horror game with procedural map generation and a compelling narrative, intended for all stakeholders (panel members, supervisors, designers, players).

1.2 Scope
To develop a horror game where players must solve puzzles to escape a procedurally generated environment. The game features: procedurally generated maps with diverse variations, puzzle generation (labyrinths, weight puzzles, memory puzzles), an AI agent that chases the player, difficulty scaling, and a narrative that unfolds as the player progresses.

1.3 Definitions, Acronyms, Abbreviations
- 3D: Three-Dimensional
- UI: User Interface
- ERD: Entity-Relationship Diagram
- EERD: Enhanced Entity-Relationship Diagram

2. OVERALL DESCRIPTION
2.1 Product Perspective
V-Escape is a single-player horror game that uses procedural generation to create unique, replayable experiences, focusing on exploration, puzzle-solving, and evasion.

2.2 Product Functions (Summary)
- Diverse Map Variations (different settings, layouts, themes)
- Procedural Map Generation (random layout, logical coherence, item placement)
- Puzzle Generation (sequenced locked/unlocked rooms, item scattering)
- Narrative Creation (character story, player engagement)
- Player movement, object interaction, inventory system

2.3 User Classes and Characteristics
- Casual Players: Prefer easy-to-understand mechanics and light entertainment.
- Hardcore Players: Seek deep, challenging experiences and complex puzzles.
- Puzzle Enthusiasts: Drawn to games with complex puzzle mechanics.
- Horror Enthusiasts: Seek tension, fear, and atmospheric settings.
- Speed Runners: Aim to complete games quickly, focusing on efficiency.
- Completionists: Aim to explore and achieve everything within the game.

2.4 Operating Environment
PC game developed in Unity.

3. FUNCTIONAL REQUIREMENTS
3.1 Diverse Map Variations
FR1.1: The game shall generate a variety of maps and environments on each run.
FR1.2: The game shall not generate the same maps in consecutive runs.

3.2 Procedural Map Generation
FR2.1: The system will place objects and puzzles in a way that fits naturally into the environment.
FR2.2: The procedural generation algorithm must create maps with a logical structure that balances randomness with gameplay requirements, ensuring each map is solvable.

3.3 Puzzle Generation
FR3.1: The system will ensure the locking of doors is in a sequence.
FR3.2: The items to unlock them are placed in the 'at the time' explored rooms to ensure puzzles are solvable.

3.4 Narrative Creation
FR4.1: The narrative must be self-explanatory in terms of telling the players about the characters and the overall story.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Reliability Requirements
NFR1: The system should gracefully handle errors and exceptions, providing informative messages without crashing.
NFR2: The game should automatically save player progress at regular intervals or upon level completion.

4.2 Usability Requirements
NFR3: The game's user interface should be easy to navigate.
NFR4: The game should support basic accessibility options and simple control schemes.

4.3 Portability Requirements
NFR5: The game should be optimized for the target platform but designed with considerations for future portability.

4.4 Difficulty Scaling Requirements
NFR6: The game should efficiently handle increasing complexity in procedural generation and level difficulty without significant performance degradation.
