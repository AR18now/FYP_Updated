Currently in Elixir, this process of dictionary extraction from files is handled manually by
a data designer, who processes these files by identifying missing columns, interpreting the
data, and generating a suitable dictionary for it. The data received generally falls into four
main types–delimiter-separated files, fixed-length files, line data files, and XML files.
The main challenge is that these files often contain missing columns or lack a uniform
structure, requiring manual intervention by data designers to parse through and map the
data correctly. Each file type has unique requirements, which complicate the process of
dictionary extraction, slow down data analysis and data mapping, and hinder automation.
By leveraging machine learning and generative AI, this entire process can be automated.
There is a need to create algorithms that can detect missing columns, identify patterns,
and dynamically interpret the schema, eliminating the need for manual labor. This will
streamline the data and ensure that there is consistency and integrity across the entire
system. It will also give way to automation of customized documents like certificates or
reports by creating templates that map data from files using structured dictionaries.
1.2 Scope
The scope of this project is to develop a user-friendly Web App, IntelliDatum, which is
integrated with a self-developed cloud-based backend service. The service is developed
to extract structured information from heterogeneous data files. IntelliDatum will serve
as an interface for users so they can upload numerous files of varying file formats and
types, including delimiter-separated files, fixed-length files, XML files, and unstructured
line data files. The service will handle diverse data types such as strings, dates, times,
numeric values, and booleans.
A core component of IntelliDatum is an AI model that learns to recognize patterns, in-
fer field boundaries, and group records across different file structures by analyzing the
context of the data within each file. The model will interpret and convert it into struc-
tured dictionaries, making the data uniform across formats. The model will be deployed
as a scalable cloud service. The application will feature advanced functionalities such
as schema validation, error detection, and the ability to process files containing single
records, multiple records, single documents, or multiple documents simultaneously.
Another feature of IntelliDatum is a communication and data mapping module. This
module will utilize the generated dictionaries to map data from input files to predefined
document templates, allowing users to easily create customized documents such as cer-
tificates, reports, and much more. Instead of building or fine-tuning an open-source LLM

client in the appropriate file format, such as an .epf (Elixir Product File).

3.3 Data Design
IntelliDatum receives input in the form of files that are categorized into four types of files.
These include:
1. Delimiter Separated Files: Files where fields are separated by delimiters such as
commas (,), pipes (|), or other characters.
2. Fixed Length Files: Files where fields have a fixed length, with each field starting
and ending at predefined character positions, and fields are separated by space.
3. Line Data Files: Files without any obvious structure, where each line may repre-
sent data points in different formats.
4. XML Files: Files with structured, hierarchical data where the schema is embedded
in the file.
These input files are processed to create a structured data dictionary which is stored in an
EPF file. The dictionary stores the metadata of the file that includes details like file type,
whether it is a multi-record or single record file, whether it is a multi-document or single
document file, and many more. Additionally, the dictionary also stores record names,
field names, and field data types in a standard format. This structured dictionary serves as
a foundation for subsequent processes like data mapping, insight generation, etc.

1. Delimiter Separated Files: The first step is to parse the file and identify the de-
limiter being used. Next, it is determined if the file is a multi-document or a single-
document file, and in the case of multi-document files, the document separator is
also identified. Then it is determined if the file is a multi-record or a single-record
file, and the number of varying records in a file is identified. Headers are identified,
if the file contains any, and mapped to the first record. Finally, record and field
names are generated, and a data dictionary is created.
2. Fixed Length Files: The first step is to parse the file and identify the record and
field lengths. Next, it is determined if the file is a multi-document or a single-
document file, and in the case of multi-document files, the document separator is
also identified. Then it is determined if the file is a multi-record or a single-record
file, and the number of varying records in a file are identified. Headers are identified,
if the file contains any, and mapped to the first record. Finally, record and field
names are generated, and a data dictionary is created.
3. XML Files: Given that XML files already have a structured format, IntelliDatum
parses the file and extracts the dictionary from appropriate tags in the files.
4. Line Data Files: As line data files lack any proper structure, the records and fields
are identified by using the data position and context in the file. The appropriate
names are generated for these records and fields and stored in the form of a struc-
tured dictionary.
