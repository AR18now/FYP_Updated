2.2 High-Level Use Cases
Here are the high-level use cases for the Palantir360 project based on the analysis of its
documentation and workflow:
2.2.1 User Registration and Authentication
- Actors: Home buyers, renters, real estate agents, developers
- Precondition: The user must have access to the mobile app or VR system.
- Description: Users can create an account by providing personal details, such as name,
email, and password. After registration, users can log in to access the platform.
- Main Success Scenario:
• The user provides valid registration details.
• The system verifies and stores user information.
• The user successfully logs in.
- Alternate Scenarios:
• Registration with invalid information results in a failure message.
• Failed login attempts after multiple tries prompt the user to reset their password.
- Postcondition: Users gain access to the system’s features (e.g., virtual property tours).
2.2.2 Upload Property Images (Agent Module)
- Actors: Real estate agents, developers
- Precondition: The user must be logged in and have property images ready for upload.
- Description: Real estate agents can upload images of properties to the database, which
will be processed for 3D model generation.
- Main Success Scenario:
• The user selects and uploads high-quality images.
• The system validates and processes images for compatibility (resolution, focus,
lighting).
• Images are stored in the Firebase database for further use.

- Postcondition: Property images are successfully uploaded and stored.
2.2.3 Image Data Preprocessing
- Actors: System
- Precondition: Uploaded images must be available in the database.
- Description: The system processes the images, optimizing them for the 3D model
generation. The system identifies and corrects issues like blurriness, improper lighting,
and edge detection.
- Main Success Scenario:
• The system analyzes uploaded images.
• The system enhances image quality or flags problematic images for re-upload.

- Postcondition: Image data is optimized and ready for 3D model generation.

Description: The system retrieves processed images and uses Autodesk’s API / NERF

Main Success Scenario:
• The system retrieves images from the database.
• The AutoDesk / NERF API [22] processes the images and generates a 3D model.
• The 3D model is stored in the database.
Alternate Scenarios:
• Failed API call due to incomplete or corrupt data.
Postcondition: A 3D model of the property is generated and stored.

3.1 Architectural Design
The system follows a layered architecture, where the application is divided into distinct
layers for better scalability, maintainability, and separation of concerns. The presentation
layer handles the rendering of the VR environment, allowing users to navigate through
properties. This layer communicates with the business logic layer, which processes user
interactions, manages navigation, and retrieves property data. The data layer is responsi-
ble for storing and managing 3D models and property details, which are hosted on Fire-
base for scalable and secure cloud-based storage. The layered design ensures modularity
and dynamic retrieval of necessary data during user navigation.

All data is stored in a cloud-based Firebase database, ensuring scalability. The data design
prioritizes efficient retrieval and storage of 3D models, with indexing for quick access
during user navigation. Real-time updates, such as property details or environmental
