1. Introduction
1.1 Problem Statement
Aiming to improve the excess capacity that modern multi-core CPUs offer for parallel
processing, SpecBot undertakes a computing task that calls for high performance. Many
software engineers continue to write sequential code or employ such tools as code gener-
ation that do not take full advantage of parallel processing, and so performance for these
tasks is still mediocre. Even when the idea of parallelizing something is there, for some
reason it always neglects a certain aspect of the hardware configuration, making the idea
inefficient. SpecBot provides the necessary tools to transform the sequential processing
operating code into parallel processing operating code according to the user’s hardware
needs. This solution narrows the performance gap between the latest advances in hard-
ware technology and necessary software performance in order to enhance application
performance in high end computing dependent industries across the board.

2.3 Functional Requirements
This section describes the functional requirements of the SpecBot system, organized by
module.
2.3.1 Hardware Specification Module
• FR-1: The system shall extract and analyze hardware information from Unix-based

• FR-2: The system shall provide a mechanism to extract system information using

• FR-3: The system shall store and utilize the extracted hardware information for
optimizing code parallelization.
2.3.2 Static Code Analysis Module
• FR-4: The system shall generate Abstract Syntax Trees (AST) for input C/C++
code.
• FR-5: The system shall analyze the AST to identify regions suitable for paralleliza-
tion.
• FR-6: The system shall determine whether to use static or dynamic AST analysis
based on code complexity and size.
2.3.3 Data Dependency Analysis Module
• FR-7: The system shall identify shared data structures within the input code.
• FR-8: The system shall analyze dependencies between identified data structures.
• FR-9: The system shall highlight data structures that must be shared across threads
for proper parallelization.
2.3.4 Thread Enhancement Module
• FR-10: The system shall develop algorithms to determine optimal thread size for
each parallelizable loop.
• FR-11: The system shall implement mechanisms for dynamic thread adjustment
during runtime.
• FR-12: The system shall investigate and implement methods for prioritizing pro-
cess threads in the system queue.

or 30 days of continuous operation.
• In the event of a system failure, SpecBot shall automatically save the current state
of code analysis and notify the user, allowing for easy recovery and continuation of
work.
2.4.2 Usability
• The SpecBot Visual Studio Code extension shall allow users to initiate code paral-
lelization with no more than three mouse clicks or keyboard shortcuts.
• The web interface of SpecBot shall be designed to be intuitive, requiring no more
than 10 minutes for a new user to understand and start using its basic features.

3.2 Design Models
In contrast to the object-oriented approach, the procedural approach emphasizes functions
over objects. For SpecBot, the following design models apply:
• Activity Diagram: Captures the step-by-step flow from hardware specification ex-
traction to parallel code generation.
• Data Flow Diagram: Illustrates data movement through the system, from input

• State Transition Diagram: Models states of system modules managing backend

We chose the Procedural Approach as it is ideal for SpecBot because it:
• Streamlines Control Flow: Facilitates a series of sequential operations, simplify-
ing task management.
• Optimizes Hardware Utilization: Allows tight control over hardware interactions,
crucial for thread allocation.
• Ensures Efficient Execution: Reduces overhead by focusing on direct function
calls, resulting in faster execution.
This approach effectively transforms sequential code into optimized parallel code, making
it suitable for high-performance computing environments.

3.3 Data Design
3.3 Data Design
Our program uses the power of the Abstract Syntax Tree (AST) and a Data Dependency
Graph to identify potential areas for parallelization. This makes it easier to insert the nec-
essary OpenMP/OpenACC commands to achieve the desired results. The data structures
used are explained as follow:
3.3.1 Data Structures Utilized
3.3.1.1 Abstract Syntax Tree
An Abstract Syntax Tree (AST) is a data structure used in computer science to represent
the structure of a program or code snippet. By using the grammar of the programming
language, each block of code can be easily identified and converted into a tree format. It
allows nodes to be inserted into the tree more easier so it will allow use to convert the tree
back into its code form.
3.3.1.2 Data Dependency Graph
A dependency graph represents the flow of information between attributes in a parse tree.
It helps determine the evaluation order of these attributes. This allows us to decide how
many threads are needed for loop parallelization, ensuring that variables are not overwrit-
ten and the data structure’s integrity is maintained[22, 44].
3.3.2 Data Storage and Communication
