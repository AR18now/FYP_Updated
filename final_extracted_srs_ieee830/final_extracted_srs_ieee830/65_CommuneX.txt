Project: CommuneX - Neighborhood and Community Automation System
================================================================================

1. INTRODUCTION
1.1 Problem Statement
Residential communities face challenges with manual processes for property transactions, maintenance requests, event management, and communication, resulting in inefficiencies, lack of transparency, and delays.

1.2 Scope
CommuneX combines community tasks into a single platform enabling smoother operations including property buying/selling with blockchain documentation, decentralized voting system, shared resources, event management, communication and announcements, maintenance requests, user authentication and role management, and admin dashboard.

1.3 Definitions, Acronyms, Abbreviations
- Blockchain: Distributed ledger technology
- RBAC: Role-Based Access Control
- MFA: Multi-Factor Authentication

2. OVERALL DESCRIPTION
2.1 Product Perspective
CommuneX eliminates communication barriers, reduces operational delays, and guarantees an open, safe, and connected neighborhood environment by automating fundamental community tasks.

2.2 Product Functions (Summary)
- Property buying and selling with blockchain documentation
- Event management and recommendations
- Decentralized voting system
- Shared resources (offer/request services from neighbors)
- Community news and announcements
- Private messaging
- Submit maintenance requests with tracking
- User registration, login, role-based access
- Admin dashboard with comprehensive overview and user management

2.3 User Classes and Characteristics
- Resident: Use app to buy/sell property, vote, share resources, book events, communicate
- Admin: Manage system, oversee operations, user management

2.4 Operating Environment
Web application, blockchain integration, NoSQL database (MongoDB).

3. FUNCTIONAL REQUIREMENTS
3.1 Property Buying and Selling
FR1.1: The system shall allow residents to list properties for sale.
FR1.2: The system shall validate all fields in listing form.
FR1.3: The system shall store property details in database.
FR1.4: The system shall send notifications to admin for approval.
FR1.5: The system shall upload property details after admin approval.
FR1.6: The system shall allow residents to search for property based on price, location, size.
FR1.7: The system shall display resident search results.
FR1.8: The system shall allow residents to view property details (photos, descriptions).
FR1.9: The system shall allow residents to save properties.
FR1.10: The system shall provide filters to improve search results.
FR1.11: The system shall display property details and review for sale.
FR1.12: The system shall guide residents through payment processes.
FR1.13: The system shall record transaction on blockchain for confidentiality and security.
FR1.14: The system shall use blockchain for secure payments.
FR1.15: The system shall store property papers on blockchain.

3.2 Decentralized Voting
FR2.1: The system shall allow admin to create voting polls for community decisions.
FR2.2: The system shall validate all fields in voting form.
FR2.3: The system shall store data in database.
FR2.4: The system shall notify all residents on creating new voting poll.
FR2.5: The system shall deliver voting proposal to residents.
FR2.6: The system shall allow residents to view active voting proposals.
FR2.7: The system shall allow residents to cast their vote.
FR2.8: The system shall record voting on blockchain for security and transparency.
FR2.9: The system shall not allow voting again on same proposal.
FR2.10: The system shall allow admin and residents to view their vote.
FR2.11: The system shall calculate accurate voting results.
FR2.12: The system shall allow admin to announce voting results.
FR2.13: The system shall conclude voting results and make available to residents.

3.3 Shared Resources
FR3.1: The system shall allow residents to list resources for community use.
FR3.2: The system shall validate fields in resource form.
FR3.3: The system shall store details in database.
FR3.4: The system shall notify admin to approve resource listing.
FR3.5: The system shall list resources on admin approval.
FR3.6: The system shall allow residents to search for resources based on required criteria.
FR3.7: The system shall display search results.
FR3.8: The system shall allow residents to view details of each resource.
FR3.9: The system shall allow residents to save resources.
FR3.10: The system shall provide filters for easy search.
FR3.11: The system shall allow residents to book a resource.
FR3.12: The system shall display resource details and allow residents to review.
FR3.13: The system shall allow residents to decide booking period.
FR3.14: The system shall notify admin to review and approve resource.

3.4 Event Management
FR4.1: The system shall validate user credentials during login.
FR4.2: The system shall display available venues, event dates, and options for booking.
FR4.3: The system shall check and confirm availability of selected venues.
FR4.4: The system shall send booking confirmation to user upon successful reservation.
FR4.5: The system shall suggest alternate time slots or nearby venues if selected option partially available.
FR4.6: The system shall notify user if no venues available for selected date and suggest other options.
FR4.7: The system shall store user event preferences for future recommendations.
FR4.8: The system shall display event recommendations based on user preferences.
FR4.9: The system shall suggest promotional events if no personalized recommendations available.
FR4.10: The system shall notify users if recommended event fully booked or canceled and provide similar alternatives.

3.5 Community News and Announcements
FR5.1: The system shall validate user credentials during login.
FR5.2: The system shall display announcements with full content, time, date, attachments.
FR5.3: The system shall optionally enable comments section for user feedback.
FR5.4: The system shall continue displaying previous announcement until new one posted.
FR5.5: The system shall display list of available members to message.
FR5.6: The system shall allow users to send, receive, and store messages, confirming successful delivery.
FR5.7: The system shall queue messages if recipient offline.
FR5.8: The system shall notify sender if recipient not available or deactivated account.

3.6 Maintenance
FR6.1: The system shall display form for users to submit maintenance queries or incident reports.
FR6.2: The system shall assign tracking number to submitted queries and notify users of successful submission.
FR6.3: The system shall validate any attached images or files and include with query submission.
FR6.4: The system shall notify users if query submission fails due to network or system errors and allow retry.
FR6.5: The system shall display current status of all submitted queries.
FR6.6: The system shall provide detailed progress updates ("in progress," "awaiting materials," "completed").
FR6.7: The system shall provide estimated completion times and reasons for delays if queries take longer.
FR6.8: The system shall allow users to provide feedback on resolved queries or reopen if issue persists.

3.7 User Authentication and Role Management
FR7.1: The system shall allow new users to register by entering name, email address, and password.
FR7.2: Users shall access system by entering password and registered email address.
FR7.3: Users shall reset password using "Forgotten Password" option.
FR7.4: The system shall validate fields for strong password (length, uppercase, numbers, special characters).
FR7.5: The system shall grant access to specific functionalities based on assigned role.
FR7.6: Admin shall add, modify, and remove user roles through system.
FR7.7: Sensitive modules and admin functions not accessible by non-admin users.
FR7.8: The system shall permit administrator to edit user accounts (password resets, account cancellation).
FR7.9: The system shall maintain log activity, especially at admin level.
FR7.10: The system shall provide role-based permissions (polling generation, list properties, access admin dashboard).

3.8 Admin Dashboard
FR8.1: Admin view shall provide overview dashboard showing activity and participation (active residents, events, property listings, maintenance requests).
FR8.2: The system shall supply emerging messages, notifications, or issues reported in real-time.
FR8.3: The system shall allow admin to sort and filter dashboard based on activity type, date, or specific users.
FR8.4: The system shall provide rapid access to important modules (voting polls, event management, property operations).
FR8.5: Critical activities (outstanding maintenance issues, upcoming voting dates) displayed on site.
FR8.6: The system shall have resource-sharing and property listings pending approval.
FR8.7: The system shall allow admin to export reports in PDF, Excel, or other file types.
FR8.8: The system shall enable admin to grant selective rights and switch user roles.
FR8.9: Admin shall search citizens using parameters like name, email, role, or account status.
FR8.10: The system shall enable community manager to add or remove members from group.
FR8.11: The system shall enable admin to provide specific permissions and switch user roles.
FR8.12: The system shall allow administrator to suspend or deactivate user accounts permanently or temporarily.
FR8.13: The system shall allow administrator to reset user passwords and alert them of change.
FR8.14: The system shall maintain audit log of administrator actions (role changes, account deactivation, password resets).

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Reliability
NFR1: MTBF ≥ 1,000 hours.
NFR2: System unavailable due to error in functionality (property listing, voting, resource sharing, communication) defined as failure.
NFR3: Backup mechanism to prevent single failures.
NFR4: Continuous log monitoring for error detection.
NFR5: Automated backup mechanism to recover single operation within 5 minutes.

4.2 Usability
NFR6: Users learn to use within 10 minutes.
NFR7: User-friendly interface with minimal effort.
NFR8: Retrieve property listing with single interaction.
NFR9: Step-by-step guide for new property purchase process.
NFR10: Error message display for incorrect or invalid details.

4.3 Performance
NFR11: System response to user requests within 3 seconds.
NFR12: Handle user increases without degradation.
NFR13: Handle peak loads.
NFR14: Use resources (RAM, CPU, GPU) efficiently.
NFR15: 90% of web pages load within 4 seconds over ≥ 20 Mbps internet.
NFR16: Handle at least 500 residents without performance degradation.
NFR17: Handle at least 50 transactions during peak time.

4.4 Security
NFR18: Encrypt sensitive data and secure in blockchain.
NFR19: Strong authorization and authentication methods (2FA, MFA).
NFR20: Role-based access control.
NFR21: Audit all transactions and trail-related activities.
NFR22: Standard algorithms to protect sensitive information.
NFR23: Multi-factor authentication for admin login.
NFR24: Maintain user logs with at least 6 months audit details.
