1. INTRODUCTION
1.1 Problem Statement
Small to medium-sized enterprises lack resources and expertise to implement efficient security controls, struggling to safeguard external assets due to complexity, cost, and resource-intensive nature of existing external scanning and vulnerability assessment solutions.

1.2 Scope
ProbeXpert is an automated, web-based platform for external asset scanning, vulnerability assessment, and compliance reporting for SMEs. Features include automated external scanning, real-time progress tracking, IP reputation evaluation, self-healing architecture, risk assessments, compliance reports, and dynamic updates as new risks emerge.

1.3 Definitions, Acronyms, Abbreviations
- SME: Small to Medium-sized Enterprise
- IP: Internet Protocol
- RBAC: Role-Based Access Control
- MTBF: Mean Time Between Failures

2. OVERALL DESCRIPTION
2.1 Product Perspective
ProbeXpert simplifies cybersecurity management for resource-constrained organizations through a user-friendly interface that identifies and ranks vulnerabilities in assets like IP addresses and domains.

2.2 Product Functions (Summary)
- On-demand scans of external systems
- Customizable scan parameters (Deep, Shallow, Individual)
- Environmental variables analysis for risk assessment
- Risk factor calculation (probability of occurrence, impact)
- Quantitative score categorization
- Vulnerability prioritization
- Live scan results with completion percentage
- Alerts and notifications
- Detailed vulnerability reports with risk score visualization
- Exportable reports (PDF)

2.3 User Classes and Characteristics
- System Administrator: Manage and configure platform, push new plugins, manage user access
- System (Plugin): Interact with external/internal APIs, ensure version tracking
- Website User: Manage infrastructure, perform scans, respond to vulnerabilities

2.4 Operating Environment
Web application, Docker-based worker nodes, Kubernetes for load balancing.

3. FUNCTIONAL REQUIREMENTS
3.1 External Asset Scanning
FR1.1: Users shall initiate on-demand scans on IP addresses and domain names.
FR1.2: Users shall select from three scan types (Deep, Shallow, Individual) with customization.
FR1.3: Users shall add or remove specific scanning tools to tailor scans.
FR1.4: System shall generate list of detected vulnerabilities with severity levels.

3.2 Risk Factor Calculations & Vulnerability Detection
FR2.1: System shall calculate risk factor for each vulnerability based on environmental variables.
FR2.2: System shall categorize vulnerabilities based on quantitative score (severity, likelihood, impact).
FR2.3: System shall prioritize vulnerabilities, displaying most critical first.
FR2.4: Users shall trigger new scans to detect recent vulnerabilities and update risk scores.

3.3 Real-Time Progress Tracking
FR3.1: System shall provide real-time updates on active scan progress (percentage completion).
FR3.2: System shall display estimated completion times based on scan type and progress.
FR3.3: System shall send alerts and notifications when scans complete or errors occur.

3.4 Reporting
FR4.1: System shall generate detailed vulnerability reports including risk factors.
FR4.2: System shall offer risk score visualizations (bar charts, pie charts, heatmaps).
FR4.3: System shall provide summary of organization's overall risk posture.
FR4.4: System shall allow users to export reports in PDF format.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Reliability
NFR1: Data integrity shall be preserved; no scan results lost in critical failure.

4.2 Readability
NFR2: Reports shall be understandable by IT team members and individuals with minimum IT knowledge.

4.3 Usability
NFR3: Tooltips and on-screen help shall reduce need for extensive training.
NFR4: System shall provide feedback for user actions within 2 seconds.
NFR5: Built-in trained chatbot shall answer user queries.

4.4 Performance
NFR6: Reports for completed scans shall generate within 10 seconds after scan concludes.
NFR7: Kubernetes shall provide automatic load balancing for best resource distribution.

4.5 Security
NFR8: Role-based access control (RBAC) with minimum 3 user roles (Administrator, Analyst, Viewer).
NFR9: All services shall operate independently in Docker containers for fault isolation.
