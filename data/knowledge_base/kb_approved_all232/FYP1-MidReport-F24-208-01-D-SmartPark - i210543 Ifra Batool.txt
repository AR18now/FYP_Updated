1. Introduction
1.2 Scope
The scope of this project involves creating an IoT Based Smart Parking Management
System with several key deliverables. First, we will install and configure IoT sensors
in specific locations within our model city to monitor parking spaces and provide real
time updates. We will also develop a mobile application where users can view available
parking slots, make reservations, and handle payments all in one place. Alongside this,
database integration with firebase will ensure that the data from the sensors is accu-
rately stored and updated, allowing for smooth operation. To ensure the system works
as expected, we will conduct a pilot deployment in our model city, which will serve as
a testing ground before considering broader use. Additionally, user documentation will
be created to guide both the users of the mobile application and those managing the IoT
infrastructure.
The project boundaries include an initial deployment in major urban areas of Pakistan,
targeting high demand locations like commercial zones, public facilities, and residential
complexes. The system will utilize existing IoT technologies without developing custom
hardware. During the pilot phase , the system will manage up to 1,000 parking slots,
scaling up as needed based on demand and capacity. The target audience includes urban
commuters, residents of large complexes, and visitors to public and commercial areas,
ensuring a broad user base. [44].

The IoT Based Smart Parking Management System combines hardware sensors for vehi-

updates, reservations, and payments. We’ve created an ERD for the database, use case

3.1 Architectural Design
The IoT based smart car parking system is designed with a multi-layered architecture for
separation and clear communication with each layer.
The first layer is the user interface, a mobile application with easy user interaction and
navigation.
• The app allows for key functionality to be visualized by users and it communicates
with backend to process the retrieved data.
The business logic layer deals with the inner fine working of the application where the
retrieved data is processed.
• It works on the reservation and monitoring functionality.
• Also works as a bridge between database layer and user interface layer.
The last layer is that of database layer where all data is stored.
• The database holds critical information such as parking availability, reservation his-
tory, and payment records.

3.3.2 Major Data/System Entities
Each major entity plays a specific role in the system, and they interact with each other
based on user actions and sensor inputs.
• Parking Slots:
– Stored: In a relational database, each parking slot is a row in a Parking_Slots
table with attributes: Slot ID, status, and sensor ID.
– Processed: The status of each slot is updated based on inputs from the sensor.
If a user reserves a spot, the status is temporarily locked.
– Organized: Indexed by Slot ID to quickly retrieve available slots or update
the slot’s status.
• Reservations:
– Stored: In a Reservations table, with references to the user and the parking
slot.
– Processed: When a user reserves a slot, a new reservation record is created,
and the slot’s status is marked as "reserved". After the reservation expires or
the parking session ends, the reservation is removed or marked as completed.
– Organized: By Reservation ID and User ID for efficient lookup of ongoing
and past reservations.

3.3.3 Databases or Data Storage Items
The system would require a combination of databases to store different types of informa-
tion:
• Cloud Storage (like Firebase): Could be used for storing the entire backend data,
handling both structured data (slots, reservations, payments) and real-time data
(sensor logs).
• NoSQL Database (e.g., Firebase Firestore, MongoDB): Ideal for storing semi
structured data like user profiles and handling real time updates for parking slot
statuses and reservations.
