RPL control Message Flow
RPL is used with the 6LoWPAN protocol to connect nodes via DODAG. It relies on four
types of control messages[2222]:
• DODAG Information Solicitation (DIS)
• DODAG Information Object (DIO)
• Destination Advertisement Object (DAO)
• Destination Advertisement Object Acknowledgment (DAO ACK)
The Flow of control messages are shown in figure 1.11.1 The DIO message helps other
nodes join the DODAG network. The root node initiates the transmission of the DIO,
and nodes that become part of the DODAG propagates DIO messages to other nodes.
The DIO message is periodically transmitted and can also be triggered by DIS messages.
It contains important information required for the construction and maintenance of the
topology, such as the RPL Instance ID, DODAG ID, version number, and rank. RPL
uses a trickle timer to manage the frequency of DIO messages[1111], ensuring efficient and
energy-saving communication.
When a device joins the DODAG and selects a preferred parent, it sends a DAO control
message to the parent node. This message is then forwarded towards the root node. DAO
messages[1919] move upwards toward the root node, opposite to the downward movement
of DIO messages. The root node processes the information contained in the DAO message
to recognize the node joining the DODAG.
In storing mode, the parent node appends its DAO information to the packet received from
the child node before forwarding it to the root node. In non-storing mode, the DAO packet
is immediately forwarded to the root node upon receipt from the child node. DAO ACK
messages, which are optional, are sent from the root node back to leaf nodes in a down-
ward direction. Initially, the root node establishes the DODAG topology by calculating
the required information and multicasts the DIO message to neighboring nodes. Neigh-
boring nodes select their parent based on the Objective Function (OF) [ 1616] and compute
their rank value[55].
Each preferred parent acts as a router between the child node and the root node. The
root node serves as the preferred parent for all initial nodes in the network. Nodes that
are one hop away from the root node transmit DIO messages downwards, while DAO
messages[66] are sent in response to the received DIO messages.

Rank Attack
In RPL (Routing Protocol for Low-Power and Lossy Networks)[1515], rank is a metric used
to define the position of a node relative to the root node (the destination). The rank of a
node determines its distance from the root based on a variety of factors such as hop count
or link quality. Nodes with lower ranks are closer to the root[ 66], and nodes with higher
ranks are farther away.

This ranking system helps in building the network’s tree structure, known as the Destina-

A Rank Attack [1616] in RPL occurs when a malicious node manipulates its rank to disrupt

than it should have, making it appear closer to the root node as shown in Figure 1.31.3.
In RPL, nodes select parents with lower ranks to establish the most efficient paths to the
root. By falsely claiming a lower rank[55], the malicious node deceives neighboring nodes
into choosing it as their preferred parent for forwarding data.
Once the malicious node is integrated into the network’s routing structure, it can cause se-
vere disruptions. These include creating routing loops where packets are forwarded in cir-
cles, leading to increased latency and energy consumption. Additionally, the attacker[2020]
can control a significant portion of the network’s traffic and perform other malicious ac-
tivities, such as dropping packets, delaying transmissions, or selectively forwarding data,
further degrading network performance.

tion accuracy in dynamic environments.

Proposed Approach
The TRank-IoT approach improves the reliability of data exchanges within an IoT net-
work. TRank-IoT operates in several key phases: Initialization, Data Transmission, Data
Reception, Trust Calculation, Rank Attack Detection, and Trust Update. Each phase is
illustrated in the Figure 3.13.1.

3. Proposed Approach
3.1 Assumptions:
To establish a clear understanding of the proposed TRank-IoT, the following assumptions
were made:
1. Initially, all nodes are authenticated and work cooperatively to construct DODAG
tree utilizing the RPL Protocol.
2. All node in network have same energy, processing power, interface, and communi-
cation range.
3. All data exchanges are protected by a lightweight, efficient cryptographic method.
4. There is a single server node that is reliable and and resource rich.
5. All nodes are either mobile or stationary other than server node.
3.2 Phase Description:
In the Initialization Phase, all nodes configure their initial parameters and form a com-
munication structure called DODAG using the Routing Protocol for Low-Power and
Lossy Networks (RPL).
Once initialized, the system transitions to the Data Transmission Phase . Each node
selects a parent node after verifying it against a "Malicious List" to ensure trust. The
parent node manages secure data transmissions across the network. This phase focuses
on maintaining secure communication by selecting a parent from trusted nodes. Each
node collects data from its environment, divides it into packets, and sends them to the
Server for processing. These data packets contain key parameters, which the Server uses
to calculate the node’s trustworthiness.
In the Data Reception Phase , all packets received at server node for processing and
extracting key information from them.
Simultaneously, during the Trust Calculation Phase, the trustworthiness of participating
nodes is recalculated based on their behavior and previous communications. Trust values
are continuously updated to detect any suspicious behavior, ensuring rapid identification
of potential threats.
In the Rank Attack Detection Phase , the Server continuously monitors the trust val-
ues of nodes present in network, identifying possible rank attacks by analyzing drops in
trust levels. If a node’s trust value falls below a set threshold, it is flagged as potentially
malicious and added to the "Malicious List."
42
3.2 Phase Description:
Lastly, in the Trust Updating Phase , trust values are revised based on the results of
trust recalculations and attack detection. Malicious nodes are added to the "Malicious
List," preventing them from participating in future data transmissions. This continuous
updating ensures the system’s resilience, quickly adapting to changes in node behavior
and isolating threats to the network’s integrity.

The TRank-IoT framework offers a robust and lightweight solution for detecting and mit-
