1. INTRODUCTION
1.1 Purpose
This document collects, analyzes, and defines the high-level needs and features of VirtuLectra, a web-based application designed to lessen the burden on teachers and provide interactive, engaging lectures for students.

1.2 Scope
To deliver a comprehensive AI-powered educational platform that supports real-time interaction, automated content creation, and student monitoring. It aims to reduce teacher workload by automating tasks like content generation and grading, while enhancing student engagement through real-time feedback.

1.3 Definitions, Acronyms, Abbreviations
- AI: Artificial Intelligence
- LLM: Large Language Model
- MCQ: Multiple Choice Question
- RBAC: Role-Based Access Control

2. OVERALL DESCRIPTION
2.1 Product Perspective
VirtuLectra is a web-based platform that acts as an AI teaching assistant, automating course material creation and providing interactive learning experiences for students.

2.2 Product Functions (Summary)
- Real-time interaction (voice exchanges, Q&A sessions)
- Automated content creation (lecture scripts/notes, tailored quizzes)
- Student monitoring (engagement tracking via pop-up questions)
- Web portal for teachers (dashboards for management, review) and students (interface for learning materials, quizzes, lectures)

2.3 User Classes and Characteristics
- Teacher: Uses the platform to manage classrooms, generate content, and monitor student progress. Wants to decrease their workload.
- Student: Uses the platform to attend lectures, access materials, and participate in quizzes. Wants interactive and engaging learning.

2.4 Operating Environment
Web-based application accessible via standard browsers.

3. FUNCTIONAL REQUIREMENTS
3.1 Real-Time Interaction Module
FR1.1: Upon detecting voice input, the system shall process and convert it into text.
FR1.2: The system shall implement a time-out mechanism to stop listening after inactivity.
FR1.3: The system shall notify the student if voice input cannot be processed.
FR1.4: The system shall provide visual feedback when the microphone is activated.
FR1.5: The system shall convert the lecture script into synthesized speech.
FR1.6: The system shall deliver synthesized speech verbally to students.
FR1.7: The system shall convert generated text-based responses to student queries into synthesized speech.
FR1.8: The system shall ensure conversion occurs in real-time with minimal delay.
FR1.9: The system shall provide visual feedback when synthesized speech is delivered.
FR1.10: The system shall ensure synthesized speech outputs are clear.
FR1.11: The system shall detect and identify the main subject or topic of a student's question.
FR1.12: The system shall break down complex or multi-part questions into smaller components.
FR1.13: The system shall recognize key terms related to the ongoing lecture.
FR1.14: Based on identified topics, the system shall generate an appropriate answer using an LLM.
FR1.15: If a question is determined to be vague, the system shall prompt the student for clarification.
FR1.16: If a valid response cannot be generated, the system shall prompt the student to rephrase.
FR1.17: The system shall maintain a history of all questions asked during the lecture.

3.2 Automated Content Creation Module
FR2.1: The system shall extract key points and summaries for topics using an LLM.
FR2.2: The system shall structure extracted content into organized lecture notes.
FR2.3: The system shall automatically create headings for major sections.
FR2.4: The system shall use bullet points to list key points.
FR2.5: The system shall include numbering where appropriate.
FR2.6: Based on extracted content, the system shall generate a lecture script using an LLM.
FR2.7: The system shall provide smooth transitions between different sections or topics in the script.
FR2.8: The system shall format the script in a step-by-step manner.
FR2.9: The system shall ensure the script follows a logical flow from introduction to conclusion.
FR2.10: The system shall adjust the length and content of the script based on specified time duration.
FR2.11: Based on topics, the system shall create relevant MCQs using an LLM.
FR2.12: The system shall ensure the quiz covers a balanced mix of topics and key points.
FR2.13: For each MCQ, the system shall generate at least three plausible distractors.
FR2.14: The system shall ensure distractors are conceptually similar to the correct answer.
FR2.15: The system shall avoid overly obvious incorrect answers.
FR2.16: After quiz submission, the system shall compare the student's answers with correct answers.
FR2.17: Based on correct answers, the system shall calculate the total quiz score.
FR2.18: If necessary, the system shall allow teachers to manually override automated grades.

3.3 Student Monitoring Module
FR3.1: The system shall present pop-up questions to students periodically during live lectures.
FR3.2: The system shall allow the student to respond to pop-up questions.
FR3.3: The system shall log the student's response to pop-up questions.
FR3.4: The system shall calculate an engagement score based on response speed and accuracy.

3.4 Web Portal Module
FR4.1: The system shall allow users to register and log in.
FR4.2: The system shall verify email addresses during registration.
FR4.3: The system shall allow users to reset passwords via email.
FR4.4: The system shall allow users to log in using OAuth services.
FR4.5: The system shall allow users to update personal information.
FR4.6: The system shall allow users to delete their accounts permanently.
FR4.7: The system shall allow teachers to create a new classroom with a unique code.
FR4.8: The system shall allow teachers to update classroom information.
FR4.9: The system shall allow teachers to upload course material (PDF, DOCX) for AI processing.
FR4.10: The system shall allow teachers to specify topics or chapters within the material to focus on.
FR4.11: The system shall allow teachers to delete classrooms.
FR4.12: The system shall allow teachers to generate, edit, and delete quizzes.
FR4.13: The system shall allow teachers to access student records (quiz scores, participation).
FR4.14: The system shall allow teachers to schedule, reschedule, or cancel lectures.
FR4.15: The system shall allow students to join classrooms by entering a valid code.
FR4.16: The system shall allow students to leave a classroom.
FR4.17: The system shall provide students the ability to attend scheduled lectures.
FR4.18: The system shall present lecture notes during the lecture.
FR4.19: During lecture, the system shall allow students to request permission to speak.
FR4.20: When a student requests to speak, the system shall complete the current sentence before prompting.
FR4.21: The system shall offer the student to proceed with text or voice.
FR4.22: The system shall provide students the ability to take timed quizzes.
FR4.23: The system shall provide students feedback on quiz performance (scores, correct answers).
FR4.24: The system shall provide students access to past quizzes and scores.
FR4.25: The system shall notify students when a quiz is available.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Reliability Requirements
NFR1: The system shall provide notification to administrators in case of downtime.
NFR2: The system shall ensure critical data is consistent and not lost during transfers.
NFR3: The system shall perform daily backups of all critical data.

4.2 Usability Requirements
NFR4: The system shall provide a simple, intuitive user interface with self-explanatory elements.
NFR5: The UI shall be fully responsive across various devices and screen sizes.
NFR6: The system shall promptly acknowledge user actions with visual or auditory confirmation.
NFR7: The system shall provide access to a help section.
NFR8: The system shall communicate errors to users with actionable suggestions.

4.3 Performance Requirements
NFR9: User inputs (voice or text) shall be processed with minimal latency.
NFR10: Any dashboard page shall load in less than 3 seconds under normal network conditions.
NFR11: The system shall allow course materials of up to 100 MB to be uploaded.

4.4 Security Requirements
NFR12: The system shall require users to create strong passwords.
NFR13: The system shall encrypt all sensitive data using industry-standard protocols.
NFR14: The system shall enforce role-based access control (RBAC).
NFR15: The system shall implement automatic account lockout after multiple failed login attempts.
NFR16: The system shall implement protection against common security threats (SQL injection, XSS, CSRF).
