## 1. Introduction

1. Introduction
1.2 Scope
The project aims to develop an advanced software platform that integrates with web pages
through a browser extension. The software is designed to detect and analyze video adver-
tisements, track user attention metrics by monitoring glances and engagement duration. It
generates analytics of areas of interest, extracted objects of interest, utilizes demographic
data, and emotion detection analysis to gauge user reactions and interest. The platform
offers data driven analytics and meaningful insights for companies like ad companies,
leading to better targeted adverts for users.[11]
1.3 Modules
1.3.1 Browser Extension + Host Machine Module
This module will run on the user’s host machine as a browser extension. It will be respon-
sible for:
• Detecting and Localizing Adverts.
• Tracking user Gaze.
• Tracking user Emotions and Demographics.
This module will locally process the captured frames of interest and send them to the
server, which makes up the Server Module.
Features:
• Ad Detection: The browser extension will use EasyList to detect and localize video
ads on the current web page so that their data can be corresponded with the gaze
tracking part.
• Gaze Detection: The browser extension will use WebGazer.js to monitor and detect
user gaze when the advert is being played.
• Emotion and Demographic Analysis: The browser extension will use DeepFace
to detect both user emotions and demographics (age, gender) while viewing the
advert.
• Data Extraction: The browser extension will extract frames of interest in the video
ad, combine them with heatmaps generated from gaze detection, as well as with data
from emotion and demographic analysis, and send frames to the server for further
processing and object detection.
2
1.3 Modules
1.3.2 Server Module
This module will process the frames sent over by the browser extension, extract meaning-
ful insights from them, and then deliver these insights to the analytics dashboard.
Features:
• Manage Advert Information:The browser extension will send ad metadata, which
the server will use to extract the video and related data. It will then use Object De-
tection to extract objects of interest in the video and keep their record.
• Object Detection: The object detection part of the server will parse over advert
videos and extract objects of interest and store them. The extracted objects will
later be used with the processed frames from the browser extension to extract key
objects of interest and other useful analytics.
• Processing Data: The processed data from the browser extension will be combined
with the extracted objects to produce the final processed output. Meaningful ana-
lytics will be extracted from this output, which will be stored and can be accessed
anytime by the analytics dashboard.
1.3.3 Analytics Frontend Module
This module will provide a GUI where advertising companies and stakeholders will be
able to view the gathered analytics of their posted ads.
Features:
• Ad Specific Analytics: The ad-specific part will offer a detailed view of the ad
performance of a specific ad. It will provide metrics on ad performance, such as
users’ glance duration, heatmaps, objects of interest from the user’s perspective,
and their emotions and demographics.
• Report Generation: Generate detailed reports on the performances of all ads
posted by the advertising company. Provide an analysis of each ad and how well
users are responding to it, and what is the common object of interest for most view-
ers.
• User Demographics: View all your ad viewers’ demographics such as age and
gender, as well as their emotional responses while watching the ads, allowing ad-
vertising companies to make content specifically tailored to their consumer market.

3.1 Architectural Design
Figure 3.1: System Architecture
The architectural design diagram outlines the structure of our video advertisement analy-
sis platform, detailing how various modules and technologies work together to deliver the
desired functionality. It illustrates the flow of data, components, and processes from user
interactions to the analytics provided to advertisers.
Browser Extension & Host Machine Module: This module operates locally on the
user’s device, detecting ads on web pages using EasyList. It tracks user gaze with We-
bGazer.js and captures emotions and demographic data with DeepFace. By processing
video frames locally, we ensure that raw user data, especially facial images, is handled
on the client side for privacy. The extension extracts key information like gaze duration,
emotions, and demographics, securely transmitting it to the server for further processing.
13
3. System Overview
Server Module: The server receives the processed data (frames, gaze tracking data, emo-
tions, and demographics) from the browser extension. It employs object detection algo-
rithms (like YOLO) to identify objects in video ads that may have captured user attention.
The server combines user interactions with detected objects, generating insights about
which elements attracted attention and how users emotionally responded. This processed
data is then stored in the server-side databases for further analytics and reporting.
Analytics Dashboard: The dashboard serves as the interface for advertisers to view gen-
erated reports and insights. It pulls data from the server, offering detailed analytics, in-
cluding heatmaps to indicate where users focused their gaze, emotional responses, de-
mographics, and object detection results. Advertisers can filter and analyze this data to
optimize future ad campaigns, enhancing engagement and targeting based on user behav-
ior.
Data Flow: Data flows from user interactions with video ads to the browser extension,
where gaze, emotions, and demographic data are captured and partially processed. This
data is sent to the server for object detection and further analysis before being displayed
on the analytics dashboard. Throughout the architecture, we prioritize privacy, ensuring
sensitive data like facial information is processed locally on the user’s device and not
shared with the server.

3.2.3 System Sequence Diagram
The integrated System Sequence Diagram (SSD) illustrates the interactions between four
key components: the Web User, Browser Extension, Server Module, and Analytics Fron-
tend. The process begins when theWeb User navigates a webpage, prompting theBrowser
Extension to monitor and detect any video advertisements using the EasyList technology.
Upon identifying an ad, the extension localizes and collects the ad metadata, sending this
information to the Server.
When the Web User watches a video ad, the Browser Extension tracks user engagement
by initiating WebGazer .jsfor gaze tracking, generating heatmaps, and usingDeepFace for
demographic and emotion detection. The resulting gaze data, emotional responses, and
demographic insights are tagged with timestamps and sent to the Server.
The Server Module performs object detection on the ad frames using the YOLO model,
identifies key objects, and combines the detected objects with the user engagement data.
This information is stored for further analysis.
Finally, the Advertising Company accesses the Analytics Frontend to log in and view the
performance metrics of specific ads. The advertiser can request ad-specific data such as
gaze heatmaps, demographic information, and emotion metrics. The system generates
comprehensive reports that can be downloaded or exported, providing valuable insights
for optimizing future ad content and targeting strategies.
