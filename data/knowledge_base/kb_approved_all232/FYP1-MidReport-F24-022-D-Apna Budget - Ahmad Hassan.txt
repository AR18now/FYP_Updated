## 1. Introduction

1. Introduction
1.2 Scope
Our application is focused on the Pakistani market. Our application will not be focused
on transactions. We will not be dealing with any money or wallets. Furthermore, our
application will be focused on basic predictions
We will focus on income and spending graphs and trends of user activities. From that
we’ll generate predictions and suggest appropriate actions. Our application will not be
focused on investments.
1.3 Modules
We are building a client mobile application that will help users in Pakistan manage their
finances effectively. Our four main modules are:
1.3.1 Module 1: SMS and Notification Reading For Automated Trans-
actions
This module focuses on user authentication as well as the automatic sms and notification
parsing to enter transactions. It includes secure login ability, manual transaction entry,
and the ability to categorize these transactions
1. User Authentication
2. SMS and Notification Reading for Automatic Transaction Retrieval
3. Ability to Manually Add Transactions
4. Ability to Categorize Transactions
5. Creating Budgets for Different Transaction Categories
6. Data Encryption and Masking for Sensitive Information
1.3.2 Module 2: Uploading Bills and Receipts via Images
In this module users with be able to upload images and screenshots of receipts and bills,
which will also be automatically processed and categorized just like with the SMS and
notifications feature. It also includes sorting and filtering, along with notifications related
to categorization, budgets, and transactions.

Following are the requirements for module 1:

• FR 2) SMS and Notification Reading for Automatic Transaction Retrieval
– FR 2.1 The system shall automatically detect and read SMS messages con-
taining the transaction details.
– FR 2.2 The system shall automatically detect and read notifications containing
the transaction details.
– FR 2.3 The system shall extract relevant transaction information (amount,
time, merchant name, and date) from the SMS and notification messages.
– FR 2.4 The system shall categorize the extracted transactions based on prede-
fined categories such as spendings, earnings, utilities, and bills.
– FR 2.5 The system shall display a notification to the user after a successful
transaction has been read and categorized.
– FR2.6 The system shall allow users to manually verify and edit the transaction
details.
– FR2.7 The system shall update all interlinked information such as user plans
when a new transaction is read
• FR3) Ability to Manually Add Transactions
– FR3.1 The system shall provide an interface for users to manually add transac-
tion details such as amount, date, time, merchant, category, and offline/online
transaction
– FR3.2 The system shall allow users to select from predefined categories when
adding a transaction manually.
– FR3.3 The system shall update all interlinked information such as user plans
when a new transaction is entered manually
• FR4) Ability to Categorize Transactions
– FR4.1 The system shall automatically categorize transactions based on key-
words and patterns in the transaction message.

2.5 Non-Functional Requirements
This section specifies nonfunctional requirements.
• NFR 1) Login and Registration
– NFR 1.1 The registration process shall be completed within 5 seconds after
the user submits their details.
– NFR 1.2 The system shall ensure a secure login process with response times
of less than 3 seconds.
– NFR 1.3 The system shall comply with security best practices to protect user
credentials.
• NFR 2) SMS and Notification Reading for Automatic Transaction Retrieval
– NFR 2.1 The system shall retrieve and process SMS and notification transac-
tions within a minute of receiving the SMS.
– NFR 2.2 The system shall be capable of parsing transaction details from at
least 95% of all SMS and notifications.
– NFR 2.3 The system shall ensure the privacy and security of SMS and notifi-
cation data by encrypting transaction details both at rest and in transit.
– NFR 2.4 The system shall attempt to differentiate between online and in-
person transactions wherever possible, otherwise, it will allow the user to add
this manually.

2.5 Non-Functional Requirements

– NFR 3.3 The system shall provide confirmation feedback to the user upon

– NFR 3.4 The system shall maintain a history of all manually added transac-

– NFR5.4 The system shall maintain budget history and performance data for

database, allowing users to review past budgets.

Chapter 3
System Overview
Apna Budget’s mobile app is built on Flutter, offering a cross-platform experience across
Android and iOS with offline support using SQLite. Firebase handles remote updates,
ensuring a dynamic user experience without the need for constant app updates. It’s all
about efficiency and user convenience.The backend uses Kubernetes to manage multiple
microservices, including user management, SMS processing, and budgeting. Nginx di-
rects traffic through an API Gateway, while MongoDB and Redis handle data storage and
caching. With Kafka facilitating message processing, the system is scalable and reliable.
Security is key, so Twilio provides OTP authentication, keeping user data safe. We rely
on the ELK stack for logging and monitoring, giving us real-time insights and helping
maintain system integrity and performance at all times.
3.1 Architectural Design
The Apna Budget system is divided into key modules: the frontend, backend services,
and the data layer, ensuring each part has a focused responsibility. The frontend, built on
Flutter, communicates with the backend via an API Gateway, making the user experience
seamless across Android and iOS. Backend services, including budgeting, SMS process-
ing, and user management, are hosted in a Kubernetes cluster, allowing for scalability
and isolated development. MongoDB stores persistent data, while Redis acts as a cache
to enhance performance. Kafka enables efficient communication between microservices,
keeping everything in sync. This modular design not only simplifies scaling but also im-
proves maintainability, ensuring each module can evolve independently without affecting
others.
