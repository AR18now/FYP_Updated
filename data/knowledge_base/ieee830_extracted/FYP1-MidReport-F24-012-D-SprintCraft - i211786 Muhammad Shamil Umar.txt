1.2 Scope
The project aims to create an intelligent software solution that automates the generation
of user stories from Software Requirements Specification (SRS) documents. By accept-
ing an SRS document and a user prompt as inputs, the software will generate contex-
tually accurate user stories. Key functionalities include parsing and analyzing the SRS,
maintaining context throughout the process, ensuring alignment with the specified re-
quirements and self evaluating the user stories. The software will allow customization
through prompts and will incorporate self-evaluation mechanisms to ensure accurate and
relevant user story generation. Additionally, the tool will maintain a history of interac-
tions for consistent output across sessions. The scope of the project includes developing
a user-friendly interface and a backend engine consisting of fine tuned LLMs working in
harmony.
1.3 Modules
1.3.1 User Input Module
This module allows users (e.g., Product Managers, Business Analysts) to upload an SRS
document along with prompts. It provides a user-friendly interface where users can
specify what kind of user stories they want to generate, whether it’s for functional, non-
functional, or specific sections of the provided SRS.
Key Features:
1. SRS document upload
2. Prompt field for user story specification
3. Input validation for ensuring correct file format and prompt structure
1.3.2 Natural Language Processing (NLP) Engine
The NLP engine processes the uploaded SRS document and interprets the user’s provided
prompt. It extracts key elements such as requirements, actors, and system behaviors from
the SRS and translates them into user stories. The engine ensures that technical language

This is the core module responsible for transforming the extracted data from the NLP en-

After generating user stories, this module allows users (e.g.,Product Managers, Business

users can modify the generated stories, add acceptance criteria, and provide feedback on
any necessary changes or introduce a perspective for context.
Key Features:
1. User story editing
2. Acceptance criteria addition
3. Feedback mechanisms for adjusting story generation rules

1.4 User Classes and Characteristics
User class Description
Product Manager A Product Manager is responsible for overseeing product develop-
ment. They will use the system to input an SRS document and
prompt for generating user stories. Product Managers require de-
tailed and accurate user stories that are aligned with business and
technical requirements.
Software Developer Software Developers use the generated user stories to implement fea-
tures and functionality. Developers rely on clear, technical user sto-
ries with well-defined acceptance criteria and may need the ability to
request revisions or more detailed descriptions.
Business Analyst Business Analysts bridge the gap between stakeholders and the de-
velopment team. Analysts refine the SRS before submission and use
the system to ensure user stories align with business goals. They may
work collaboratively to edit or expand stories as necessary.
QA Tester QA Testers use the user stories to create test cases and scenarios. A
QA Testers primary interaction with the system is to ensure stories
have clearly defined acceptance criteria and are testable.
Stakeholders Stakeholders review the output stories at a high level, ensuring they
align with the project’s vision. Their interaction is primarily for feed-
back and approval, as they may not be involved in the detailed cre-
ation or technical aspects of the stories.
Project Supervisor Project Supervisors monitor overall progress and ensure that user sto-
ries align with defined goals. They check stories for consistency with
the project scope and may use the system to generate reports or sum-
maries based on the stories.

2.3 Non-Functional Requirements
The non-functional requirements for this system define the critical quality attributes that
will ensure the system’s reliability, performance, and overall user experience. These re-
quirements are measurable, ensuring the system meets specific performance benchmarks,
scalability expectations, security standards, and user accessibility needs.
2.3.1 Performance
The system must deliver rapid processing to facilitate real-time usage. From processing
the input document and prompt to generating the users stories and then scoring generated
user stories, the entire process must be fast and precise. An estimated time of 10-15 sec-
onds is desirable. The system allow simultaneous users activities without any observable
delay or significant performance deterioration. This performance metric ensures that the
system remains efficient, even during periods of high demand.
2.3.2 Scalability
As the system is expected to handle growing user bases over time, it must be scalable. The
system should support multiple concurrent users, adapt to increased workload without

2.3 Non-Functional Requirements

mechanisms, such as adding more servers or cloud-based resources when necessary.The
system architecture must be structured to accommodate an increased volume of traffic
and/or document size, maintaining optimal performance.
2.3.3 Security
Data security is paramount for this system, given that it handles sensitive project docu-
mentation and proprietary information. All stored data, including SRS documents and
generated user stories, must be encrypted using AES-256 encryption, ensuring maximum
data protection. In addition, the system must implement secure user authentication meth-
ods, following OAuth 2.0 standards to protect user accounts and access controls. Role-
based access control (RBAC) will further ensure that different types of users (e.g., admins,
developers, and reviewers) have appropriate levels of access, preventing unauthorized data
manipulation.
2.3.4 Usability
For the system to be widely used/accepted, it must prioritize user friendliness. The user
interface will be designed to be intuitive and user-friendly, ensuring users can easily up-
load SRS documents and prompts, even without technical knowledge. Compliance with
WCAG 2.1 accessibility standards will be mandatory. Error handling within the system
must be comprehensive, providing users with clear, actionable feedback to correct input
mistakes without external support, thereby ensuring a seamless user experience.
2.3.5 Relibility
The system is expected to function continuously with minimal interruptions. To this end,
it will maintain a service uptime of 90%, ensuring availability for users around the clock.
In the event of a system failure, automatic failover mechanisms must be in place to restore
services within 1-10 minutes of downtime. Additionally, daily backups of user data to
safeguard against data loss or corruption will be implemented, ensuring the ability to
restore the system to a working state without significant data loss.
2.3.6 Maintainability
The system’s architecture must be designed for easy maintenance and future updates. The
codebase should adhere to SOLID design principles, promoting modularity and ease of
