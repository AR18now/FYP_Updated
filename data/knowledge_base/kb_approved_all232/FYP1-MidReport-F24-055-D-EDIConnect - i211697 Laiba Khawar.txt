Chapter 1
Introduction
EDI, which stands for Electronic Data Interchange, is the inter-company communication
of business documents in a standard format. It has been widely viewed as a technology
pivotal to supply chain management that has also provided benefits to firms on multi-
ple levels. It replaces paper-based documents such as purchase orders or invoices with
online documents. These EDI files usually follow a standard, which contains segments,
elements, and other contextual features allowing users to add their data accurately into
the specific sections.
Internationally, there are multiple standards being followed in different industries, for
example, Tradacoms, X12, EDIFACT, EANCOM, SWIFT, HL7, etc. In addition, some-
times some companies created their own custom standards for localised use. In B2B
(business-to-business) transactions, these EDI standards are quite helpful.
Furthermore, companies mostly use ERP systems (Enterprise Resource Planning) to man-
age and integrate the important parts of their businesses, to connect planning, purchasing
inventory, sales, marketing, finance, human resources, and more. So during the transac-
tions, the information being sent and received by the users is stored in the ERP systems.
Therefore, we can say that the information contained by the EDIs is fed into the ERP sys-
tems, and through this process, a company deals with their customers efficiently. Globally,
there are various ERP systems being used, like Microsoft Dynamics 365, SAP (Systems
Applications and Products), etc.
Hundreds and thousands of transactions are being done by the companies in a single
day. Having a seamless mechanism to handle the transactional data will be beneficial
for the companies. Internationally, there are more than 300 standards being used across
various industries. Therefore, having an interpreter of messages being exchanged by the
companies will play a crucial role. Therefore, generating mappings can allow users to
have a better knowledge of the data.
Moreover, if we look into the process of how these EDIs help customers, we’ll be able to

1.1 Problem Statement
In the business world, Electronic Data Interchange (EDI) plays a major role in inter-
company communication, but the diversity of EDI standards and custom formats creates
significant challenges. Information contained by the EDI files is fed into the Enterprise
Resource Planning (ERP) systems, and through this process, a company deals with their
customers efficiently. Globally, there are various ERP systems being used, like Microsoft
365, SAP, etc. Companies are required to convert the data into their own used EDI stan-
dard and map that data into the ERP systems.
These mappings take a lot of time and human energy; therefore, our project addresses
these issues by developing a system that not only translates EDI data from one interna-

1.2 Scope
Our goal is to reduce hindrances that occurred in business due to differences in EDI stan-
dards and ERP systems. It makes it convenient for the companies to apprehend the mes-
sage and generate a mapping that will make it easy for the user to enter the message into
their respective ERP systems.
Our tool, which is a customisable front-end, will provide the user with the ability to
easily comprehend the information being exchanged and will generate mappings from
one standard to another standard of EDI file, custom EDI files to standard EDI using
complex semantic mapping techniques and rule-based NLP techniques where necessary.
Additionally, we will also provide mappings for EDI to ERP systems in acceptable format,
but our tool won’t provide any sort of integration facility. Following are the objectives of
our project:
• Identify EDI standard or custom EDI file.
• Data pre-processing of EDI files sent by user.
• Generate mappings across standards.
• Generate mappings from custom EDI file EDI standard.
• Generate mappings for EDI file to user’s ERP system accepted format.
• User friendly interface

3.1 Architectural Design
With the high complexity of EDI standards and ERP formats having an architecture that
differentiates between each of the layers. Layered Architecture will clearly define how
mapping, LLM integration, validation, and user management are done all interact in struc-
tured layers, while keeping the design easy to understand and manage. Since there is no
such need for having independent scaling, therefore, there’s no such need left for mi-
croservices architecture. A layered architecture enables
1. Maintainability
2. Scalability
3. Reliability
The architecture is defined into 3 layers as mentioned bellow:
1. Presentation Layer:
(a) UI module: This module allows users to interact with the system. It will al-
low end users to upload their EDI documents, specify mapping requirements,
perform validations, and view the generated mappings. Additionally, users
can opt to use predefined templates for quicker and more efficient mapping
processes, providing a seamless and intuitive experience.
2. Business Logic Layer:
(a) Mapping engine: This is the core module of the system we are building, as it
generates the mappings using the pre-defined set of rules for all sorts of map-
pings. This module interacts with the LLM module for further enhancements
in the mappings generated.

3.2.1 Information Domain
The information domain precisely defines how and where data is collected. Since our
project lacked an explicit dataset, the data was gathered from authentic sources such as
stedi and HL7 websites [33] [11], which provided the necessary data standards and rules.
The collection includes relevant data, concepts, rules, and constraints that the system must
adhere to.
In our case, for an EDI application, the information domain consists of:
• EDI Standards: Various EDI formats like EDIFACT, X12, and their corresponding
fields, segments (or elements), transaction sets, and along with this EDI files are
also stored here.
• Business Rules: Rules that govern how data should be transformed or validated
between systems (e.g., mapping of invoice data between different formats).
• Mapping Templates: Rule-based or NLP-generated templates that define the map-
ping between different EDI standards.
• ERP Formats: Various EDI formats like Microsoft Dynamics 365, SAP, and their
corresponding fields and files.
• Custom EDI files: Diverse non-EDI files and their corresponding fields, without
any specific format followed.
The structure of the data hierarchy is represented as follows:
[ Standard Name] → [ Transaction Set Info] → [ Segments] → [ Elements]

3.2.2 Data Structures Transformation
Data structure transformation refers to the process of converting data from one format or
structure to another. This is crucial in EDI systems, as different standards or versions have
varying structures. Some types of data transformation include:
• Syntax Transformation: Converting data between various syntax (e.g., XML to
JSON, or X12 to EDIFACT).
• Schema Mapping: Mapping fields from one schema to another, ensuring that the
data conforms to the target system’s requirements (e.g., mapping fields from an X12
850 Purchase Order to an EDIFACT ORDERS transaction).
• Element/Segment Mapping: Transforming individual elements or segments within
a transaction set from one structure to another based on predefined or dynamic rules
(e.g., mapping "Invoice Amount" from a custom format to a standard format).
• Data Validation and Enrichment: Validating the data during transformation and
possibly enriching it with additional information (e.g., adding missing fields or con-
verting units).
