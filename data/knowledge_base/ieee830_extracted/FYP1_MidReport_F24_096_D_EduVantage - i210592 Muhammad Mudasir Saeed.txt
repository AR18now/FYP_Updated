1. Introduction
Furthermore, many students struggle to grasp how their distinct academic profiles fit into
various university programs. They lack direction on how their GPA, test scores, and aca-
demic history align with the requirements of various colleges. The process is further ham-
pered by the lack of personalization in existing platforms, which do not offer specialized

There is a strong demand for an automated, intelligent solution that may simplify the uni-

provide real-time updates on changing deadlines and admission requirements, giving stu-
dents the information they need to make sound judgments.
1.2 Scope
This initiative seeks to transform the higher education search process by focusing on the
top 150 universities in the United States, United Kingdom, and the European Union[ 1212]
. It would provide users with highly personalized recommendations based on their spe-
cific profiles, such as academic achievements, exam results, and job goals. The system
will evaluate and recommend the most appropriate university programs based on each
user’s individual requirements. It will also make thorough scholarship recommendations
to help customers identify appropriate financial aid opportunities. To guarantee that users
have access to the most up-to-date information, the platform will continuously monitor
university websites for updates and instantly refresh its database to reflect any changes
in program specifics, admission requirements, or deadlines.Users will benefit from com-
plete comparisons of universities, which include program options, fees, and rankings. An
interactive preference query system will enable users to dynamically refine their criteria,
allowing for more precise and informed decisions about their educational routes.

1.3.5 NLP-Based Query Analysis
Using techniques like Tokenization, Parts-of-Speech Tagging(POS), Named Entity Recog-
nition (NER) and using LLM, this NLP-Based module will be implemented.[1313]
1. User Query Processing and Keyword Extraction
(a) Using NLP techniques such as NER the system will extract relevant entities
and keywords from the user prompt/input. This will allow user to input his
prompt without the need of a specific template.
2. Profile-Based Keyword Extraction
(a) Using Tokenization and POS tagging, the system will also extract relevant
information from the user profile such as CGPA, Bachelors, Preference of
region and universities etc if such entities and useful information is not written
by the user in the prompt they are giving.
(b) If the user overlaps this information within the query and makes changes to the
entities the system will make sure to cross-check and ensure the most updated
data (The user query) is considered.
3. Automated Prompt Generation Now we have the extracted keywords from the user
query and profile these unstructured keywords will be given to the LLM for prompt
generation. The pre-trained LLM will take this set of keywords and generate a
prompt again which will be given as input to the fine-tuned LLM for searching/Analysis
for providing user universities that best fit the users profile and preferences.

(b) Annotate data to identify relevant terms like "Program Name," "Fee," "Appli-
cation Deadline," and "Admission Requirements."

1.3.7 LLM Precision Tuning of Processed Data
It focuses on fine-tuning the Large Language Model (LLM), specifically improving its
accuracy in determining the optimum fit for users based on their academic profile, pref-
erences, and university requirements. The purpose of this module is to ensure that the
LLM interprets user queries correctly and ranks colleges based on the tailored fit score,
utilizing the structured data generated by the NER model.
1. Initial Fine-Tuning on Processed Data
(a) Fine-tuning LLaMA or GPT-based models: Fine-tune a pre-trained LLM (such
as LLaMA, GPT-3, etc.) with structured data (e.g., program information, re-
quiremnets) and user profiles (GPA, test scores, preferences).
2. Fit Score Calculation
(a) The LLM is trained to generate the fit score by weighting significant variables
(such as GPA, exam results, and experience) [44].
(b) During training, the LLM learns how to normalize user input (such as GPA or
test scores) so that it may be properly compared to program criteria.
(c) The model is designed to handle various combinations of user criteria, dynam-
ically modifying the fit score in response to new input.
3. Ranking and Sorting of Programs
(a) The fine-tuned LLM ranks university programs in real time using the fit score.
The rating will change dynamically when the user adds new preferences or
updates their profile.
(b) The model gives a natural language explanation for each fit score, indicating
why some programs are a good fit. Like the following: "Your GRE score of
320 is above the minimum requirement of 310 for University X."
4. Integration with Retrieval-Augmented Generation (RAG)

(a) RAG provides additional instructions to the LLM to ensure that responses
match the most recent database information and user-specific demands [11].
(b) Incorporate external knowledge sources, including updated program specifics,

sponse.
1.3.8 Priority URL Retrieval Engine
This module focuses on retrieving relevant URLs and data based on the user queries and
profile information, specifically when the user selects the ’deep analysis’ option
1. Query Simplification Using LLM
(a) After the user writes a prompt and selects ’deep analysis’ this prompt as well
as the user data are sent to an LLM to simplify this query and generate a more
refined query to use on the search engine.
(b) GPT-4 will be used to generate a new refined query from the user query al-
ready provided.
2. TF-IDF Scoring for URL Ranking
(a) URLs will be ranked on the basis of their Term Frequency-Inverse Document
Frequency (TF-IDF) scoring.
(b) URLs will be ranked if they have a TF-IDF higher than 0.5.
(c) Scikit-learn [ 66] will be used to rank the web pages by implementing the TF-
IDF vectorizatoin.
3. Web Crawling to Identify Sub-URLs
(a) Puppeteer will be used for web crawling to discover sub-URLs which are more
in tune with the user query to scrape specific data.
(b) Specific sub-URLs will be added to a list for further extraction.
4. Data will be scraped from these sub-URLs NER will be applied on this data and the
resulting data will be used for further data processing and LLM Fine-Tuning.

This typically caters for graduate students, that is, students seeking admis-

student through his or her graduate application journey. The advisors help
students bring their academic and career goals in line with the most appro-
priate programs through fit score interpretation and program comparisons.
Advisors use the system to access student profiles, track application dead-
lines, and recommend programs based on eligibility and preferences. They
play an important role in counseling students on appropriate academic deci-
sions and help students meet deadlines and program requirements.

Admissions
Officers
University admissions officers represent various academic institutions and
are responsible for managing and reviewing student applications. Their in-
terest lies in ensuring that the programs offered by their institution are accu-
rately represented in the EduVantage system. They want to attract qualified
and eligible students who meet the program’s criteria, which helps maintain
high acceptance rates. Officers may also use the platform to track trends in
student preferences and ensure that program details such as fees, deadlines,
and admission requirements are kept up to date.
Parents/
Guardians of
Students
Parents or guardians of students play a supportive role in their child’s higher
education journey. They may not have the same level of technical expertise
as their children but are still invested in researching programs and keeping
track of application deadlines. Parents are particularly concerned about fi-
nancial factors such as tuition fees, scholarships, and overall program costs.
They use the system to assist their children in comparing programs and en-
suring that application deadlines are met.

System Overview
The EduVantage system’s architecture is built to effectively manage the complexity of
connecting students with appropriate graduate programs according to their choices and
characteristics. The Presentation Layer, Business Logic Layer, and Data Layer are the
three tiers of the presentation architecture used by the system. This division makes sure
that data management, business processing, and user interface management are all han-
dled separately yet work together harmoniously.
3.1 Architectural Design
3.1.0.1 Architectural Layout
• Presentation Layer: Users interact with the system at this front-end layer. A web-
based client that offers an easy-to-use interface for accessing all system features is
used to build it.
• Business Logic Layer: This layer functions as the system’s central processing
hub, where all choices, logic, and business rules are carried out. Serving as a bridge
between the Presentation Layer and the Data Layer, it executes application logic
and guarantees data integrity.
• Data Layer: the back-end layer, which manages, stores, and retrieves data. It
stores data in a schema-less fashion using MongoDB, a NoSQL database, enabling
flexibility in managing the range of data types utilized by the system.
