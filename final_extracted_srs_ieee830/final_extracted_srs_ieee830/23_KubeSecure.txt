1. INTRODUCTION
1.1 Problem Statement
Cyberattacks are becoming more frequent and sophisticated; estimated $12.5 billion lost to cyberattacks in 2023. Kubernetes clusters are prime targets for DDoS attacks and other malicious activities. Traditional detection methods cannot keep up with evolving cyber attacks.

1.2 Scope
KubeSecure is an AI-based anomaly detection system for Kubernetes clusters using deep learning models (Transformer-based, LSTM) to monitor network traffic and detect security risks. Features include real-time anomaly detection, automated threat mitigation (rate limiting, traffic redirection to honeypots), static rule writing for Cilium, YAML file misconfiguration scanning, and web/Android application for alerts.

1.3 Definitions, Acronyms, Abbreviations
- DDoS: Distributed Denial of Service
- LSTM: Long Short Term Memory
- YAML: YAML Ain't Markup Language
- RBAC: Role-Based Access Control

2. OVERALL DESCRIPTION
2.1 Product Perspective
KubeSecure provides flexible and comprehensive protection by learning continuously from new attack patterns, recognizing both known and unknown attack vectors.

2.2 Product Functions (Summary)
- Kubernetes cluster monitoring with Prometheus/Grafana
- Network traffic collection and duplication to deep learning model
- Real-time traffic analysis
- Automated threat mitigation (rate limiting, Cilium rule updates)
- Honeypot for traffic analysis
- YAML file scanning for misconfigurations
- Admin panel for rule management
- Android app for alerts

2.3 User Classes and Characteristics
- Security Team: Monitor cluster, respond to anomalies
- DevOps Engineers: Manage infrastructure, review YAML files
- Admin Users: Write Cilium rules, manage configurations

2.4 Operating Environment
Kubernetes cluster, Cilium for network policies, Prometheus/Grafana for monitoring, Android application.

3. FUNCTIONAL REQUIREMENTS
3.1 Anomaly Detection
FR1.1: Network traffic metrics shall be collected from Cilium.
FR1.2: Cluster and application metrics shall be collected from Prometheus.
FR1.3: Grafana dashboards shall show metrics from Prometheus and Cilium.
FR1.4: The system shall get streaming network traffic to deployed application.
FR1.5: Incoming traffic shall be duplicated to deep learning model pod.
FR1.6: Trained deep learning model pod shall be deployed for traffic analysis.
FR1.7: When malicious traffic batch detected, system shall issue alert.
FR1.8: Prometheus shall call webhook to run python script for rate limiting upon alert.
FR1.9: Python script shall dynamically update Cilium rules or call eBPF script.
FR1.10: Alerts shall be sent to Android application, Slack, and email.

3.2 Network Policies
FR2.1: The system shall detect simple DoS attacks (TCP Flooding) by tracking violations of static rules.
FR2.2: Upon recognizing rule violation, system shall rate limit incoming traffic.

3.3 Honeypot for Traffic Analysis
FR3.1: The system shall identify malicious traffic based on suspicious packet loads, aberrant traffic dynamics, attack origins.
FR3.2: The system shall forward malicious traffic batches to TPOT Honeypot for investigation.
FR3.3: Prior to forwarding, system shall feature metadata (source IP, detection time).
FR3.4: TPOT Honeypot shall generate enhanced analysis reports on traffic behaviors, attack trends, exploit vectors.

3.4 Yaml File Checker
FR4.1: The system shall scan uploaded Kubernetes .yaml files for configuration errors (RBAC misconfigurations, resource allocation issues, security statement errors, obsolete API versions).
FR4.2: The system shall identify misconfigurations by evaluating against industry standards.
FR4.3: The platform shall issue notifications (email, dashboard, system signals) for identified issues.

3.5 Web Application
FR5.1: The platform shall support web submissions of Kubernetes .yaml files for analysis.
FR5.2: Upon submission, system shall process and scan for misconfigurations.
FR5.3: Admin can view and setup Cilium rules via dropdown menu.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 User-Friendly
NFR1: System optimized for easy learning and operation.

4.2 Real-Time Data Processing
NFR2: System shall process real-time data with minimal delay.

4.3 Scalability
NFR3: System shall scale to support increased network traffic and monitoring needs.

4.4 Reliability
NFR4: System shall have redundancy and failover mechanisms for continuous operation.

4.5 Interoperability
NFR5: System shall be compatible with existing tools (CI/CD pipelines, monitoring systems, security tools).

4.6 Security
NFR6: Traffic analysis deep learning pod shall be isolated from external access.
NFR7: Under high-level DDoS attack, pod shall remain secure and stable.
