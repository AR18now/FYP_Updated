SOFTWARE REQUIREMENTS SPECIFICATION (IEEE 830)
Project: IntraSec - Client-less Network Risk Assessment and Asset Management Tool
================================================================================

1. INTRODUCTION
1.1 Purpose
This document defines the requirements for IntraSec, a client-less risk assessment and asset management tool designed to provide real-time inventory of network assets and identify security risks without requiring agent installation on individual machines.

1.2 Scope
To develop and deploy a client-less system for internal networks that discovers and inventories devices, services, and software; continuously tracks assets; integrates with the NIST vulnerability database to assess risks; assigns risk scores; generates actionable reports; and automates patch deployment. It also includes continuous monitoring, real-time alerts, and notifications.

1.3 Definitions, Acronyms, Abbreviations
- CVE: Common Vulnerabilities and Exposures
- CPE: Common Platform Enumeration
- NIST: National Institute of Standards and Technology
- API: Application Programming Interface
- RBAC: Role-Based Access Control
- MFA: Multi-Factor Authentication

2. OVERALL DESCRIPTION
2.1 Product Perspective
IntraSec is a centralized, automated security solution for internal networks, operating via microservices and RESTful APIs, designed to improve security posture and asset management.

2.2 Product Functions (Summary)
- Asset discovery and management (real-time inventory)
- Risk assessment (integration with CVE/CPE databases, risk scoring)
- Patch management (repository and deployment)
- Reporting (vulnerability summaries, recommendations)
- Alerts and notifications

2.3 User Classes and Characteristics
- Network Security Team: Uses IntraSec to assess vulnerabilities and take proactive measures for threat detection and mitigation.
- IT Administrators: Oversee installation, configuration, and integration of IntraSec with existing tools.
- System Administrators: Manage the technical side, including patch deployments and addressing vulnerabilities.

2.4 Operating Environment
Web-based application with backend microservices communicating via RESTful APIs, using Elasticsearch and PostgreSQL databases.

3. FUNCTIONAL REQUIREMENTS
3.1 Module 1: Assets Inventory
FR1.1: The system shall discover and display all devices, software, and services in real-time.
FR1.2: The system shall allow users to view, filter, and manage asset details through a front-end interface.
FR1.3: The system shall continuously track changes in asset details and update them in the PostgreSQL database.
FR1.4: The system shall integrate with Active Directory to centralize asset management.

3.2 Module 2: Risk Assessment
FR2.1: The system shall perform vulnerability scanning of assets using ElasticSearch.
FR2.2: The system shall continuously sync with the NIST Vulnerability Database API.
FR2.3: The system shall calculate and assign risk scores based on the severity of detected vulnerabilities.
FR2.4: The system shall provide a dashboard to display risk scores, vulnerability details, and remediation recommendations.

3.3 Module 3: Report Generation
FR3.1: The system shall generate standardized reports including identified vulnerabilities, severity, and risk score.
FR3.2: The system shall provide exportable reports for auditing and documentation.
FR3.3: The system shall generate detailed analysis reports with remediation recommendations.
FR3.4: The system shall allow users to view, filter, and export reports.

3.4 Module 4: Patch Deployment
FR4.1: The system shall automate the deployment of patches for identified vulnerabilities.
FR4.2: The system shall provide an interface to monitor the status of patch deployment.
FR4.3: The system shall notify users if manual intervention is required when automatic patch deployment fails.
FR4.4: The system shall report the progress and status of patch deployment.

3.5 Module 5: Alerts and Notifications
FR5.1: The system shall generate real-time alerts for detected vulnerabilities or security incidents.
FR5.2: The system shall allow users to customize alert thresholds for specific types of vulnerabilities.
FR5.3: The system shall provide an interface for users to view active alerts and notifications.
FR5.4: The system shall send notifications for vulnerabilities requiring immediate action.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Reliability Requirements
NFR1: The system shall aim for an up-time of at least 95% during operational hours.
NFR2: In case of failure, the system shall log the event and notify administrators within 5 minutes.
NFR3: The system shall recover from critical failures and resume operation within 10 minutes.

4.2 Usability Requirements
NFR4: Users shall be able to perform key tasks such as viewing asset information and generating reports in 3 interactions or fewer.
NFR5: 80% of new users shall be able to navigate and complete common tasks without needing external documentation after 10 minutes.

4.3 Performance Requirements
NFR6: The system shall be able to generate an asset inventory report for up to 100 assets within approximately 15 minutes.
NFR7: The dashboard page shall fully load within 5 seconds for 80% of users on a >10 Mbps connection.
NFR8: Database queries shall return results within 5 seconds for typical operations.

4.4 Security Requirements
NFR9: 80% of administrative users shall be required to use multi-factor authentication (MFA) for logging in.
NFR10: The system shall maintain logs of access to sensitive data for at least 3 months.
NFR11: The system shall lock user accounts after 5 consecutive failed login attempts, with notification sent to the administrator within 10 minutes.
