1. Introduction
1.1 Problem Statement
Mental Health is a topic that has gained much attention in recent years and is being taken
much more seriously. To coincide with this the portrayal of mental health in media is ob-
served with a higher degree scrutiny to prevent any negative stereotypes from developing.
Our team hopes to show the struggles of those with mental health issues and brain disor-
ders while also creating an experience that can be enjoyed by consumers. We also would
like to highlight the potential of procedural generation in game development. The use of
procedural generation would allow developers to create quality products while reducing
the workload and the costs needed. This is due to the increasing demands of quality and
scope of games in recent years which procedural generation is one solution to meet these
demands.
1.2 Scope
The scope of our project focuses on creating a procedurally generated video game that
blends entertainment with raising mental health awareness. The game offers players an
immersive experience where they will face challenges from the perspective of someone
dealing with mental health issues. Each game level will be designed with guidance from
mental health professionals to simulate the real-world difficulties faced by individuals
with these conditions. The game will be a 2D, top-down, single-player experience, aim-
ing to promote social awareness while delivering engaging gameplay. Set in a survival
environment, the player’s goal is to escape a mental asylum, but without any violent or
aggressive content, this is to combat the stereotype of mental illness leading to violent be-
haviors. The game’s adaptive difficulty will cater to different skill levels, allowing players
of all abilities to enjoy the experience while contributing to the message of empathy and
understanding.

2.3.2 Atmosphere
The game seeks to convey the feelings and troubles a person with mental health issues
goes through and the bleak setting of an asylum. In order to immerse the player in the
game, the tone and atmosphere must remain consistent. This will be achieved using the
assets created and the sound effects. These features must be coordinated to convey the
oppressive and dark setting of an asylum.
2.3.3 Replayability
The aim of procedural generation is not only to lessen the workload of developers and
reduce costs, but to also provide more entertainment. Since the levels are created at run
time, the player will always get a different experience every time they run the game. This
allows a game’s lifetime to increase since completing the levels does not mean the player
has seen all there is to see. Every time the player starts a new game or level, the layout of
the level, challenges and objectives will be slightly altered.
2.3.4 Portable
We are aiming for low end gaming and want out game to reach the broadest audience pos-
sible. To achieve this the game must be able to run on any system regardless of processing
power. The use of 2D assets will help achieve this, since they are less process heavy than
3D graphics. Godot’s lightweight nature also means it will be easier for older and weaker
systems to run the game.

3. System Overview
The project will be using the architecture as shown above. The architecture is component
based, each module is focused on certain functional requirements that need to be met.
• The main module is the Game engine which will be handling the major functions of
the game. It will be interacting with all the other modules in order to achieve this.
• The Game Assets will comprise the graphics and sounds the the engine will use to
create the game world and levels.
• The Game physics will be the light that the player controls to see around the level
and will interact with enemy AI.
• The Game AI handles the behaviours of the entities within the levels and how they
respond to the player’s actions.
• The story will dictate how each of the levels will be generated and using which
assets. The design of the assets will also be influenced by the narrative and setting.
Enemy AI will affect how the dialogue will play out, and the story in turn will
determine the enemy behaviour
• The Hardware Abstraction will be used to store the state of the game between ses-
sions and will handle the storage of assets. The input provided will tell the engine
how to move the player and what the AI should react to.
3.2 Design Models
Procedural Approach
Our project will follow a procedural methodology as this will fit the nature of how the
game will run. The game will constantly check the states of certain values and run the
needed processes.
The flow chart showcases the runtime processes of the game. Although this is a simpli-
fication of the actual flow of the game, the basic structure is similar. The game starts
and the player chooses to either load an old game or start a new one. If a new game is
started, a new level is generated along with NPCs and enemies. If an old game is loaded,
no generation needs to take place and the old state is loaded. The player is then spawns
into the game world and what follows depends on the player’s input. If the player inputs
escape, the game will pause and the option to save, exit or unpause are offered. If the
player inputs a direction, then the engine will update the player’s position on in the world.
When the player moves, the engine will also check to see if an enemy has spotted the
player, if so the enemy AI will begin the chase. This flow will follow a cycle until the
player reaches the exit, loses all sanity or pauses and exits.
