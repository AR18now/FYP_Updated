SOFTWARE REQUIREMENTS SPECIFICATION (IEEE 830)
Project: LeafSpec - AI-Powered Plant Species Identification from Leaf Images
================================================================================

1. INTRODUCTION
1.1 Purpose
This document outlines the requirements for LeafSpec, a mobile application that uses a custom-trained CNN to recognize plant species from leaf photos, helping botanists, researchers, and enthusiasts.

1.2 Scope
The project focuses on developing a mobile app (Flutter) that accurately classifies plant species from leaf images. It aims to be better than existing competitors in terms of accuracy, intuitiveness, and ease of use.

1.3 Definitions, Acronyms, Abbreviations
- CNN: Convolutional Neural Network
- UI: User Interface
- MTBF: Mean Time Between Failures

2. OVERALL DESCRIPTION
2.1 Product Perspective
LeafSpec automates the tiring and tedious task of manual leaf classification, reducing errors and increasing efficiency for professionals and enthusiasts.

2.2 Product Functions (Summary)
- Data Preprocessing (resizing, augmentation).
- Model Training (CNN).
- App Development (Flutter) with admin/user dashboards.
- Model testing and improvement.

2.3 User Classes and Characteristics
- End User: Botanists, researchers, or plant enthusiasts who use the app for species identification.
- Developer: Creates and maintains the app and models.
- Testers: Ensure the application is working properly.

2.4 Operating Environment
Multi-platform mobile application (iOS, Android).

3. FUNCTIONAL REQUIREMENTS
3.1 Data Preprocessing
FR1.1: The leaf image provided by the user shall be resized to a standard resolution accepted by the CNN.
FR1.2: Data cleaning shall be done to improve the model's accuracy.

3.2 Model Training
FR2.1: The system shall train the model on a preprocessed dataset of 10-15 species.
FR2.2: Model's testing and validation accuracy along with confidence scores shall be used to judge its overall accuracy.
FR2.3: The updated model shall be saved to the database.

3.3 User Interface (UI) development
FR3.1: The user interface shall be friendly, easier to use, and have intuitive navigation.
FR3.2: Output shall contain the identified specie along with confidence score and description.

3.4 Admin and User Dashboards
FR4.1: Admin Dashboard shall allow admins to add, remove, or delete species or perform maintenance.
FR4.2: The application shall have a user interface for identification.

3.5 Integration of Model with Application
FR5.1: The application shall be integrated with the CNN model so leaf images can be sent for classification.

3.6 Testing the Model
FR6.1: The model shall be regularly evaluated using a testing dataset.
FR6.2: An error message shall be provided by the system if the confidence score is less than 20%.

3.7 Improvement of Model
FR7.1: The model's accuracy should be improved by catering to more species and enhancing the dataset.
FR7.2: The system shall use feedback to improve the application.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Reliability
NFR1: The system will have an MTBF target of 500+ operation hours.
NFR2: A failure is defined as whenever uploading images failed, leaf classification failed, or a user did not get a result after 2 minutes.
NFR3: If an error is detected, re-processing of the image will be done.
NFR4: If a failure happens, the system shall automatically restart and indicate to the development team.

4.2 Usability
NFR5: The interface shall require a minimum number of clicks (approximately 3-5) to upload leaf images.
NFR6: Application buttons must be clearly labeled.
NFR7: The feedback mechanism shall not appear as a pop-up every time; there shall be an option.
NFR8: The application shall be accessible on iOS, Android, and Web.

4.3 Performance
NFR9: The custom-trained CNN shall be able to classify a specie within a few seconds.
NFR10: Data retrieval from the database shall be done within (5-10 seconds).
NFR11: The system shall store feedback submission right after the user clicks "Send Feedback".
NFR12: The system shall be fast enough to retrieve previous classification data within 5-10 seconds.
NFR13: Error messages shall be displayed as soon as the issue is identified.

4.4 Security
NFR14: The system shall encrypt all sensitive user data (passwords, email, user history).
NFR15: The software would be locked for the user for approximately 10-15 minutes if a user unsuccessfully tries to login more than 10 times.
NFR16: A captcha system shall be implemented to avoid bots.
