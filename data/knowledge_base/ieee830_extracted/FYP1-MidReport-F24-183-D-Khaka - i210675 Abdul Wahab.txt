2.3 Non-Functional Requirements
This section specifies nonfunctional requirements. These quality requirements should be
specific, quantitative, and verifiable. The following are some examples of documenting
guidelines.
2.3.1 Performance
App must ensure fast response times to maintain a smooth user experience. Image layer
generation should be processed within 60 seconds, even for high-resolution images. File
storage should be optimized to minimize load times without compromising quality.
2.3.2 Usability
App’s interface should be intuitive and user-friendly, guiding users through the sketching
process with clear instructions. Additionally the app should be responsive and functional
on all devices.

2.3 Non-Functional Requirements
2.3.3 Security
App should implement strong authentication measures such as two-factor authentication
to protect user accounts. Uploaded images and sketches must be handled securely, with
encryption for both data in transit and storage. The app should also include mechanisms
for content moderation to prevent the sharing of inappropriate or harmful material in the
art gallery.
2.3.4 Scalability
As the number of users and images increases, the app must be able to scale efficiently.
This requires using cloud storage solutions to handle the growing data volume and imple-
menting caching mechanisms to optimize frequently accessed content. Load balancing
should be in place to distribute traffic evenly, ensuring the system performs well even
during peak usage.
2.3.5 Reliability
The app should be available for use 24/7, ensuring that users can use it any time. It should
be fault tolerant and always ensuring that server issues or high traffic loads aren’t affecting
the services. Backups should also be created of users uploaded images to protect them
from being lost if system fails at any given time.
2.3.6 Maintainability
The app must be easy to maintain, with a modular code. Clear and thorough documenta-
tion is also necessary, especially for the APIs and app components.
2.3.7 Compatibility
The app should be compatible so all types of users with any type of browsers can use
it freely without any problems. Furthermore, it should be responsive so the site can be
viewed on desktop and mobile.

Chapter 3
System Overview
The system will be deployed in a client-server environment, with a tiered architecture
ensuring that each functionality is managed independently.
Functionality
This project will be a platform for users where they can upload, process, and interact
with images, turning them into multi-layered sketches. The system allows users to apply
artistic stylizations to their images, manage individual layers, receive feedback on their
work and get instructions on how to recreate the layers.
Design
The design follows a multi-tiered architecture, with three tiers:
• Presentation Tier: This includes the user interface and a caching mechanism. The
user interface is responsible for all interactions, including image uploads, feedback ,
and sketch previews. Caching improves performance by storing frequently accessed
data.
• Application Tier: This is where the business logic is implemented. It consists of
several modules:
– Authentication : Handles user logins.
– Image Processing : Applies artistic stylizations and transforms images into
layers.
– Layering : Breakdown of images into editable layers.
– Feedback and Recommendation Modules : Feedback and tool suggestions
to guide users.
– Instruction Module : Provides textual and auditory instructions to users .
21
3. System Overview
• Data Tier: This tier manages all data storage. User data and sketch metadata are
stored in a relational database, while large image files are kept in a file storage
system. This ensures efficient storage and fetching of both structured and unstruc-
tured data.
