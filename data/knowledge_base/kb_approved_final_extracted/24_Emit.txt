1. INTRODUCTION
1.1 Problem Statement
Organizations often prioritize external traffic monitoring, overlooking internal network threats. Insider threats, driven by disgruntled employees, are rising and considered more dangerous than external attacks. Since COVID-19, remote work has increased risk, with 73% of IT executives believing remote workers pose greater risk than on-site employees.

1.2 Scope
Emit focuses on monitoring and protecting internal networks. Features include agent creation for log consolidation, log preprocessing (Wazuh), web application for visualization, dataset creation for AI model training, static rule detection, and AI-based behavioral analysis for real-time insider threat detection.

1.3 Definitions, Acronyms, Abbreviations
- SIEM: Security Information and Event Management
- AI: Artificial Intelligence
- SOC: Security Operations Center
- RBAC: Role-Based Access Control

2. OVERALL DESCRIPTION
2.1 Product Perspective
Emit detects insider threats both statically and using AI models, with agents configured according to identified insider threat behaviors.

2.2 Product Functions (Summary)
- Insider threat behavior identification
- Agent creation and configuration
- Log consolidation from end systems
- Dataset creation and labeling
- Static rule detection
- AI model training
- Real-time behavioral analysis
- Web application dashboard
- Alert visualization
- Incident management
- Role-based access control

2.3 User Classes and Characteristics
- Security Analysts: Monitor system, respond to alerts, need real-time alerting and detailed logs
- IT Administrators: Install, configure, maintain system, need system management tools
- HR Personnel: Conduct employee conduct research, need simplified reports
- Compliance/Legal Officers: Ensure compliance with legal regulations, need audit trails
- Executive/Business Managers: Need high-level summaries and risk assessments

2.4 Operating Environment
Web application, MongoDB for data storage, Kafka for log streaming, Wazuh for log preprocessing.

3. FUNCTIONAL REQUIREMENTS
3.1 Module 1: Agent Creation
FR1.1: The system shall provide environment for creating monitoring agents that collect/generate logs.
FR1.2: The system shall enable scripting agent development using Python and PowerShell.
FR1.3: The system shall provide templates and examples for developers.
FR1.4: The system shall provide configuration file/interface for parameters (log types, collection frequency, filters, thresholds).
FR1.5: The system shall permit definition of agents for polling or real-time deployment.
FR1.6: The system shall have methodology for testing agents in production environment.
FR1.7: The framework shall contain ability to run and create scenarios to test agents.
FR1.8: The system shall provide facility for deployment on multiple end systems via command-line scripts.
FR1.9: Automated deployment options using Ansible, Puppet.

3.2 Module 2: Logs Management
FR2.1: The system shall have tools for retaining logs generated by deployed agents in central repository.
FR2.2: The system shall support Classification Shelves and Dunnage Warehousing Systems for varying capacity logs.
FR2.3: Clean logs by removing non-essential or damaged data elements.
FR2.4: Standardize log entries to maintain global order.
FR2.5: System shall have module for log preprocessing to support audits.
FR2.6: System shall have tools to detect threats using rule set.

3.3 Module 3: Model Training
FR3.1: The system shall employ labeled dataset for machine learning model training.
FR3.2: The system shall implement methods to efficiently prepare and load dataset.
FR3.3: The system shall incorporate performance evaluation of classifier algorithms (decision trees, random forests, SVM).
FR3.4: The system shall provide facility for setting model parameters and hyperparameters.
FR3.5: The system shall introduce training infrastructure for carrying out model training process.
FR3.6: The system shall provide dataset partitioning into training, validation, and test sets.
FR3.7: The system shall perform evaluation metrics (accuracy, precision, recall, F1-score, AUC-ROC).
FR3.8: The system shall offer means of illustrating evaluation results.
FR3.9: The system shall allow adjustments based on evaluation results.
FR3.10: The system shall allow integration of AI model based detection engine into main application.

3.4 Module 4: User Dashboards
FR4.1: The system shall provide customizable dashboards displaying real-time security alerts and performance metrics.
FR4.2: The system shall allow rearrangement and configuration of dashboard widgets.
FR4.3: The dashboard shall include widgets for detected threats, system health status, historical trends.
FR4.4: The system shall provide options for users to choose displayed metrics.
FR4.5: The system shall provide interactive visualizations (charts, graphs, heatmaps).
FR4.6: Visualizations shall allow drill-down into specific alerts.
FR4.7: Users shall filter visualizations by severity, type, time range.
FR4.8: The system shall provide access to detailed logs.
FR4.9: Users shall view logs in structured format.
FR4.10: The system shall include search functionality for specific log entries.
FR4.11: Users shall filter logs by severity, event type.
FR4.12: The system shall provide export functionality (CSV, PDF).
FR4.13: The system shall include tools for tracking and managing security incidents.
FR4.14: The system shall provide workflow for escalating incidents based on severity.
FR4.15: Users shall document incident details (actions taken, evidence).
FR4.16: The system shall provide features for assigning incidents to specific personnel.
FR4.17: The system shall automate workflows (notifying relevant personnel).
FR4.18: The system shall integrate with static rule detection and AI-assisted detection systems.
FR4.19: Integration shall provide real-time data feeds.
FR4.20: The system shall aggregate alerts from both detection systems.
FR4.21: The system shall implement role-based access control.
FR4.22: The system shall allow administrators to define roles and permissions.
FR4.23: The system shall require user authentication (username/password, 2FA).
FR4.24: The system shall send alerts and notifications immediately when threat detected.
FR4.25: Users shall configure notification preferences (alert types, channels).
FR4.26: The system shall maintain notification history.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Reliability
NFR1: MTBF ≥ 6 months.
NFR2: Failure defined as inability to gather, process, or interpret logs, or positive/negative risk assessments impairing performance.
NFR3: Self-recovery: host-based agents reconfigured; manual trigger for persistent errors.

4.2 Usability
NFR4: Security analysts shall examine and evaluate generated alerts with ≤ 3 interactions.
NFR5: Web application shall provide real-time visualizations with ≤ 2 second response time.
NFR6: Security analysts shall view monitored user list within one interaction.

4.3 Performance
NFR7: Incoming logs processed and analyzed in real-time with ≤ 5 second processing limit.
NFR8: Threat warnings displayed ≤ 2 seconds after detection.
NFR9: Smooth log consolidation of all host-based agents.

4.4 Security
NFR10: System shall resist unauthorized external access for ≥ 8 hours.
NFR11: Host-based agents shall have encryption-in-transit and at-rest for confidentiality and integrity.
NFR12: RBAC shall prohibit unauthorized viewing, modification, or deletion of sensitive log data.
NFR13: Unauthorized access/modification attempts shall raise alarm to security team within < 5 seconds.
