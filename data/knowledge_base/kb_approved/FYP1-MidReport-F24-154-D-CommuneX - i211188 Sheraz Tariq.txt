## 1. Introduction

1. Introduction
1.2 Scope
The Scope of Neighborhood and Community automation system involves the comprehensive
platform that integrate with various modules to improve the community operations. The func-
tionalities involve the user authentication, seamless communication platform, incident report-
ing and community voting system through block chain. Moreover, the property documents will
be handled and created using a blockchain. The blockchain system will provide the safe trans-
action system for the communities and the real time notification. Event management include
managing the events in the community as well as the resource would be shared in the commu-
nities to enhance the automated system. The community will do their payments and transaction
using blockchain. The project aims to enhance communication, efficiency and automate the
community on advance culture or in a global village.
1.3 Modules
1.3.1 Property buying and selling
Description: This module will simplify the processes and procedures of the properties whether
it is the buying or selling, money transactions or just viewing the prices of the plots and houses.
1. Feature 1: Buying and selling
2. Feature 2: Block chain based documentations
1.3.2 Event Management
Description: This module will help the community members to book any sort of events, places
for birthday parties or just a personalized event. Our model will also recommend the events to
the members.
1. Feature 1: Event Booking
2. Feature 2: Event recommendation.
1.3.3 Decentralized voting system
Our platform will have the functionality of a voting system which will let the users to record
their say in the important decisions of the community. The voting process will be smoother and
efficient and will be kept secure.

3.1 Architectural Design
The CommuneX system is designed to manage several aspects of Neighborhood and Commu-
nity automation which include the buying and selling of the properties, the decentralized voting
system for the community decisions, property documents stored on a block chain to maintain
the security and transparency and resource sharing among the community members. By de-
composing these modules into the system we will be providing the complete web application
for the community purpose. These modules integrates with each other maintain the desired
functionality.
3.1.1 System Architecture Design
The architecture diagram for CommuneX outlines a system that integrates various aspects of
community management. Users interact through a client interface, such as mobile or web
applications, which connect to a backed powered by Node.js. The server processes requests,
manages data flow, and interacts with both a blockchain for secure decentralized operations
(like voting or property transactions) and a database that stores community-related information.
The design ensures secure data handling and efficient management of community services like
communication, voting, and shared resources.

3.3 Data Design
The community automation system information domain is converted into the data structures
which help us to store the data processing and the entities like properties, voting creation, vot-
ing results, voting, resources, community engagement, private messaging and resource book-
ings. These entities store in non structural SQL like MongoDB while the property transaction,
property papers and the other content stored on block chain. By arranging the data in this way
our system ensure data transparency, security and information protection to improve the user
experience.
The database design for the community automation system will be:
3.3.1 User Management
It stores data of each users including userID, name, email, password, role and the contact
information. The userID assign to the user automatically by the mongoDB.
3.3.2 Property Management
The property management contains the list of the property for sale and the transaction details.
For the property listing the entities are ownerID, location, size, price details, photos, status for
availability.
The transaction details include propertyID, buyerID, sellerID, date of transaction, price and the
status.
3.3.3 Decentralized Voting system
The voting system contains two databases one for the voting creation and other is for casting a
vote.
The voting proposal creation includes votingID, title, description of voting, options for vote,
deadline and status of the voting.
The voting includes proposalID, voterID, voting option and the timestamp for the voting.
3.3.4 Resource Sharing
The resource sharing contains two databases. One is for resource listing and other is for re-
source booking.
The resource booking contains each resource ID, ownerID, type, description, availability, pho-
