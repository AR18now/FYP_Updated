1.3 Modules
1.3.1 Module 1 - Codebase Analysis
1. Project Structure Extraction: The tool will go through and analyse the currently
active project to extract the folder structure which will help in extracting informa-
tion. The system will identify all the relevant files (.py files) as well as extract and
structure the content within them.
2. Abstract Syntax Tree Generation: Using the previously structured code, the tool
will generate an Abstract Syntax Tree (AST) for each file. The ASTs will provide
a representation of the code, allowing us to perform a in-depth analysis of the code.
3. Code Dependency Graph Generation: The tool analyze the imports and create
a code dependency graph. This graph will show how different files are related
to each other, facilitating the analysis of code smells and visualizing the inter-file
dependencies that exist within the project.
1.3.2 Module 2 - Detection Module
1. Real Time Detection of Code Smells: The tool, upon opening a project, starts a
real-time detection process for code smells. As users make changes to the code and
it becomes compilable, AST analysis is performed on the modified file.
2. User Triggered Detection of Code Smells:In the tool, the detection of certain sub-
set of code smells will meed to be manually triggered by the user. By default, it will

only detect code smells present in the currently opened file. However, if needed,
the user can choose to scan the entire project to find these issues everywhere.
1.3.3 Module 3 - Refactoring Module
1. Refactoring Readiness Assessment of Identified Code Smells: The system will
evaluate whether identified code smells are suitable for automated refactoring by
large language models (LLMs). This feature will include the analysis of various
factors, including code complexity, and dependency relationships, to determine if
an LLM can effectively refactor the code.
2. Refactoring of Identified Code Smells: After detection of code smell, users can
then choose to either manually adjust the code or utilize the automatic refactoring
option to change their code depending on the results of the Refactoring Readiness
analysis.
3. Filter and Refactor specific code based on detected code smells: By filtering
code smells such as long functions, inconsistent naming, and god objects, users can
decide which issues they would like to address first and apply specific refactoring
techniques to improve code quality.
4. Cascading Code Refactoring: The refactoring process ensures that every im-
pacted file within the code base is consistently updated. Any modifications made to
one file will be reflected across the entire code base.
5. Isolated Compilation and Testing of Refactored Code base: Users can com-
pile and run a temporary version of the refactored code base before accepting the
changes, allowing them to validate and test the modifications without altering the
original code base.
6. Isolated Compilation and Testing of Refactored Code (Supplementary): Users
can compile and run a temporary version of the refactored code before accepting
the changes, allowing them to validate and test whether the modifications are se-
mantically same.
7. Personalized Refactoring Suggestions (Supplementary): Refactoring sugges-
tions will be tailored to the user’s coding style and patterns. When a user writes
code, the tool analyzes the user’s coding practices. It then generates refactoring
recommendations that align with these preferences. This personalized approach
ensures that the refactored code remains consistent with the user’s coding practices.

1.3 Modules
1.3.4 Module 4 - Change logs Module
1. Code smell detection and Refactoring tracker: This tracks the detection and
refactoring of code smells, while also keeping track the structure and dependencies
between files. It provides a detailed history of how code smells have been managed.
2. File Dependency Tracker: It also tracks the inter-dependencies within the project
that have evolved over time, offering developers a clear understanding of the project’s
progression and the impact of their changes.
3. Revert Readiness Assessment: This feature is to ensure that any refactoring that
the user wants to revert back to his original code is analyzed and determined to be
possible to be reverted without causing error in the code or inconsistency.
4. Code rollback: This feature allows users to rollback certain changes made through
automated refactoring given that the code snippet has passed the revert readiness
assessment.

1. VS Code Extension Rulesets:
(a) Rulesets for Detection: Users can specify certain rules that the tool will con-

consider during automated code refactoring. For example:
i. Include and Exclude Refactoring of Certain Code Smells for a Repository
ii. Include and Exclude Files for Refactoring of Code Smells
1.3.6 Module 6 - Website development Module
1. Shareable Logs Access: The user can add viewers to their project. These viewers
will be able to view and analyze the history and evolution of different code smells
over the life cycle of the project. This ensures that the viewers fully grasp the
evolution of code smells within a code base over time and how they were handled
or refactored.

3.1 Architectural Design
The architecture includes two client interfaces: a VS Code extension and a web applica-
tion, which interact with an API Gateway. The gateway is connected to a load balancer
provided by Amazon AWS, responsible for distributing incoming traffic efficiently. The
API Gateway routes requests to one of four microservices: Code Smell Detection, Refac-
toring, Refactoring Readiness, and Changelogs.
Among these microservices, the Changelogs service requires authentication for access,
while the Code Smell Detection, Refactoring, and Refactoring Readiness services do not
require authentication. However, the Detection and Refactoring microservices have the
capability to interact with the Changelogs microservice to store and retrieve data. For
these interactions, they must pass user authentication to gain access to changelog data.
The Changelogs microservice has access to a MongoDB database to store and manage
changelog information. It acts as the central hub for all data related tasks across the
system.
In addition to these microservices, there is an LLM (Large Language Model) microservice
that performs advanced processing tasks. This microservice uses Amazon SQS to manage
task queues, where Detection and Refactoring services send their tasks. Once a task is
processed by the LLM, the completed response is added to a message queue, which the
API Gateway accesses to send corresponding messages back to the appropriate client
sockets.
To aid its operations, the LLM microservice uses a VectorDB, enabling vector retrieval for
tasks requiring advanced data processing or contextual information. It also uses a cache
to store frequently accessed vectors, improving response times and reducing redundant
data fetching.

3.3 Data Design
The JSON schema defines the structure of how data is stored and managed within a VS
Code extension for Python projects. Each project has its workspace data, tracking folder
structure, file data, inter-file dependencies, and potential code smells. The following key
components are detailed in the schema:
• Project Structure:
– Each project workspace consists of three primary data objects:
* Folder Structure: This object reprsents the complete structure of the
project directory. The FolderStructure includes all files and subfold-
ers, tracking the entire hierarchy of the project’s file system.
* File Data: For each Python file (.py) in the project, information is main-
tained:
· Code and AST: The raw code of the file and its parsed Abstract
Syntax Tree (AST) are stored for analysis and code analysis.
· Functions: The list of functions defined within the file..
56
3.3 Data Design
· Classes: Details of any classes within the file, capturing variables,
methods, and calls.
· Global Variables: Information about all global variables declared in
the file.
· Imports: Data on all imports from other files, such as imported mod-
ules, classes, or functions, including their names and types. This
helps in understanding the dependencies.
* Dependency Graph: A separate object that represents the inter-file de-
pendencies across the project:
· This is a map of FileNodes, where eachFileNode contains the name
of a .py file.
· Each FileNode has a set of DependencyNodes that represent its re-
lationships with other files within the project.
· A DependencyNode keeps track of a list of UtilizedEntities,
which are details about functions, classes, or variables that are either:
· Exported to another file.
· Imported from another file.
· This structure allows tracking of how each file interacts with others
in terms of sharing and utilizing code elements.
• Code Smells:
– The schema includes an object dedicated to tracking and managing code
smells across the project. This data structure supports 10 different types of
code smells, each as an object that stores:
* Line Numbers and Description: The specific parts of the code where
the smells were identified, along with a description of the issue.
* Tracking and Analysis: For each code smell type, attributes are used
to track whether the analysis was successful, any errors that occurred,
and the specific data related to the smell.
– This data is imprtant for assessing code quality and identifying potential areas
of improvement.
• Attributes and Data Types:
– FileNode Object: A separate object that tracks each .py file’s key attributes,
including its name and its code content. It serves as a reference to single files
and helps manage their connections to other files in the project through the
dependency graph.

– DependencyNode: A sub-object that provides the details of a file’s depen-
dency, specifying the UtilizedEntities shared between files and whether

– CodeSmells Object: For each code smell detected, details such asline_numbers,
