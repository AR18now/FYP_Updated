The system will incorporate secure cloud storage, ensuring that both quizzes and grading

and arrays, and will focus on scanning handwritten codes, grading automation, plagia-

addresses the need for an efficient grading system for programming courses.

User class Description

grading process. The system will be used by them to make quizzes and track

Following are the requirements for module 1:
1. FR1: Teachers and teaching assistants (TAs) will be able to upload handwritten
quiz submissions in specified formats (JPEG and PNG) using the system.
2. FR2: The system shall automatically extract text from handwritten quiz submis-

3. FR3: To increase model accuracy during training, the system should enable inte-
gration with external datasets (IAM, MNIST, InfityCBD-2 etc.).
4. FR4: The system will keep track of every quiz submission in the database, associ-
ating it to the student’s individual ID and quiz number.
2.2.2 Module 2: Model Training
Following are the requirements for module 2:
1. FR1: The technology will enable the use of hybrid datasets, like EMNIST and
quizzes submitted by students, to train machine learning models, including neural
networks.
2. FR2: The system must enable saving of the trained model for further use and up-
grades.
3. FR3:To enhance performance, the system should allow users (teachers or system
administrators) to modify important model parameters including learning rate, batch
size, and epochs.
4. FR4:Following each model training session, the system is required to produce per-
formance metrics, such as accuracy, recall, and F1-score.
5. FR5:The system shall allow re-training of the model using new datasets or student
submissions to ensure the model remains accurate and up-to-date.
2.2.3 Module 3: Solution Generation
Following are the requirements for module 3:
1. FR1: Teachers can create custom grading guidelines and rubrics for particular ques-
tions in the system.
2. FR2: The system shall compare the student’s handwritten solution with the model-
generated solution using a text similarity algorithm.
3. FR3:The system shall generate alternate proper solutions if students have solved
the problem using different logic yet reached the correct answer.
4. FR4:Any differences between student and generated solutions should be recorded
by the system for later analysis.

2.3 Non-Functional Requirements
2.3.1 Reliability
1. REL-1: Under typical operating conditions, the GradeMate system must have a
mean time between failures (MTBF) of at least a thousand hours.
2. REL-2: With a high accuracy rate, the system will identify grading errors or mis-
matches in student submissions.
3. REL-3: To prevent data loss in the case of a system failure, the system must enable
teachers to access student information and grading history from the most recent
checkpoint.
4. REL-4: Automated grading of quizzes shall have a less failure rate. Error logs will
be maintained to record and notify any system failures
2.3.2 Usability
1. USE-1:The GradeMate system’s user interface (UI) shall be intuitive and easy to
use, for new users (teachers, TAs, and students).
2. USE-2: With just three clicks, the system should enable teachers to upload quizzes.
3. USE-3: Students will only need to click on their dashboard to view their graded
quizes.
4. USE-4: The system will offer error alerts and help for typical user mistakes, includ-
ing uploading the wrong file format for a quiz.
9
2. Project Requirements
2.3.3 Performance
1. PER-1: The system shall process the upload of a handwritten quiz in under 60
seconds with faster internet connection.
2. PER-2: For quizzes with 20 submissions or more, automatically graded results must
be accessible in 10 minutes.
3. PER-3: During peak hours, the system can accommodate up to 100 students and 50
teachers using it concurrently without experiencing any performance reduction.
4. PER-4: For the best possible user experience, the GradeMate system must load
each page or activity in less than two seconds.
2.3.4 Security
1. SEC-1: The system must guarantee that encryption standards are applied to all quiz
data, student submissions, and grade.
2. SEC-2: The users should add credentials to enforce user authentication for instruc-
tors and teaching assistants.
3. SEC-3: To guard against unauthorized access or modifications, the system must
record and keep track of every activity made by users (teachers, TAs, and students).
These logs must be kept for a period of 90 days.
4. SEC-4: Access control will be put into place to make sure that only individuals
with permission can read and edit tests, grades, or student information. After three
unsuccessful unauthorized access attempts, the user account will be locked and
alerts will be sent out.

3.3 Data Design
3.3.1 Transformation of Information Domain into Data Structures
Structured, semi-structured, and unstructured data (such as handwritten quizzes) are all
handled by the system. The process of converting information into data structures is as
follows:
1. Quizes: The primary data entity is quizzes, each of which has different attributes: .
It is linked to student submissions and grades.
22
3.3 Data Design
2. Users: Multiple user roles, such as students, teachers, and teaching assistants (TAs),
are supported by the system. A user’s name, email address, user type, credentials,
and user ID are among their attributes.
3. HandWritten Quiz Data:To extract text from student quiz images, OCR (Optical
Character Recognition) is used. These images are stored and linked to the corre-
sponding quizzes and students.
4. Grades: Each quiz attempt is graded, and the grades are stored in structured format
with attributes like marks, and quizNumber.
3.3.2 Storage of Major Data Entities
1. Relational Databases: Relational databases like MySQL are used to store structured
data, such as user information, quiz results, and grades. The following tables are
widely used:
• Users Table: Stores user data, differentiating between students, teachers, and
TAs.
• Quizzes Table: Holds details about every quiz.
• Grades Table: Using foreign keys to link students and quizzes, this table holds
the grades for each attempt at a quiz.
2. Handwritten Quiz Images: Handwritten quizzes are kept in cloud storage systems.
These files are linked to quizzes and students in the relational database.
3. Plagiarism Data: The system stores data from plagiarism detection processes. The
Plagiarism Results Table holds attributes such as plagiarismScore, and comparison-
Details
3.3.3 Processing of Major Data Entities
• OCR Processing: OCR tools are used to extract text, which is then compared against
a set of model solutions.
• Automated Grading: The system compares students’ responses to model solutions
using a combination of machine learning and pre-established grading rubrics. Ac-
curacy, completeness, and compliance to the question structure are taken into con-
sideration while awarding grades.
• Plagiarism Detection: To identify possible plagiarism, programs such as MOSS
will compare student submissions to both previous and current data.
