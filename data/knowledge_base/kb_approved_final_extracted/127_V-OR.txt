SOFTWARE REQUIREMENTS SPECIFICATION (IEEE 830)
Project: V-OR - Virtual Operating Room for Surgical Training
================================================================================

1. INTRODUCTION
1.1 Purpose
This document outlines the requirements for the Virtual Operating Room (V-OR), an advanced, immersive VR platform for medical students and professionals to practice complex surgeries (ASD closure, Pacemaker Implantation, CABG) in a realistic, risk-free environment.

1.2 Scope
The platform focuses on three cardiac surgeries. Features include a surgical simulation engine (interactive 3D models, step-by-step guidance, dynamic scenarios), virtual training mode (guided/independent practice), real-time feedback and performance evaluation, and a user management system (profile, progress tracking, instructor access).

1.3 Definitions, Acronyms, Abbreviations
- VR: Virtual Reality
- ASD: Atrial Septal Defect
- CABG: Coronary Artery Bypass Grafting
- 3D: Three-Dimensional

2. OVERALL DESCRIPTION
2.1 Product Perspective
V-OR provides an affordable, ethical, and repeatable alternative to traditional surgical training methods, overcoming limitations of physical simulators and cadaver availability.

2.2 Product Functions (Summary)
- Surgical Simulation Engine: Interactive 3D models, step-by-step guidance, dynamic scenarios.
- Virtual Training Mode: Practice with or without guidance, independent practice mode with evaluation.
- Real-time Feedback and Performance Evaluation: Real-time error detection, detailed performance reports.
- User Management System: Authentication, profile management, progress tracking, instructor monitoring.

2.3 User Classes and Characteristics
- Trainee: Medical students/first-year surgeons performing simulated cardiac surgeries (≈200-500 users).
- Doctor: Supervisor/instructor assessing trainees, adjusting scenarios (≈50-100 users).
- Simulation Admin: Technical staff managing the platform, setting up environments.
- 3D Model Designer: Develops and maintains high-fidelity 3D anatomical models.
- System Manager: Manages user verification, data organization, and access control.

3. FUNCTIONAL REQUIREMENTS
3.1 USER MANAGEMENT SYSTEM
FR1: The system shall allow users to login and manage their accounts.
FR2: User roles shall include students, surgeons, and administrators.
FR3: Users shall be able to update their profiles, view training progress, and feedback reports.
FR4: The system shall track and securely save the user's progress in different training modules.

3.2 SURGICAL SIMULATION ENGINE
FR1: The platform shall include interactive 3D models of organs, tissues, and surgical tools.
FR2: The system shall provide step-by-step guidance during training surgeries.
FR3: The dynamic scenarios generated by the system should mimic real-world complexities.

3.3 VIRTUAL TRAINING MODE
FR1: Users shall be able to take an assessment surgery with no help, receiving a performance evaluation.
FR2: Users shall be able to choose the surgery they want to perform from the 3 options (ASD, pacemaker, CABG).

3.4 REAL-TIME FEEDBACK & PERFORMANCE EVALUATION
FR1: The system shall be able to detect real-time errors during surgery and notify the user.
FR2: The system shall be able to provide detailed feedback and evaluate user performance using a predefined criterion.

3.5 USER INTERFACE
FR1: The system shall provide an intuitive dashboard for users to navigate to the selection of surgeries and track progress.

3.6 REPORTING AND ANALYTICS
FR1: The instructors shall be able to review the feedback and assessment reports of the students.
FR2: Instructors shall be able to provide feedback to those students' reports.

3.7 COMPATIBILITY
FR1: The system shall be compatible with Oculus VR headset and hand sensors.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Scalability
NFR1: The system will be scalable to accommodate additional complex surgeries in the future.

4.2 Compliance
NFR2: The system will comply with medical training standards.
NFR3: The system should comply with data protection regulations such as GDPR.

4.3 Data Integrity
NFR4: The system should ensure that all performance data is captured and stored without loss.

4.4 Maintainability
NFR5: The code should be modular and well documented.
NFR6: The system should include detailed documentation covering all major components.

4.5 Compatibility
NFR7: The VR environment should be compatible with multiple VR headsets, including Oculus Quest 2.
NFR8: The system should be compatible with major operating systems like Windows and Linux.

4.6 Security
NFR9: User authentication will be required for all access, with role-based access control.
NFR10: Sensitive data will only be accessible to authorised users.

4.7 Reliability
NFR11: The system will be highly reliable, with minimal downtime.

4.8 Availability
NFR12: It will be available 24/7.

4.9 Usability
NFR13: The interface should be user-friendly and intuitive for medical students and professionals.
NFR14: The VR experience should be immersive and realistic.

4.10 Performance
NFR15: The system should provide real-time feedback during surgeries without any noticeable delay.
