## 1. Introduction

1. Introduction
1.2 Scope
The automation of UML diagram creation from natural language software requirements
is the main goal of this project. It is pertinent to academic learning, software develop-
ment, software engineering automation, and system design. It will cover the generation
of PlantUML diagrams by utilizing natural language processing and program synthesis
techniques. The system is intended for use by software engineers, system architects,
project managers, and the academic community, and will support any input software re-
quirements and high-level specification without modification. The scope includes the
development, testing, and deployment of the AiUML system, to integrate it into existing
software development workflows and educational environments.

1.4 Modules
AiUML is composed of several key modules, each responsible for different aspects of the
UML diagram generation process. The approach being followed by our system is based
on Generative AI. The modules are as follows:
1.4.1 Module I: Dataset Collection and Preparation
This module is responsible for gathering and preparing datasets necessary for training our
system. It involves collecting different software requirements text corpora, annotating and
labelling it to ensure that the data is formatted for large language models, to make sure
that it covers a broad spectrum of software requirements.
1.4.2 Module II: Fine-Tuning LLM
This module consists of fine-tuning a text-to-code Large Language Model called Code
T5. The generative model is trained on an extensive dataset and the weights are adjusted,
such that it correctly generates code for a diagram when given natural language text as
input.
1.4.3 Module III: Requirements Pre-processing
The Requirements Pre-processing Module is the first step, responsible for processing raw
textual software requirements. It uses models to prepare the text for deeper analysis by
the subsequent layers. Key functions include:
• Text Normalization: The module makes sure the text is standardized and clean
enough to be processed. This involves doing things like resolving frequent spelling
variants, eliminating extraneous punctuation, and changing text to lowercase.
• Tokenization: Tokens, sometimes referred to as words or subwords, are the tiny
pieces of the text that the language models will process. This phase is essential for
the subsequent steps that establish each token’s function in the UML diagram.
This preparation allows the system to handle natural language inputs without requiring
any manual adjustments.
4
1.4 Modules
1.4.4 Module IV: Requirements Disambiguation (with Pre-trained
Models)
In this module, the processed text is passed through a hybrid ML and spanBERT solution.
The ML model detects the ambiguity in requirements, specifically anaphoric ambiguity.
The spanBERT model resolves the detected ambiguity. This step is crucial for subsequent
processing.
1.4.5 Module VI: DSL Code Generation and Diagram Rendering
This module is responsible for generating DSL code in PlantUML or Mermaid.js from
the Code T5 LLM. It ensures that the UML elements and their relationships are correctly
represented in code form. After generating the code, this module uses PlantUML library
or Mermaid.js library to interpret and render the final UML diagrams. By the end of this
module, there will be a visual representation of the system’s architecture, which users can
customize or export in SVG, PNG, and text formats.
1.4.6 Module VII: Diagram Optimization with AI
This module is responsible for optimizing the diagram using a conversational LLM and
CodeT5 in conjunction. After initial diagram generation, this module allows user to input
feedback on the generated diagram and helps optimize diagram through prompts, like
adding/removing elements, optimizing relationships according to standards or changing
the layout of diagram.
1.4.7 Module VIII: Diagram Customizer and Editor
This module provides users with the ability to edit diagram codes directly within a DSL
editor. It also allows for the customization of diagrams by applying different colors and
themes, giving users greater control over the visual appearance and style of their UML
diagrams. This functionality enhances the user experience by offering flexibility in design
and presentation.
1.4.8 Module IX: Session and Diagram Code Storage
This module is responsible for storing the previous sessions of the user, ensuring that
workspaces are maintained effectively. It saves previous diagrams and stores their corre-
sponding codes in formats such as PlantUML or Mermaid.js. This functionality allows

1.4.9 Module X: User Interface
This module involves the development of a web application that provides an intuitive
interface for users. It allows users to input textual software requirements and interact
with the generated UML diagrams in real-time. The interface supports features such as
exporting diagrams, editing them, and customizing their appearance. Additionally, this
module integrates with the backend services to store user sessions, retrieve previously
saved diagrams, and manage diagram code storage in formats like PlantUML.

System Overview
AiUML is a system developed to automatically create UML diagrams from natural lan-
guage requirements. Its primary function is to receive user input in the form of text, find
any ambiguities in the requirements, and then produce precise and understandable UML
diagrams, such as class, use case, sequence, and entity-relationship diagrams. AiUML
helps simplify the process of designing object-oriented software by converting complex
user requirements into easy-to-understand visual diagrams.
The design methodology for this system is Procedural. The related design artifacts are
given below.
3.1 Architectural Design
AiUML follows Client-Server Architectureconsisting of a client-side web interface and
a server on the back-end which is hosting multiple services.
Client Side:
• On the client side, users interact with the system via a simple web interface where
they input textual requirements, customize diagrams, and view the generated out-
puts.
Server Side:
• The back-end Server handles text processing, ambiguity detection, and diagram
generation using large language models.
• The server also has services for diagram optimization using pre-trained machine
learning models, customization, editing and exporting diagrams.

3.7 Data Design
AiUML system transforms its information domain into structured data entities, which
are stored and managed in a Database. This database consists of key entities such as
User, Session, Requirements, and Diagram, each corresponding to a table that supports
the system’s functionality.
3.7.1 Key Entities:
All these entities are interconnected and processed by the back-end services. The database
items include:
• User Table
• Session Table
• Requirements Table & Optimization Requests Table
• Diagrams Table & Diagram Code Table
• Ambiguities Table
• Themes Table (pre-defined)
For example, the User table stores user information likeUserID, Username, and Password,
while the Requirements table holds the natural language text that users input for diagram
generation. The Session table tracks active user sessions and allows the system to retrieve
and manage ongoing activities. The Diagram table stores generated UML diagrams, their
metadata, and code, which can later be accessed for editing or export.
3.7.2 Storage and Processing:
The Database is the central storage for AiUML, with specific entities (tables) that store
data related to users, sessions, requirements, diagrams, and ambiguities. Each entity in the
ERD corresponds to a table in the database, and the relationships between these entities
are reflected in how the data is processed:
• Requirements are processed by the Text Processing Service and Ambiguity Detec-
tion Service to generate accurate UML diagrams.
• Sessions are maintained to track user activities and allow for session-based diagram
retrieval and editing.
54
3.8 Entity Relationship Diagram
• Diagrams are stored and managed through the Diagram Generation Service, which
also integrates with the AI-based Diagram Optimizer for refinement.
This structure facilitates smooth data processing, retrieval, and storage, and thus supports
the overall functionality of AiUML system.
