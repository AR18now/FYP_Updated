SOFTWARE REQUIREMENTS SPECIFICATION (IEEE 830)
Project: CrimeSight - Real-Time Crime Detection and Reporting System
================================================================================

1. INTRODUCTION
1.1 Purpose
This document outlines the requirements for CrimeSight, a system that aims to revolutionize traditional security by detecting and reporting crimes live, allowing for timely action by authorities.

1.2 Scope
The system detects and classifies crimes (fire, fight, weapon, destruction of property) within indoor environments. It alerts users and authorities via WhatsApp and mobile app, provides person tracking, remote feed viewing, clip sharing, and analytics.

1.3 Definitions, Acronyms, Abbreviations
- AI: Artificial Intelligence
- CCTV: Closed-Circuit Television
- MTBF: Mean Time Between Failures

2. OVERALL DESCRIPTION
2.1 Product Perspective
CrimeSight shifts from a reactive crime reporting system to a proactive one by using machine learning to continuously monitor and instantly report suspicious activities.

2.2 Product Functions (Summary)
- Multi-Crime Classification and Detection (Fire, Fight, Weapon, Destruction).
- Crime Analytics and Statistics.
- Crime Reporting via WhatsApp and app.
- Suspicious Activity Tracking.
- Clip Sharing.
- Remote Feed Viewing.
- Emergency Contacts.

2.3 User Classes and Characteristics
- Property Owner: Primary user who owns the property and uses the system to watch activities and receive alerts.
- Law Enforcement Users: Authorized users from agencies who are alerted and receive video evidence.

2.4 Operating Environment
Desktop and mobile applications.

3. FUNCTIONAL REQUIREMENTS
3.1 Multi Crime Classification And Detection
FR1.1: The system shall detect and classify crimes into four categories: Fire Detection, Fight Detection, Weapon Detection, and Destruction of Property.
FR1.2: The system shall distinguish between normal and crime scenarios.
FR1.3: The system shall process video feeds from multiple cameras simultaneously.

3.2 Crime Analytics
FR2.1: The system shall display statistics of crime occurrences on a day-to-day, month-to-month, and year-to-year basis.
FR2.2: The system shall generate patterns in crime timings, types, and frequency.
FR2.3: The system shall provide safety advice based on the top crimes being committed.

3.3 Crime Reporting
FR3.1: The system shall automatically report to relevant authorities using WhatsApp.
FR3.2: The system shall automatically report to the Camera Setup owner using the Mobile Application.
FR3.3: The system shall have options to choose alerts to authorities based on the type of crime.
FR3.4: All crime reports shall include video evidence.

3.4 Suspicious Activity
FR4.1: The system shall allow the user to select an individual through the Desktop Application and track them through the multiple camera setup.
FR4.2: The system shall allow the user to set the time limit to track based on past available camera feeds.
FR4.3: The system shall provide the user with the combined clips from multiple cameras of the steps taken by the individual.

3.5 Tracking Analysis
FR5.1: The system shall provide the user with a relevant summary of what the tracked individual did.
FR5.2: The system shall provide analytics relevant to the individual tracked on both Desktop and Mobile Applications.

3.6 Clip Sharing
FR6.1: The system shall allow the user to combine multiple clips from a single feed.
FR6.2: The system shall allow users to share camera feeds with other users.
FR6.3: The system shall allow the user to specify the number of times the shared clips can be downloaded.

3.7 Remote Feed
FR7.1: The system shall allow users real-time access to camera feeds remotely on both Mobile and Desktop Applications.
FR7.2: The system shall allow the user to simultaneously view feeds from multiple cameras.

3.8 Emergency Contacts
FR8.1: The system shall provide the user with relevant emergency contacts for relevant authorities.
FR8.2: The system shall allow the user to customize emergency contact lists.
FR8.3: The system shall provide the user with immediate and easy access to contact emergency services.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Reliability
NFR1: The system would be highly reliable, covering discrepancies in the model.
NFR2: The system would achieve an average MTBF of 600 hours.
NFR3: The system would manage crashes by sending proper logs and trying to recover within 5 minutes.

4.2 Usability
NFR4: The camera feeds will be available on the home page requiring no extra button clicks.
NFR5: Pressing on any Emergency Contact will result in a call being made directly.
NFR6: The system will be user-friendly for new users.

4.3 Performance
NFR7: The system shall classify activity type for 2 cameras simultaneously within a second.
NFR8: Alerts will be notified to relevant authorities within 5 seconds of finalizing crime detection.

4.4 Security
NFR9: The application will use industry standard encryption algorithms for all personal user data and security feeds transmission.
NFR10: The system will secure users using two-factor authentication.
NFR11: The system will make sure the shared clips are only available for a specific period of time.
NFR12: The system will manage a log of all audits.
