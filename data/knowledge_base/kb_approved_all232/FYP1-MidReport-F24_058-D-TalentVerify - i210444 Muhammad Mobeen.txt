1.2 Scope
TalentVerify is a web based application for employers to streamline the interview process.
The scope of this iteration focuses on the creating and usage of deep learning models that
can accurately identify audio and video emotions which will later be used for evaluating
the soft skills.
An interface for both employer and regular user allows them to interact with the system.
The employer sets the preferences to create an instance of the interview to begin the au-
tomate hiring process. Users will be authenticated and then allowed to partake in the
interview.
During the interview process key interviewee features are extracted and evaluated us-
ing distinct models for comprehensive analysis of the candidates emotions. Moreover, a
multiple entity detection system is also employed to find if the candidate is getting any
external assistance.

Chapter 3
System Overview
Our system allows employers to conduct easy automated interviews. The employer can
choose how many candidates to pass on, as well as the soft and technical skill score
thresholds. The employer may invite candidates to participate in these interviews by link-
ing them on the job posting. Our web app will then setup and conduct automated inter-
views of all the candidates and screen out the number specified by the employer based on
how well they did on both their technical skills and their soft skills(confidence, nervous-
ness, clarity etc). In the course of holding each interview, our automated agent will parse
through each candidates resume, and generate personalized questions to assess each can-
didate as well as questions based on the job description provided by the employer. These
candidates are then provided to the employer alongside data on how the interviewees in
general performed on both their soft and technical skills.
3.1 Architectural Design
We chose a micro-service architecture for our application, with 3 core services, Job Post-
ing, User, and Interview Conduction. Interview conduction also contains some critical
sub-services that all need to work for it to function.We chose this architecture because it
is important in our case to ensure that independent tasks can be completed even if some
functionality of the system is unavailable. For example, an employer should still be able
to view their dashboard and candidate shortlists, even if the job posting service is unavail-
able. Micro-service architectures also have the added benefit of scaling well, and dealing
with asymmetrical load on different parts of the application.

3.3 Data Design
Our database management system of choice is MongoDB. We chose MongoDB as it is a
great free cloud based service with great scaling support. It is also NoSQL which allows
us to store things like avatar parameters and weights easily. All of our user data is stored
on a central database. Our database contains all user information, past and present scores,
past and present interviews, statistics on candidate performance. All weights and neces-
sary parameters for interviewer avatars are also stored inside of this database. Whenever
an employer sets up an interview, we store this request along with a processed version
of the job description. This processed version is generated by passing the description
through an LLM to extract only necessary information. The same processing is applied
to uploaded resumes, which are stored alongside their generated scores and any remarks
generated by our scoring system.
All of our neural networks are stored on servers in the form of weight files, with backup
copies on offline systems. All temporary data is stored server-side whenever necessary.
