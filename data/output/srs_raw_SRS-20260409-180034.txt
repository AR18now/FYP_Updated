INTRODUCTION
Purpose: To provide a structured set of requirements for the development of a bank application that facilitates new customer onboarding through document submission, liveness checks, electronic signing, compliance reviews, integration with credit bureaus, logging decisions, status updates, secure messaging, and handling upload retries.
Scope: This SRS covers the core functionalities required for new customer onboarding including document submission, liveness check, electronic signature, compliance review, credit bureau integration, logging, status updates, secure messaging, and upload retry mechanisms. It does not cover security, scalability, maintainability, or availability aspects.
Definitions/Acronyms:
- KYC: Know Your Customer
- Liveness Check: Verification process to ensure the person submitting documents is alive and present during the submission
- E-Signature: Electronic form of signing legal documents
- Credit Bureau: Organization providing information about individuals' financial history
References: IEEE Std 830-1998, ISO/IEC 25010:2011
Overview: The bank app will streamline the onboarding process by automating key steps while ensuring regulatory compliance and maintaining high standards of security and privacy.

OVERALL DESCRIPTION
Product Perspective: The product perspective focuses on creating a seamless experience for both new customers and compliance officers, integrating various backend systems and services.
Product Functions: The primary functions include document submission, liveness verification, electronic signing, compliance review, credit bureau integration, logging, status updates, secure messaging, and upload retry mechanisms.
User Characteristics: Target users are new customers who wish to onboard themselves and compliance officers responsible for reviewing and approving applications.
Constraints: The system must comply with local regulations regarding KYC processes, handle large volumes of data securely, and ensure real-time processing capabilities.
Assumptions/Dependencies: The system assumes access to necessary APIs from credit bureaus and other third-party services. Dependencies include network stability and timely responses from external systems.

SPECIFIC REQUIREMENTS
External Interface Requirements
- User Interface: The application provides interfaces for new customers to submit KYC documents, perform liveness checks, and e-sign agreements; for compliance officers to review flagged applications, request clarifications, and approve/reject them.
- Hardware Interface: No specific hardware interface is required beyond standard mobile devices capable of running the application.
- Software Interface: Integration with credit bureaus via API calls for retrieving credit reports.
- Communication Interface: Secure in-app messaging between new customers and compliance officers.

Functional Requirements
FR-01: Submit KYC Documents
- Input: New customer uploads KYC documents.
- Processing: The system verifies file types and sizes before storing them temporarily.
- Output: Confirmation message displayed to the user indicating successful upload.
- Priority: High
FR-02: Perform Liveness Checks
- Input: User captures video or photo proof of identity.
- Processing: The system uses facial recognition algorithms to verify the user's identity.
- Output: Success or failure notification based on the outcome of the liveness check.
- Priority: High
FR-03: E-Sign Agreements
- Input: User selects agreements to sign electronically.
- Processing: The system generates digital signatures and stores them securely.
- Output: Confirmation message and signed agreement document available for download.
- Priority: Medium
FR-04: Review Flagged Applications
- Input: Compliance officer accesses flagged applications.
- Processing: The system highlights critical issues requiring attention.
- Output: List of flagged applications along with details and requested actions.
- Priority: High
FR-05: Request Clarification
- Input: Compliance officer requests additional information from the applicant.
- Processing: The system sends a clarification request to the applicant via in-app messaging.
- Output: Notification sent to the applicant and logged internally.
- Priority: Medium
FR-06: Approve or Reject Application
- Input: Compliance officer makes final decision after review.
- Processing: The system records the decision and logs it for audit purposes.
- Output: Decision communicated back to the applicant and stored in the database.
- Priority: High

Non-functional Requirements
Usability: The application shall be intuitive and easy to navigate, especially for first-time users.
Reliability: The system must function consistently without errors or crashes, even under heavy load conditions.
Performance: The application shall respond quickly to user inputs within 2 seconds, with minimal latency.
Portability: The application can run on Android and iOS platforms without significant modifications.

SYSTEM FEATURES
Feature 1: Document Submission
- Allows new customers to upload KYC documents.
- Verifies file formats and sizes.
- Stores documents temporarily until further action.

Feature 2: Liveness Verification
- Captures video/photo proof of identity.
- Uses facial recognition technology.
- Provides pass/fail result.

Feature 3: Electronic Signing
- Enables users to select agreements.
- Generates digital signatures.
- Stores signed documents securely.

Feature 4: Compliance Review
- Highlights flagged applications.
- Displays detailed issues.
- Logs all actions taken by compliance officers.

Feature 5: In-App Messaging
- Facilitates communication between new customers and compliance officers.
- Sends notifications upon receipt of messages.

Feature 6: Upload Retry Mechanism
- Supports automatic retries for failed uploads.
- Allows users to resume incomplete tasks.
