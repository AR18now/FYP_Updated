1. INTRODUCTION
1.1 Purpose
This document outlines the requirements for Wi-Fi Vision (Wi-Vi), a smart AI-based surveillance system that uses WiFi signal strength (Channel State Information) and deep learning to detect and classify human presence and movements, offering a cost-effective alternative to traditional CCTV.

1.2 Scope
The system uses ESP32 NodeMCU SoCs to record CSI values, which are transmitted to a cloud-deployed LSTM model via Flask API for real-time prediction. A responsive web app (ReactJS) and an interactive mobile app (Flutter) allow users to monitor their space, view real-time data/heatmaps, receive alerts, and integrate with emergency services.

1.3 Definitions, Acronyms, Abbreviations
- CSI: Channel State Information
- IoT: Internet of Things
- LSTM: Long Short-Term Memory
- API: Application Programming Interface
- JWT: JSON Web Token

2. OVERALL DESCRIPTION
2.1 Product Perspective
Wi-Vi provides an innovative, cost-effective, and energy-efficient surveillance solution by leveraging existing WiFi infrastructure and deep learning, eliminating the need for expensive cameras and complex installations.

2.2 Product Functions (Summary)
- CSI IoT Recorder: Record and display Wi-Fi CSI values using ESP32 and OLED screen.
- LSTM with Flask API: Deploy trained LSTM model on cloud, exposed via API for real-time prediction.
- Web Application (ReactJS): User authentication, real-time monitoring, dashboard, system configuration, historical data review, user management, emergency services integration.
- Mobile Application (Flutter): Real-time monitoring, push notifications, system configuration, event history, offline access, secure access.

2.3 User Classes and Characteristics
- Homeowners: Primary users monitoring their spaces (approx. 80% mobile use).
- Organizations: Monitor business premises (approx. 150 adopting organizations, primarily web use).
- Emergency/Law Enforcement Agencies: Interact during security incidents via integration.

3. FUNCTIONAL REQUIREMENTS
3.1 Channel State Information IoT Recorder
FR1: The system shall record Wi-Fi Channel State Information (CSI) values after a specific time interval.
FR2: The system shall transmit Channel State Information (CSI) data to Flask API endpoint after a specific time interval.

3.2 Long Short-Term Memory with Flask API
FR1: The model shall communicate with ESP32 through Flask API.
FR2: The model shall process Channel State Information (CSI) data in real-time.
FR3: The model shall detect variations in Channel State Information (CSI) data.
FR4: The model shall classify the presence of human activity or objects based on variations in CSI data.
FR5: The model shall predict for each event with a confidence score.
FR6: The model shall trigger an event in case of detected anomalies.

3.3 Web Application in ReactJS
FR1: The system shall have an interactive and robust user interface.
FR2: The system shall allow users to securely login to access their security system.
FR3: The system shall allow users to upload the architectural plan of their premises.
FR4: The system shall display real-time human monitoring and object detection data.
FR5: The system shall provide heatmaps for visualization.
FR6: The system shall provide a dashboard to display system status and analytics.
FR7: The system shall allow users to configure their system settings based on their preferences.
FR8: The system shall allow users to access surveillance data for the previous two days period.
FR9: The system shall provide a role-based access control to users for permissions.
FR10: The system shall provide an option for integration with emergency services.

3.4 Mobile Application in Flutter
FR1: The system shall allow users to securely login to access their security system.
FR2: The system shall send push notifications in case of security incident.
FR3: The system shall display real-time human monitoring and object detection data.
FR4: The system shall provide heat maps for visualization.
FR5: The system shall provide a dashboard to display system status and analytics.
FR6: The system shall allow users to configure their system settings based on their preferences.
FR7: The system shall provide users with cached surveillance data to access it even when offline.
FR8: The system shall provide a role-based access control to users for permissions.
FR9: The system shall provide an option for integration with emergency services.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Reliability
NFR1: The system shall maintain a reliable architecture with a 99.9% uptime.

4.2 Scalability
NFR2: The system shall be scalable to monitor larger spaces up to the transmission range of WiFi signals.

4.3 Usability
NFR3: The system shall provide usable and interactive web and mobile applications.
NFR4: The mobile app shall have support for offline usage.

4.4 Performance
NFR5: The system shall meet real-time requirements for monitoring, recording CSI values at regular intervals.
NFR6: The model shall have a simple input to process data and predict the pattern.

4.5 Maintainability
NFR7: The system shall be developed using industry line development techniques to provide a modular architecture.
NFR8: The system shall be documented properly.
NFR9: The applications shall be containerized.

4.6 Availability
NFR10: The system shall be available all the time with maximum uptime.

4.7 Security
NFR11: The system shall employ industry line security to provide secure and easy to use.
NFR12: User authentication shall utilize JsonWebTokens (JWT).
NFR13: The system shall leverage Role-Based Access Control.

4.8 Interoperability
NFR14: The system shall support integration with external systems like emergency services for alerts during security incidents.
