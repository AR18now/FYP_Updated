2.2 Functional Requirements
2.2 Functional Requirements
2.2.1 Data Acquisition & Preparation (Module 1)
This module focuses on gathering, processing, and evaluating data that concerns sketches
and sprites and also includes putting the data in different stages.
1. The system shall allow users to upload datasets of sketches and sprites.
2.2.2 Fine-Tuning & Data Optimization (Module 2)
This module is devoted to the process of fine-tuning the model of transforming sketches
into sprites as well as working with the output data for its further processing.
1. The system shall create a sprite out of an uploaded or fine-tuned sketch using image-
to-image model.
2.2.3 Model Training & Image Optimization (Module 3)
In this module, advanced image processing techniques are adopted to cut out the im-
perfections of sprites, and finally sprite sheets are composed based on the single sprites
generated in previous steps.
1. The system shall allow image processing functions such as resolution scaling, back-
ground removal, and noise reduction to enhance sprite quality.
2. The system shall automatically generate sprite sheets from individual sprites, orga-
nizing them in a format suitable for game development or animations.
2.2.4 Post-Processing & Validation Phase (Module 4)
The emphasis of this module is on post-processing the sprites and sprite sheets that have
just been generated, followed by detailed testing and validation to confirm that every
aspect of the system’s output is redeployed.
1. The system shall ensure that the product is simply operational by providing error
recovery features, error logs, and error resolution details in case there are any mal-
functions.

2.3 Non-Functional Requirements
This section contains the non-functional requirements concerning the A2S system. These
quality requirements have been made such a way that they are specific, quantifiable, and
verifiable, and therefore the system has given performance, levels and cuts in reliability
as regards various aspects.
2.3.1 Reliability
The A2S system must be able to have a high reliability metric and hence have very few
failures when in operation. The reliability will be assessed in terms of Mean Time Be-
tween Failures (MTBF) and any occurrence of a failure has to be reported and documented
as such.
RLB-2: It shall be possible for the system, in the event of an external system failure, to
restart the application and have recovered and functioning within 30 seconds of the restart
this will help in avoiding workflow interruptions for the user.
RLB-3: The system will incorporate strategies for detecting faults such that failures that
will occur will be detected and recorded with a 99% performance efficiency of the rate of
detection.
2.3.2 Usability
A2S will put emphasis on an interface that is friendly to the users, so that the users can
easily learn and efficiently use the interaction with A2S. It will also ensure that the design
will avoid most of the errors and in case there are any, proper error recovery will be in
place.
USE-1: It will be possible for the users of A2S to upload a hand drawn sketch, wait for
a sprite to be produced, and only perform three action steps in retrieving the generated
sprite.
USE-2: Users will be able to acquire sufficient knowledge on how to navigate and use the
A2S system in less than 15 minutes using interactive onboarding tutorials and training.
2.3.3 Performance
The A2S application must work optimally to give quick and active results to the users.
Performance is to be evaluated through checking how long certain operations would take.

Since Art2Sprite is a document-oriented model, we constructed a JSON schema for the
data model instead of the traditional Entity Relation Diagram. Since there is only a storage
for images, a traditional relational database is an excess. The JSON schema is more
suitable for the structuring the data depending on the forms input and outputs from the
models. [11]
Art2Sprite depends on three major communication steps, each with their own JSON
schema. These are the "UserInputs", "SpriteGeneration", and "SpritesheetGeneration".
Each of them is shown together above. All of the JSON elements have their specific de-
scriptions, and we will explain the "UserInput" schema. Art2Sprite requires user input
from a Web Form that receives a sketch image from the user as well as a more descriptive
prompt, and the framerate, sprite resolution and animation they require. For now, we have
decided this information is enough as part of dropdown options, represented by key values
of each option. Each of these attributes have their own datatypes such as URI for image,
string for text prompt, etcetera. All of this information is required for the AI models to
work properly and produce an output which the user desires.
Creating a JSON schema is valuable for the team in the future, to ensure consistency while
building APIs and validating the structure of information being shared. By enforcing these
requirements, errors will be prevented during the early development process. Since using
a JSON schema will standardize the data structure, all team members will understand the
same schema without ambiguity or confusion and all tools and systems will also have no
compatibility problems following the expected format.
