1. INTRODUCTION
1.1 Problem Statement
Centralized systems for storing sensitive student data are vulnerable to cyber attacks, illegal alteration of data, and system failures. Educational institutions grapple with log obfuscation, slow response time to harmful actions, and leaking of crucial information.

1.2 Scope
OneBlock is a blockchain-based student portal providing tamper-proof log system with immutable records stored on blockchain. Features include real-time log collection from student activities, administration decisions, system changes; smart contracts for handling log changes; decentralized distribution for enhanced availability; data visualization (graphs, charts, tables); report generation; faculty announcements; and admin user management.

1.3 Definitions, Acronyms, Abbreviations
- LMS: Learning Management System
- RBAC: Role-Based Access Control
- MTBF: Mean Time Between Failures

2. OVERALL DESCRIPTION
2.1 Product Perspective
OneBlock ensures tamper-proof, decentralized data management enhancing trust and accountability through blockchain technology.

2.2 Product Functions (Summary)
- Secure login and authentication for faculty
- Dashboard for real-time monitoring and alerts
- Tamper-proof log viewing with filtering options
- Announcement feature for faculty updates
- Real-time log collection and storage on blockchain nodes
- Smart contracts for log changes
- Immutable data storage
- Decentralized distribution of logs
- Graphs and charts for visualizing logs and trends
- Detailed reports on alerts and system changes
- User management (roles, permissions)
- Node and network management

2.3 User Classes and Characteristics
- Student: View academic logs (submissions, attendance, grades), receive announcements
- Faculty: Input and manage student data, post announcements, generate reports, monitor logs
- Administrators: Manage user accounts, monitor blockchain nodes, ensure system security
- Blockchain Network Participants (Validators): Verify and validate transactions
- Testers: Evaluate functionality, security, performance

2.4 Operating Environment
Web application, Ethereum blockchain, smart contracts, Firebase.

3. FUNCTIONAL REQUIREMENTS
3.1 Client Web Application
FR1.1: The system shall provide secure login and authentication for faculty members.
FR1.2: Faculty shall be able to post announcements through web application.
FR1.3: The system shall allow faculty to generate detailed reports on student activities, logs, anomalies.
FR1.4: Admin shall have filtering options for logs based on hashes, instructor name, or browser used.

3.2 Blockchain Log Collector
FR2.1: The system shall collect and store logs from student activities, faculty actions, administrative decisions in real-time using blockchain.
FR2.2: The system shall ensure all stored logs immutable (cannot be altered or deleted once recorded).
FR2.3: The system shall distribute logs across multiple blockchain nodes for data availability.
FR2.4: The system shall validate logs using smart contracts to detect and flag suspicious changes automatically.
FR2.5: The system shall provide decentralized access to logs.

3.3 Data Visualization and Reporting
FR3.1: The system shall provide graphical representations of log data (charts, graphs, tables).
FR3.2: The system shall enable real-time updates of visual data.
FR3.3: The system shall allow exporting reports in PDF, CSV, Excel formats.
FR3.4: The system shall provide detailed breakdown of anomalies or suspicious activities in visual format.

3.4 Admin Web Application
FR4.1: The system shall provide administrators with secure login and role-based access control.
FR4.2: Admins shall have access to audit trail detailing system changes, user permissions, administrative actions.
FR4.3: The system shall log all notifications and alerts for review.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Reliability
NFR1: MTBF ≥ 500 hours.
NFR2: Recovery from failure within 15 minutes.
NFR3: Automated error detection and correction mechanisms.
NFR4: Failover strategies for critical system components.

4.2 Usability
NFR5: Key tasks (log viewing, report generation) with ≤ 3 interactions.
NFR6: Faculty and students learn core functions ≤ 30 minutes.
NFR7: Clear error messages with error recovery rate ≥ 90% within single attempt.
NFR8: System responsiveness ≤ 2 seconds for any action.

4.3 Performance
NFR9: Handle up to 1,000 concurrent users without degradation.
NFR10: Critical transactions (blockchain log entry) processed within 5 seconds.
NFR11: Data retrieval actions response time < 3 seconds for 95% of queries.
NFR12: Support data visualization of up to 10,000 log entries without performance issues.

4.4 Security
NFR13: Blockchain technology guarantees 100% immutability of student and system logs.
NFR14: Unauthorized access attempts logged with alert to admin within 5 seconds.
NFR15: Resilient against DDoS attacks, maintaining 99% uptime.
NFR16: All sensitive data encrypted.
