Project: CodeCommClassifier - Automated Code Comment Classification

1. INTRODUCTION
1.1 Purpose
This document outlines the problem and proposed solution for CodeCommClassifier, a project aimed at improving code comprehension by classifying source code comments into categories like functionality descriptions, metadata, etc.

1.2 Scope
The research focuses on developing a model to classify code comments with higher accuracy than existing baselines, using advanced transformer models to help developers quickly locate relevant information.

1.3 Definitions, Acronyms, Abbreviations
- SVM: Support Vector Machine
- NLP: Natural Language Processing
- BERT: Bidirectional Encoder Representations from Transformers
- F1-score: A measure of a model's accuracy

2. OVERALL DESCRIPTION
2.1 Product Perspective
The system automates the classification of code comments, helping developers to optimize code comprehension and streamline maintenance efforts by reducing manual searching through extensive documentation.

2.2 Product Functions (Summary)
- Dataset acquisition from NLBSE.
- Data preprocessing (cleaning, normalization).
- Feature extraction (TFIDF).
- Oversampling (EDISON) for low-resource languages.
- Model selection and evaluation (CodeBERTa, GraphCodeBERT, PLBART).
- Domain post-training.
- Evaluation metrics (Accuracy, F1-score, Precision, Recall).

2.3 User Classes and Characteristics
- Software Developers: Primary beneficiaries who need to quickly access relevant comments.
- Open-Source Communities: Who will benefit from better documented code.
- Organizations: Involved in software maintenance.

2.4 Operating Environment
Tool intended for integration into software development workflows.

3. FUNCTIONAL REQUIREMENTS
3.1 Dataset Acquisition
FR1.1: The system shall acquire a large corpus of code comments from the NLBSE Tool Competition Website.

3.2 Preprocessing
FR2.1: The system shall remove null spaces and special characters from the raw data.
FR2.2: The system shall convert text to lowercase.
FR2.3: The system shall split the cleaned dataset into training and testing sets.

3.3 Feature Extraction
FR3.1: The system shall extract features by converting code comments into vectorized representations (e.g., TFIDF).

3.4 Oversampling for Low-Resource Languages
FR4.1: The system shall apply the EDISON oversampling technique to balance datasets, especially for low-resource languages like Pharo.

3.5 Model Selection and Evaluation
FR5.1: The system shall experiment with multiple transformer-based models (CodeBERTa, GraphCodeBERT, PLBART).
FR5.2: The system shall evaluate each model's performance on the validation dataset.

3.6 Training the Selected Model
FR6.1: The system shall optimize model performance by tuning parameters (batch size, learning rate, epochs).
FR6.2: The system shall use early stopping to prevent overfitting.
FR6.3: The system shall apply hyperparameter optimization.
FR6.4: The system shall employ multi-level layer aggregation to capture complex patterns.

3.7 Domain Post-Training
FR7.1: The system shall conduct domain post-training to specialize the model in code comment classification.

3.8 Evaluation Metrics
FR8.1: The system shall use Accuracy, F1-score, Precision, and Recall to evaluate model performance.

4. NON-FUNCTIONAL REQUIREMENTS
4.1 Performance
NFR1: The system shall process and classify comments within a reasonable time frame.
NFR2: The system shall handle large datasets efficiently.

4.2 Accuracy
NFR3: The system shall aim to achieve classification accuracy higher than the baseline.

4.3 Robustness
NFR4: The system shall be robust to variations in comment styles and programming languages.
